# Actions Index

> Scope: Phase 5 (actions, spec `specs/006_actions/design.md`) — tasks act on
> the inbox. Two new tables (migration `0011_actions.py`): `task_action_rules`
> (the grants — `trigger`/`action_type`/`mode` vocab, default-deny by absence:
> no rule for a (task, trigger) means that action can never fire) and
> `task_actions` (the audit ledger — exactly one of `source_event_id`/
> `source_link_id` CHECK-enforced, two partial unique indexes for
> idempotency). A pure evaluator (`app/actions/rules.py`: `evaluate_event`/
> `evaluate_link`, no IO, no `app.db.models` import at all), a never-commits
> CRUD/audit repo (`app/actions/repo.py`), a firing engine
> (`app/actions/engine.py`: `fire_rules_for_event`/`fire_rules_for_link`, own
> commit strictly AFTER the caller's own commit of the firing evidence,
> optional `rules=` batching to dodge N+1 across a loop, publishes
> `action_updated {task_id}`, `mode='auto'` → enqueues
> `workers/action_tasks.execute_action`), the Celery execution worker
> (`workers/action_tasks.py`: `execute_action` + the synchronous
> `execute_action_inner` seam the approve route also calls directly), the
> Gmail write-client additions (`gmail/client.py`: `archive_thread`/
> `label_thread`/`create_draft`/`modify_thread_labels`, every one DB-scope-
> preflighted before any network call, no `gmail.send` capability anywhere),
> the HTTP surface (`api/actions.py`: rules CRUD + approve/reject/undo), the
> `has_write_scopes` auth-state field + the OAuth-callback-written
> `users.gmail_granted_scopes` column, the typed `"type": "event"|"action"`
> `/api/reviews`/`/api/activity` union (extended in `api/tasks.py`, not this
> module), and the client rules/actions UI (`client/src/actions/ActionCard.tsx`,
> `pages/task/{RulesSection,TaskActionsPanel}.tsx`, the HUD `ReviewTray`/
> `ActivityTicker`'s mixed-type rendering, the HUD scope-migration banner).
> Cross-refs: TASKS_INDEX (the task engine this hooks into — events, links,
> entities, refold); WORKERS_INDEX (Celery task/session/late-import
> conventions this module reuses verbatim); INBOX_SYNC_INDEX (`gmail_sync`'s
> fresh-link firing hook on the sync path); CLIENT_INDEX (the shared
> `TasksProvider`/SSE-singleton conventions this UI plugs into).

## Routes / Tasks / Entrypoints

**HTTP (server/app/api/actions.py, prefix `/api`):**
| Route | Status | Behavior |
|-------|--------|----------|
| `POST /tasks/{task_id}/rules` | 201 | Create a rule; 422 if `task.kind != "tracker"` or `_validate_rule_fields` fails; bumps `task.version` + publishes `task_updated`. |
| `GET /tasks/{task_id}/rules` | 200 | `{"rules": [...]}`, all non-deleted rules for the owned task. |
| `PATCH /tasks/{task_id}/rules/{rule_id}` | 200 | Merges `exclude_unset` fields onto the rule, re-validates the FULL merged shape, bumps + publishes. |
| `DELETE /tasks/{task_id}/rules/{rule_id}` | 204 | Soft delete; idempotent (already-deleted → silent no-op, no bump/publish). |
| `POST /actions/{action_id}/approve` | 200 / 409 / 500 | Synchronous `execute_action_inner`; 409 `"action is not pending"` (wrong status) or `"action source no longer valid"` (stale source → auto-rejected inline); 200 with the action body otherwise (incl. a `status:'failed'` MissingScopesError outcome). |
| `POST /actions/{action_id}/reject` | 200 / 409 | 409 if not `'proposed'`; else `status='rejected'`. |
| `POST /actions/{action_id}/undo` | 200 / 409 | 409 if not (`executed` + reversible type) or missing undo data or a `MissingScopesError`; else replays the exact inverse via `modify_thread_labels`, `status='undone'`. |

**HTTP (server/app/api/tasks.py, Phase 5-extended routes — full route table → TASKS_INDEX.md):** `GET /api/reviews`/`GET /api/activity` now return a merged `event|action` union; `POST /tasks/{id}/threads` (attach), `DELETE /tasks/{id}/threads/{thread_id}` (detach), `POST /tasks/{id}/events/{id}/approve|revert`, `POST /tasks/{id}/entities/{id}/state` (manual edit) all gained a rule-firing/auto-reject hook (see Files table above for exactly which and under what guard); `POST /tasks/{id}/entities/{id}/merge` gained neither.

**Celery task:** `execute_action(user_id, action_id)` (`app.workers.action_tasks.execute_action`) — ONLY ever enqueued by `actions/engine.py`'s `mode='auto'` dispatch, never called directly by any route.

**Rule-firing call sites (not routes — internal hook points, `actions_engine.fire_rules_for_event`/`fire_rules_for_link`):** `workers/task_engine_tasks.py`'s `process_task_updates` (batched `rules=`), `extract_for_thread` (single-pair), `backfill_task`'s tracker branch (both its link-writing and extraction phases); `workers/gmail_sync.py`'s `_fire_link_rules` (called from `partial_sync_inbox`/`full_sync_inbox`/`extend_inbox_history`, gated on `_write_task_links`' `fresh` list); `api/tasks.py`'s `attach_thread` (gated on `newly_attached`), `approve_event`, `edit_entity_state` (gated `if event.thread_id:`, structurally always false for this route).

## Data flows / cross-subsystem touchpoints

Rule fires from an applied event (extraction, approve, or manual-edit no-op case):
```
Worker/API commits an applied TaskEvent
  ──[AFTER commit]──> actions_engine.fire_rules_for_event(event, rules?)
    ──[evaluate_event: pure match]──> ActionIntent list
    ──[insert_intent: SELECT-first + IntegrityError backstop]──> Postgres task_actions (status='proposed')
    ──[db.commit()]──> Postgres
    ──[PUBLISH user:{uid} action_updated {task_id}]──> Redis ──> SSE ──> Browser (ReviewTray/ActivityTicker/TaskActionsPanel refetch)
    ──[mode='auto' only]──> execute_action.apply_async([user_id, action.id]) ──> Worker
```

Rule fires from a freshly-attached link (sync/backfill triage, or manual attach):
```
gmail_sync / task_engine_tasks / api.tasks.attach_thread commits an upsert_link() call
  ──[LinkUpsert(link, newly_attached=True) only]──> actions_engine.fire_rules_for_link(link, rules?)
    (same insert/commit/publish/auto-enqueue shape as above)
```

Auto-execution:
```
Worker.execute_action(user_id, action_id)
  ──[_source_still_valid? no ──> status='rejected', commit, publish, return]
  ──[yes]──> execute_action_inner
    ──[dispatch by action_type]──> gmail_client.{archive_thread|label_thread|create_draft-via-_execute_draft_reply}
      ──[MissingScopesError, pre-network]──> status='failed', error="needs permission: ...", commit, publish
      ──[other exception]──> propagates ──> execute_action's except: rollback, _record_action_failure (fresh session), re-raise
      ──[success]──> status='executed', result={...}, executed_at=now, commit, publish
```

Synchronous approve (same dispatch, no Celery):
```
Browser ──[POST /api/actions/{id}/approve]──> API
  API ──[status!='proposed']──> 409
  API ──[execute_action_inner(SAME db)]──> (identical dispatch/outcomes as above, same session)
    ──[rejected outcome]──> 409 "action source no longer valid"
    ──[executed OR failed]──> 200 + action body
```

Undo:
```
Browser ──[POST /api/actions/{id}/undo]──> API
  API ──[status!='executed' or non-reversible type, or missing result data]──> 409
  API ──[gmail_client.modify_thread_labels(inverse of result)]──> Gmail v1
    ──[MissingScopesError]──> 409
    ──[success]──> status='undone', commit, publish ──> 200
```

Revert / detach auto-reject (same transaction as the revert/detach itself):
```
api.tasks.revert_event / detach_thread ──[event.status flips]──> actions_repo.reject_proposed_for_source(source_event_id | source_link_id)
  ──[flips every still-'proposed' TaskAction matching that source to 'rejected']──> (same commit as the route's own)
```

## Decision points & gotchas

- **Default-deny by absence, no master dial.** A rule row IS the grant — no rule for a (task, trigger) pair means that action can never fire, ever, regardless of anything else. There is no separate on/off setting anywhere.
- **`draft_reply` can never auto-run — three independent guards, not one.** (1) `api/actions.py::_validate_rule_fields` rejects `action_type='draft_reply' and mode='auto'` at rule write/patch time (422). (2) `actions/engine.py::_insert_and_dispatch` raises `RuntimeError` if a `draft_reply` action's rule is somehow `mode='auto'` at dispatch time (would only happen from a bypassed guard #1, e.g. a hand-edited DB row). (3) `workers/action_tasks.py::execute_action` (the Celery auto-only entrypoint) refuses and marks the action `failed` if a `draft_reply` action reaches it at all. Approving a `draft_reply` action is ALWAYS the synchronous `POST /actions/{id}/approve` path — it never goes through the auto-execute Celery task.
- **No send capability anywhere — not a missing scope, a missing CODE PATH.** No `gmail.send` scope constant, no `users().messages().send()` call, anywhere in `gmail/client.py` (confirmed by grep; a tested invariant). `create_draft` only ever creates a Gmail draft.
- **Idempotency is two-layered, mirroring `task_events`' own precedent.** `insert_intent`'s SELECT-first check is the fast path; the `with db.begin_nested(): ... except IntegrityError` around the actual insert is the backstop for the race window between that SELECT and the flush — both keyed off the SAME two partial unique indexes the migration created. Replaying an event (refold) or re-upserting a link (re-triage) can never double-fire a rule.
- **Fire-after-commit is two levels deep.** Every hook site commits the firing evidence (the applied event, or the freshly-attached link) FIRST; only then calls `fire_rules_for_event`/`fire_rules_for_link`, which itself commits the `TaskAction` insert(s) as a SEPARATE transaction. A bug in rule-firing can never roll back the evidence it fired on; a bug in the evidence-write already prevented firing from being reached at all.
- **`refold_entity` never fires rules — structural, not a special-cased skip.** It directly recomputes `entity.state` as a fold over already-`applied` events; it never calls `repo.apply_event` (the ONE function every actual firing hook site wraps), so it has no path into `fire_rules_for_event` even though revert/detach/merge all call it as part of their own request.
- **`merge_entity` is the one entity-mutating route with NO Phase 5 hook at all.** `repoint_entity_events` reassigns the loser entity's events to the winner WITHOUT changing their status — they stay `applied`, so any actions they already sourced remain legitimately evidenced. There is nothing to revert and nothing new to fire; only `refold_entity(winner)` runs (see the bullet above for why that alone can't fire anything either).
- **Manual entity-state edits are hooked but structurally always a no-op.** `edit_entity_state` DOES call `fire_rules_for_event`, but wrapped in `if event.thread_id:` — a user-origin `append_event` call passes no thread_id/message_id (it isn't extracted from an email), so this branch is dead code in practice for this one route, by construction, not by an oversight (the module carries an explicit comment to this effect). An `entity_entered_stage` rule always needs a concrete thread to archive/label/draft-reply against.
- **Scope preflight is always synchronous and pre-network.** `require_scopes`/`_require_scopes_missing` are pure functions with no IO; every Gmail write method in `gmail/client.py` calls one of them and raises `MissingScopesError` BEFORE building a Gmail client or making any request — a missing scope never consumes a retry budget and never triggers an implicit browser consent prompt.
- **System-label defense is doubled.** `api/actions.py`'s `_GMAIL_SYSTEM_LABELS` set rejects a rule that NAMES a system label at write/patch time (422); `gmail/client.py::label_thread`'s `lbl["type"] == "user"` filter independently ensures the runtime match against existing Gmail labels can never resolve to a system label either, even if the write-time guard were somehow bypassed.
- **Approve is 200-with-status-in-body except for one 409.** `POST /actions/{id}/approve` returns HTTP 200 for BOTH an `executed` outcome and a `failed` (MissingScopesError) outcome — only a stale-source `rejected` outcome (the evidence was reverted/detached out from under a still-proposed action between propose and approve) is surfaced as 409. The client must read the action body's own `status` field to know whether the Gmail write actually landed.
- **Client tracker-only gating lives at the call site, not inside the components.** Neither `RulesSection` nor `TaskActionsPanel` checks `task.kind` internally — `TaskDetail.tsx` repeats `{detail.kind === 'tracker' && <Component .../>}` at two separate render points. A future third consumer of either component would need to add its own gate; it isn't enforced centrally.
- **`deleteRule` is the one action/rule fetcher NOT using `throwWithDetail`.** Every other Phase 5 fetcher (`createRule`, `patchRule`, `approveAction`, `rejectAction`, `undoAction`) surfaces a server 422/409 `detail` string verbatim via `throwWithDetail`; `deleteRule` throws a bare `Error(`delete rule: ${status}`)` on a non-204 response instead — harmless in practice since the route is a soft-delete that 404s only on ownership/existence, not validation, but inconsistent with the sibling fetchers.
- **`action_updated` is handled differently by cross-task vs. single-task consumers.** The HUD's `ReviewTray`/`ActivityTicker` refetch on ANY `action_updated` event, unfiltered (correct for them — they aggregate every task's actions). `TaskActionsPanel` filters to `e.task_id === taskId` (correct for it — a single task's panel refetching on an unrelated task's action would be wasted work, though harmless).
- **The scope-migration banner is session-only, by design.** `HudPage`'s `scopeBannerDismissed` is plain `useState`, never persisted — it reappears on the next full page load specifically so a user who dismisses it and then forgets for weeks keeps seeing it periodically, rather than a permanent dismiss silently hiding the one thing telling them their rules are inert.
- **"Needs permission" rules save fine and self-activate.** Per spec §1 (confirmed structurally: `require_scopes` is called only inside the Gmail write path, never at rule create/patch time in `api/actions.py`) — a rule configured before a user re-consents saves normally and shows the client's "needs permission" badge (`RulesSection`, driven by `useAuth()`'s `has_write_scopes`), then activates automatically the moment `gmail_granted_scopes` updates on the next OAuth callback — the evaluator has no scope awareness at all, only the DISPATCH layer (`gmail/client.py`) does.
