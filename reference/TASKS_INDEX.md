## The 8-step validator, in order (`transitions._process_one`)

Every proposal from `extract_transition.parse_response` runs through exactly these steps, in this order, and lands in exactly one of applied / pending_review / dropped. **Phase 2B Task 1** threads a `pending_reason: str|None` through steps 2/3/5/8 (replacing the old bare `forced_pending: bool`) â€” whichever guard is the FIRST to force pending wins and stamps its own reason string; every later guard's own force-pending branch is gated `if pending_reason is None` so it never overwrites an earlier reason. `proposed_entity` (the LLM's verbatim entity string) is set only alongside step 2's `near_duplicate_entity` reason. Both fields are written to the `pending_review` `TaskEvent` row only (`None`/`None` on `applied`).

1. **Shape.** `field=="stage"` â†’ value must be in `schema.all_stages()`; else `schema.attr(field)` must resolve and `coerce_value` must succeed. Failure â†’ dropped (malformed output is noise, never even reviewed).
   - **Step 1b â€” message reference (foreign-message-id guard).** `message_id` must resolve to an actual message in `message_by_gmail_id` (the just-fetched thread's own messages) â€” otherwise dropped before entity resolution even runs. Without this, a hallucinated/omitted `message_id` would silently resolve to `internal_message_id=None` downstream: wrong provenance in the audit log at step 5's fence check, and no dedup against a rerun of the same hallucinated citation at step 7 (`message_id=None` never collides).
2. **Entity resolution.** Singleton task â†’ `get_or_create_entity('_self')`. Multi-entity: exact `normalize_key` match on the known-entities roster; else best `difflib` similarity â€” `>= 0.6` auto-matches; `[0.4, 0.6)` routes to the closest existing entity but **forces `pending_review`** (`pending_reason="near_duplicate_entity"`, `proposed_entity=` the LLM's raw entity string) regardless of the LLM's `is_new_entity` flag (a near-duplicate needs a human, not an auto-mint); `< 0.4` mints a **`PendingCreate` sentinel** (deferred â€” see Files) if `is_new_entity` was set, else drops (no entity, no flag â†’ nothing to do). This is the first guard in the chain, so its `pending_reason` (when set) can never itself be overwritten by a later step â€” but per the paragraph above, no later step is allowed to overwrite it either way.
3. **Stage legality.** Only for `field=="stage"` **and only if no earlier guard already claimed `pending_reason`**: current stage already terminal â†’ forces pending (`pending_reason="terminal_locked"`; only users move terminal entities out); a backward move (`all_stages.index(new) < index(current)`) â†’ forces pending (`pending_reason="backward_move"`). Forward/lateral/skip moves are unrestricted (schema-level rule; this step is the validator's own backward/terminal gate).
4. **Evidence.** Whitespace-normalized `evidence_quote` must be an exact substring of the whitespace-normalized full thread text â€” else dropped outright (fail-closed; the cheapest hallucination guard, no pending-review fallback for a fabricated quote).
5. **Correction fences.** If the entity (real, not `PendingCreate` â€” a not-yet-created entity has no prior corrections to fence against by construction) has a `latest_applied_user_event`: the proposal's evidence message's `gmail_internal_date` must be strictly newer (ms-epoch, naive-as-UTC guard â€” see Files) than that event's `created_at`, else forces pending â€” `pending_reason="fence_blocked"`, but **only if `pending_reason` is still unset** from an earlier step.
6. **No-op.** `entity.state.get(field) == new_value` â†’ dropped (not even reviewed â€” nothing changed).
7. **Idempotency.** SELECT-first check (`find_event_for_message_field(task_id, internal_message_id, field)`) â€” an existing row â†’ dropped as a duplicate. The partial unique index `uq_task_event_msg_field` is only the race backstop for a genuinely concurrent insert on a **migrated** DB; it is absent from the `create_all` test fixture (raw SQL in the migration, not a SQLAlchemy `Index()` on the model) â€” this SELECT-first path is what tests actually exercise, and the index itself is verified by review, not a unit test.
8. **Confidence gate + deferred mint.** Only now â€” after every guard above passed â€” does a `PendingCreate` sentinel actually get materialized into a real `TaskStateEntity` row (`get_or_create_entity`), because both outcomes below need a real `entity_id`. Then: `not pending_reason and confidence >= settings.task_apply_confidence` (default 75) â†’ `status="applied"` (`apply_event` folds it into `entity.state`, bumps `task.version`); else â†’ `status="pending_review"`, and if no earlier step claimed a `pending_reason`, this branch defaults it to `"low_confidence"` (a plain confidence-gate deferral, nothing else forced it) â€” this is the ONLY step that can produce that particular reason. Both branches insert via `_append_event_guarded`, which wraps the flush in a `db.begin_nested()` SAVEPOINT and swallows `IntegrityError` as the same idempotency race backstop mentioned in step 7 (scoped to just this one insert, so a genuine race on one proposal can't poison every other proposal already flushed-but-uncommitted in the same `validate_and_stage` batch).

```
POST /api/tasks/{id}/threads (user attach) â”€â”€[upsert_link origin=user; if add_example: append_example(positive) onto task.criteria; commit; apply_async]â”€â”€>
  Worker.extract_for_thread â”€â”€[extract_for_pair, 1 LLM call, prompt now includes recent_user_events as user_corrections]â”€â”€> task_events/entities â”€â”€[PUBLISH task_updated]

DELETE /api/tasks/{id}/threads/{thread_id} (user detach, synchronous â€” no Celery) â”€â”€>
  upsert_link(detached) + if add_example: append_example(nearmiss) onto task.criteria
  + list_pending_events_for_thread â†’ reject each (Phase 2B Task 1)
  + revert applied events for thread + refold touched entities â”€â”€[commit; PUBLISH task_updated]
```

## Decision points & gotchas

- **Sticky user links.** `repo.upsert_link`'s one rule: an existing `origin='user'` row is never silently downgraded by an `origin='llm'` call (no-op, returns `None`). This is ALL the protection a user's explicit attach/detach decision needs against a later automatic reclassify or backfill re-run â€” there's deliberately no separate "is this link user-owned?" guard anywhere else.
- **Correction fences are strictly-newer, ms-epoch, naive-as-UTC-guarded.** `transitions._to_ms_epoch` treats a naive `created_at` as UTC before comparing â€” needed because SQLite (the test dialect) doesn't round-trip `tzinfo` through commit+`expire_on_commit`+reload, so a reloaded row's `created_at` can come back naive even though every write is `datetime.now(timezone.utc)`. No-op on Postgres (real `TIMESTAMPTZ`). The fence itself: after ANY user-origin applied event on an entity, only evidence **strictly** newer than that event may move it again â€” a message with `gmail_internal_date` equal to or older than the fence forces `pending_review`.
- **Refold semantics: user wins created_at ties.** `refold_entity` sorts an entity's applied events by `(created_at, 0 if origin=='llm' else 1)` before folding â€” so on an exact-timestamp tie, the llm-origin event folds in first and the user-origin event folds in **last**, winning. Fields with no surviving applied event (e.g. its one applied event got reverted/rejected) are dropped from `state` entirely, except `stage`, which always gets an explicit `None` default so the board can rely on the key existing.
- **Approval re-dates the event so user decisions survive refolds.** `POST .../events/{id}/approve` sets `event.created_at = datetime.now(timezone.utc)` immediately before `apply_event` â€” `refold_entity`'s own fold key is `created_at` ("assertion time," per the bullet above), so an approved event that was originally proposed well before other events later applied to the same entity would otherwise re-sort back to its old position on any future refold (a revert/detach/merge touching that entity) and lose the fold to a newer-but-since-approved-over applied event, silently undoing the user's explicit approval with no error and no signal. Re-dating means the approved event jumps to the top of the feed â€” its `created_at` now reads as the approval time, not the original proposal time; the original proposal time isn't lost, just no longer on this row directly â€” it's recoverable via a `message_id` join back to the source `InboxMessage.internal_date`. Gmail provenance fields (`message_id`/`gmail_message_id`/`evidence_quote`) are untouched â€” only the fold-ordering key moves.
- **Rejecting a minted new-entity pending deletes the orphaned empty entity.** Step 8 of the validator mints a real `TaskStateEntity` row even for a `pending_review` outcome (both the applied and pending branches need a real `entity_id` on the event they write) â€” if the rejected event was that entity's ONLY history and nothing was ever folded into its `state`, rejecting it would otherwise strand an empty, permanently-visible board row with no path to ever remove it. `reject_event` guards on `event.entity_id is not None` and calls `repo.delete_entity_if_orphaned(task_id=..., entity_id=event.entity_id, excluding_event_id=event.id)`, which hard-deletes the entity only if zero OTHER events reference it and `entity.state` carries no non-null values (`{}` and refold's own `{"stage": None}` default both count as empty). An entity with other history (an applied event, another still-pending proposal, a manual state edit) survives untouched. This doesn't bump `task.version` or need a dedicated SSE nudge beyond the `pending_count` change `_publish_task_updated` already carries â€” the client's `TaskDetail` already refetches the board on reject, so a vanished entity is simply absent from that refetch (see CLIENT_INDEX for the one not-explicitly-covered scenario: a drawer/deep-link open on that exact entity id at the moment of reject).
- **Confidence scales 0-100.** `TASK_APPLY_CONFIDENCE=75` (step 8's auto-apply gate) and `TASK_LINK_CONFIDENCE=60` (auto-link gate in `_write_task_links` and `backfill_task`'s tracker branch) are both `config.py` defaults, independently tunable. Note the two prompts' confidence handling differs: `extract_transition._parse_one` **drops the whole proposal item** if confidence isn't an exact int in `[0,100]`; `triage_thread.parse_response` **clamps** an out-of-range numeric confidence into `[0,100]` instead (only drops for non-numeric/bool/missing) â€” an LLM off-by-a-hair on a tracker score still counts, but a malformed extraction proposal doesn't get a guessed confidence.
- **Difflib entity similarity, NOT `pg_trgm`.** `ENTITY_MATCH_THRESHOLD=0.6` / `ENTITY_PENDING_THRESHOLD=0.4` are pure-Python `difflib.SequenceMatcher.ratio()` on `normalize_key`-normalized strings â€” a deliberate deviation from the spec's `pg_trgm` mention, so entity resolution is dialect-independent and unit-testable on the SQLite `create_all` fixture, which has no Postgres extensions at all.
- **The partial unique index only exists in migration-built DBs.** `uq_task_event_msg_field` (`task_id, message_id, field` WHERE `message_id IS NOT NULL`) is `op.execute` raw SQL in `0007_tasks.py`, not a SQLAlchemy `Index()` attached to the `TaskEvent` model â€” so `Base.metadata.create_all()` (the test conftest's fixture setup) never creates it. Step 7's SELECT-first check (`find_event_for_message_field`) is therefore the **only** guard the test suite exercises; the index itself is a race backstop for a genuinely concurrent duplicate insert on Postgres, verified by code review rather than a unit test (per `transitions._append_event_guarded`'s own comment).
- **Version bumps on every PATCH, attach, detach, and delete â€” but NOT on reject or criteria mutations.** `bump_version` is called by `apply_event`, `refold_entity`, PATCH's own explicit `bump_version` call, `attach_thread`, `detach_thread`, and `delete_task` (all unconditionally). `reject_event` does **not** bump â€” its immediate `task_updated` publish carries whatever version the task already had. **Confirmed post-2B:** `attach_thread`/`detach_thread`'s new `append_example` call (Phase 2B Task 2) mutates `task.criteria` in the same request but never touches `task.version` â€” a criteria-only change is invisible to `version`-based change detection entirely; only `pending_count` moving (see next bullet) or a later `bump_version`-calling operation would surface it. A client relying on "every `task_updated` implies a version change" would miss all of these paths â€” this is exactly why the 2B client's `TasksProvider` treats `pending_count` as an equally load-bearing second signal (see CLIENT_INDEX).
- **Detach auto-rejects that thread's still-pending events (Phase 2B Task 1), **and** detach bumps version unconditionally.** `DELETE /tasks/{id}/threads/{thread_id}` flips every `list_pending_events_for_thread` result to `status="rejected"` before its existing revert/refold logic runs (see api/tasks.py lines 473-476). This is deliberate â€” a detached thread's proposals citing evidence from a thread the task no longer tracks must not remain approvable. Detach also calls `bump_version` unconditionally (line 506), even if there were zero applied events to revert (so `refold_entity` never runs and would have bumped nothing). Rejected-via-detach events are indistinguishable on the wire from a manual reject (`status="rejected"`, no separate reason/marker) â€” `pending_reason` is preserved from before the flip (it was already set when the event was staged as pending_review) rather than cleared or overwritten.
- **Pending-reason vocabulary is fixed at 5 values, `proposed_entity` fires exactly once.** `pending_reason` is written by the validator's own guard chain only (`near_duplicate_entity`/`backward_move`/`terminal_locked`/`fence_blocked`/`low_confidence`) â€” the API layer never writes it (the one manual-state-edit event a user creates via `POST .../entities/{id}/state` is `status="applied"` immediately, so it never gets a `pending_reason` at all). `proposed_entity` is non-null in exactly one case (`pending_reason=="near_duplicate_entity"`) â€” for every other pending event (and every applied/rejected/reverted event) it's `None`, even if the underlying proposal DID reference an entity name (that name is simply discarded once resolution succeeds normally).
- **`extract_transition._parse_one`'s float-confidence clamp only helps a numeric-but-wrong-type response â€” a `bool` or truly out-of-range value still drops the whole proposal item.** Mirrors `triage_thread.parse_response`'s existing clamp (documented above), but the two prompts' contracts are otherwise unchanged: an out-of-range **int** confidence (e.g. `150`) is still dropped outright by `extract_transition`, only a **float** (`82.5`) gets rounded + clamped. `triage_thread` clamps both float AND out-of-range int alike â€” the two parsers are still NOT identical, just closer than before.
- **Learning-loop example cap is a silent FIFO eviction, and `_neutralize_tags` only protects the criteria text, not the classify prompt's own escaping (there is none â€” it never needed any).** `cap_examples` drops the OLDEST blocks once a criteria string exceeds `EXAMPLE_CAP=30` (positives+nearmisses combined, not per-tag) â€” a task that's been corrected 40 times over its lifetime silently loses its first 10 corrections' worth of criteria signal with no user-visible warning; only the newest 30 survive. `_neutralize_tags` exists solely so an email's own literal `<positive>`/`<nearmiss>`-shaped substrings can't be mistaken for real block markup by `cap_examples`'s regex (or by a human/LLM reader of the raw criteria text) â€” it does nothing to protect against prompt injection via the *content* of an example (the classify prompt already treats all criteria text as untrusted-but-literal, same as before this feature existed).
- **The 422-detail contract: every schema/manual-edit/merge validation failure is `HTTPException(422, str(exc))` â€” a bare string, never a structured body.** `PATCH /tasks/{id}` (state_schema re-validation, additive-change violations), `POST .../entities/{id}/state` (coerce_value failures), and `POST .../entities/{id}/merge` (self-merge) all raise this way; FastAPI serializes it as `{"detail": "<the string>"}`. The 2B client's `throwWithDetail` helper (`lib/api.ts`) is built specifically around this shape â€” see CLIENT_INDEX for the client-side contract and its one carve-out (a list-shaped `detail`, e.g. from pydantic body-level validation, falls back to a generic message instead of trying to render an array).
- **Backfill's "matched" count excludes already-attached-below-threshold threads â€” a landmine for a future re-backfill endpoint.** In `backfill_task` phase 1, a candidate whose THIS-run confidence falls below `TASK_LINK_CONFIDENCE` is skipped via `continue` **before** `upsert_link` is ever called â€” it is never added to `matched_thread_ids`, even if that thread already has an attached link from a previous run. Phase 2's extraction candidate set (`matched_threads`) is derived solely from `matched_thread_ids`, so such a thread is excluded from re-extraction THIS run too, despite still being attached in the DB. A future "re-run backfill" endpoint that assumes `matched` â‰ˆ "everything currently attached" would silently skip these.
- **Triage replaced classify on the sync path (one call, dual-write); `classify()` itself is deleted (Phase 4).** `gmail_sync._triage_batch` (partial/full/extend) and `workers/task_engine_tasks._run_bucket_backfill`/`_run_tracker_backfill` (via `backfill_task`) call `llm.classify.triage()` â€” one Haiku call per thread returns both the bucket pick AND tracker relevance (D2: no doubled LLM volume vs. running two separate calls). The dual-write on the sync path: bucket_id feeds the existing upsert as before; every `(task_id, confidence) >= TASK_LINK_CONFIDENCE` tuple gets `upsert_link(origin="llm", state="attached")` in the same transaction. `classify()` â€” the pre-triage, bucket-only classifier `triage()` replaced back in Phase 2A â€” carried zero production call sites from that point on (kept only as a tested building block; `triage_thread`'s bucket-section rendering still delegates to `classify_thread` so the zero-tracker `triage()` case matches what `classify()` used to produce byte-for-byte) and is now gone outright; `triage()` is the only classify-stage entrypoint left in the codebase.
- **Propose retry symmetry: a 2-call cap, not "retry only on schema-invalid."** `propose_task_draft` spends **at most one retry total** per draft, no matter which of the two failure modes the first attempt hits (unparseable JSON â€” including what a transient API error degrades to, since `call_messages` returns `""` on any exception â€” vs. parseable-but-schema-invalid). Critically: if attempt 1 is unparseable and the retry comes back parseable-but-schema-invalid, that schema failure does **not** get its own second retry â€” it falls straight to `_FALLBACK_SCHEMA_DICT` rather than compounding into 3 LLM calls. At most 2 propose calls ever happen for one draft.
- **`GET /api/tasks/draft/{draft_id}` â€” the one 403 in an otherwise 404-only router â€” no longer exists (Phase 4.5).** Historical note: it used to return 403 "not your draft" for a wrong-owner draft_id (an intentional asymmetry to every other ownership check in this file, which return 404 for both "not yours" and "doesn't exist"). The whole draft-poll route, `_DraftBody`, and `draft_cache.py` are deleted â€” `GET /api/jobs/{id}` is its replacement and follows the router's usual 404-only convention (`_require_owned_job`), no 403 carve-out. Don't assume a stale client still polling the old path gets anything but a 404/405.
- **`stage_counts` key order is server-controlled and NOT alphabetical (Phase 3).** `_stage_counts` orders keys schema-stage-order-first (only when `state_schema` parses via `validate_schema`), then any observed stage values not in the schema (e.g. a stage a since-edited schema no longer declares), then a trailing `"(no stage)"` bucket for entities with `state.get("stage") is None` â€” stages with zero entities are omitted, so an empty task's `stage_counts` is `{}`. A schema-less task, or one whose `state_schema` fails `validate_schema` (hand-edited/migration-corrupted â€” the same failure mode `process_task_updates`'s per-task isolation already tolerates), falls back to observed order only. The client (`lib/api.ts`'s `TaskSummary.stage_counts` type comment, `HudPage.stageHistogramLabel`) must preserve this insertion order via plain `Object.entries` â€” sorting it would put `"(no stage)"` in the wrong place and scramble the schema's own stage sequence.
- **`GET /api/reviews`/`GET /api/activity` are the one place cross-task data leaves task-scoped ownership checks (Phase 3) â€” treat any change here as a security boundary, not a convenience query.** Every other read/write in this router is scoped by `get_owned_task`'s single-task 404 check; these two routes instead join directly on `Task.user_id == user.id` and `Task.is_deleted == False` (`repo.py`'s own module comment makes this explicit) to aggregate across ALL of a user's tasks in one query â€” a mistake here (e.g. a missing `Task.user_id` filter) would leak one user's email-derived event text/quotes to another. Both routes clamp `limit` in Python (`max(min(...))`) rather than reject an out-of-range value â€” a deliberate departure from `GET /tasks/{id}/events`'s own `limit` param, which is a declarative `Query(ge=1, le=200)` and DOES 422 on a violation; don't assume the two `limit` params share validation behavior.
- **Bucket unification is an ID-preserving migration, not a re-key (Phase 4).** `0009_bucket_unification.py` copies every `buckets` row into `tasks` under the SAME `id`, so `InboxThread.bucket_id` values are never rewritten anywhere â€” only the column's FK target changes (Postgres-only; SQLite never enforced it). This is why the migration is a single raw-SQL `INSERT ... SELECT` with no id-remapping step and no data migration needed on `inbox_threads` itself. A "default bucket" (`user_id IS NULL`) is now indistinguishable, at the row level, from any other `kind='bucket'` task except by that one column â€” there is no separate "is this a default" flag.
- **`GET /api/tasks` with no `kind` param is now an explicit tracker-only filter, not an accidental one.** Before Phase 4, `list_tasks` had nothing else to filter â€” every row already was a tracker. Post-unification, the same endpoint's default (`kind=None` â†’ `list_tasks(kind="tracker")` in `api/tasks.py`) is load-bearing: without it, `GET /api/tasks` would return bucket rows too and buckets would leak into the HUD's task grid (`HudPage`'s task cards, `TasksProvider`'s list) alongside real trackers. Any future change to this default must preserve the tracker-only behavior explicitly, not rely on "there's nothing else in the table" being true anymore.
- **Bucket-kind tasks bump `version` and publish `task_updated` only through `/api/tasks` (never the shim).** The client mutates buckets exclusively through `PATCH/DELETE /api/tasks/{id}` (via `useBuckets`' `rename`/`softDelete` calling `patchTask`/`deleteTask`), which unconditionally bump version and publish on every successful call, tracker OR bucket (nothing in `_require_owned_task`/those two routes filters by `kind`). The `/api/buckets` shim (`api/buckets.py`'s `patch_bucket`/`delete_bucket`, thin wrappers over `bucket_repo.rename`/`bucket_repo.soft_delete`) exists only for stale pre-deploy tabs and is deleted in Phase 5 â€” it is unreachable from the current client. Don't assume any backward-looking code meant to read bucket mutations from `task_updated` consumers (e.g. monitoring dashboards or other consumers) will see them if they still target the shim path rather than the real routes.
- **A bucket's own criteria can never be edited or grown after creation via any current route.** `_PatchBody` in `api/buckets.py` is name-only (matching the old bucket shape); `_CreateTaskBody`'s learning-loop `add_example` machinery (attach/detach appending to `criteria`) is wired only through `POST/DELETE /tasks/{id}/threads`, which nothing calls for a bucket-kind task (buckets have no `ThreadsPanel`/attach-detach UI). A bucket's `criteria` is therefore fixed at creation time (from the goalâ†’draft wizard's confirmed positives/negatives) for its entire lifetime â€” the only way its threads get re-evaluated is a full `_run_bucket_backfill` re-run, and there is currently no endpoint that re-triggers one after creation (creating a NEW bucket re-triages every EXISTING bucket's candidate pool too, since `_run_bucket_backfill` triages against the full active set â€” but an existing bucket's own criteria text never changes).
- **Trigger changed: bucket creation now runs `backfill_task`, not the deleted `reclassify_user_inbox`.** Before Phase 4, `POST /api/buckets` fired `workers.tasks.reclassify_user_inbox` (a bespoke retriage-everything Celery task, ~110s for a large inbox). That task and its `_reclassify_all`/`_inline_reload` helpers are gone; both `POST /api/buckets` (the shim) and `POST /api/tasks` with `kind="bucket"` now enqueue the same `backfill_task` (kind='bucket' branch) that a new tracker's creation always used. Functionally similar (retriage the user's stored history against the full classify-relevant set) but the publish contract differs â€” see the bullet above and WORKERS_INDEX/INBOX_SYNC_INDEX for the full before/after.
- **Dismissal is `dismissed_at`-only â€” the literal string `"dismissed"` never occurs as a `stage` value anywhere in this codebase (Phase 4.5, spec 005 Â§1.2, ratified).** `jobs_repo.dismiss` sets only `job.dismissed_at`; it never touches `stage`, which keeps whatever value it last had. A dismissed `draft_ready` job simply never confirms (the confirm route's 409 checks `dismissed_at is not None` as an independent, ORed condition â€” see below); a dismissed `backfilling`/`running` job keeps running server-side and writing its own progress â€” there is deliberately **no** worker-side cancellation of a dismissed job's Celery task (out of scope, per the design spec) â€” it just disappears from the panel because `list_jobs(active_only=True)` excludes anything with `dismissed_at` set, regardless of stage. Do not write (or expect) a branch anywhere that tests `stage == "dismissed"`.
- **`POST /api/jobs/{id}/confirm`'s 409 is one compound condition, not two independent checks with different messages.** `if job.dismissed_at is not None or job.stage != "draft_ready": raise HTTPException(409, "job is not awaiting review")` â€” a dismissed `draft_ready` job and a job that's simply never reached `draft_ready` (still `proposing`) or has already moved past it (`backfilling`/`done`/`failed`) all produce the identical 409 detail string. There is no way for a client to distinguish "dismissed" from "wrong stage" from this response alone.
- **`jobs_repo.dismiss`'s idempotency is a repo-level no-op guard, not a route-level check or a 404/409.** `POST /api/jobs/{id}/dismiss` always re-fetches the job (404 if not owned) and unconditionally calls `dismiss` â€” the second-call-is-a-no-op behavior lives entirely inside `dismiss` itself (`if job.dismissed_at is not None: return`), which also means a repeat dismiss never re-touches `updated_at`. Contrast this with `DELETE /api/tasks/{id}`'s idempotent soft-delete, which is a route-level `get_owned_task_any_status` branch â€” two different idempotency patterns for a similar-sounding "calling this twice is safe" guarantee.
- **`jobs.payload`'s nested shape is a producer convention, not a repo- or model-enforced contract.** `set_payload` stores whatever dict it's given verbatim (no shape validation at the repo layer, and the column itself is just `JSON(none_as_null=True)`); the `{"proposal": {name, description, state_schema, keyword_probes}, "positives": [...], "near_misses": [...]}` shape only holds because `propose_task_draft` is the one real producer and `POST /api/jobs/{id}/confirm` (server) / the client's review step (see CLIENT_INDEX) are the only consumers, and all three agree on it by convention. A hypothetical future job kind that reuses `payload` for something else would need its own shape, not this one.
- **`GET /api/jobs?active=` â€” `active=0` is NOT "include dismissed within some window," it is NO filtering at all.** `list_jobs(active_only=False)` returns literally every job for the user regardless of `dismissed_at`/`stage`/`updated_at` age. Don't read `active=0` as "the dismissed-inclusive variant of the same active-window logic" â€” the active-window computation (7-day terminal cutoff) is skipped entirely, not widened.
- **`_record_job_failure` deliberately opens a brand-new `SessionLocal()` rather than reusing the failing run's own session, and swallows its own exceptions.** The run's own `db` may be poisoned by whatever just raised (a failed mid-batch commit, a dropped connection, an error thrown from deep inside `triage()`/`extract_for_pair` with the session left in an unknown state) â€” reusing it for the failure write risks a second exception masking the first. Any exception inside `_record_job_failure` itself is logged and swallowed, never re-raised; the caller's own `raise` (after calling this helper) is what surfaces to Celery, always the ORIGINAL exception. All three of `propose_task_draft`/`backfill_task`/`retriage_deleted_bucket` also wrap their `db.rollback()` call in its own nested `try/except` for the identical reason â€” a rollback that itself fails (e.g. connection already dropped) must not replace the original exception being propagated.
- **`retriage_deleted_bucket`'s "matched" and "changed" counters mean different things than they do in `_run_bucket_backfill`, despite reusing the same variable names.** In `_run_bucket_backfill`, `matched`/`changed` are the same set (a bucket_id write only happens when the pick differs, and every write "matches" a bucket). In `retriage_deleted_bucket`, `matched` counts only threads that land in a REAL (non-`None`) bucket, while `changed` (what feeds `threads_updated`) is broader â€” it includes threads that resolve to `None`/unclassified too, since every surviving orphan's stored value (the now-dead bucket) can never be reproduced by `resolved`, so every write "changes" the row by construction even when the new value is `NULL`. The job row's own `matched` counter (written via `_write_job_progress`) reflects the narrower, real-bucket-only count.
- **A tracker-kind delete never creates a `delete_retriage` job, and neither does a zero-orphan bucket delete.** `delete_task`'s retriage hook is gated on `task.kind == "bucket"` (trackers have no `bucket_id` relationship to orphan) AND `orphan_count > 0` (a COUNT query against `InboxThread.bucket_id == task.id`). The route's pre-existing idempotent-second-call branch (`get_owned_task_any_status`, for a repeat `DELETE` against an already-soft-deleted task) does **not** re-run the orphan check at all â€” deleting the same bucket twice can never create a second `delete_retriage` job, even if new threads somehow got assigned to the dead bucket's id in between (nothing writes `bucket_id` to a soft-deleted task's id under normal operation, so this is largely theoretical).
- **`_create_task_from_fields` is shared, but the enqueue/publish/response steps deliberately are not.** `POST /api/tasks` and `POST /api/jobs/{id}/confirm` both call the same helper for kind-aware 422 validation + criteria formulation + the `task_repo.create_task` insert, so the two entry points' validation rules can never drift â€” but each caller still does its OWN commit, its OWN `backfill_task.apply_async` (confirm passes a `job_id` kwarg; direct-create doesn't), and returns its OWN response shape (`_serialize_task_detail` alone vs. `{"task": ..., "job": ...}`). Confirm's task-row insert and its job-row `task_id`/`stage="backfilling"` update share ONE commit (folding them into two transactions would let a crash between them leave a job pointing at a task that doesn't durably exist yet, or vice versa) â€” `create_task` has no second row to fold in, so it just commits alone.
