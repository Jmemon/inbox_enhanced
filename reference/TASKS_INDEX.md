## The 8-step validator, in order (`transitions._process_one`)

Every proposal from `extract_transition.parse_response` runs through exactly these steps, in this order, and lands in exactly one of applied / pending_review / dropped. **Phase 2B Task 1** threads a `pending_reason: str|None` through steps 2/3/5/8 (replacing the old bare `forced_pending: bool`) Рђћ whichever guard is the FIRST to force pending wins and stamps its own reason string; every later guard's own force-pending branch is gated `if pending_reason is None` so it never overwrites an earlier reason. `proposed_entity` (the LLM's verbatim entity string) is set only alongside step 2's `near_duplicate_entity` reason. Both fields are written to the `pending_review` `TaskEvent` row only (`None`/`None` on `applied`).

1. **Shape.** `field=="stage"` Рєњ value must be in `schema.all_stages()`; else `schema.attr(field)` must resolve and `coerce_value` must succeed. Failure Рєњ dropped (malformed output is noise, never even reviewed).
   - **Step 1b Рђћ message reference (foreign-message-id guard).** `message_id` must resolve to an actual message in `message_by_gmail_id` (the just-fetched thread's own messages) Рђћ otherwise dropped before entity resolution even runs. Without this, a hallucinated/omitted `message_id` would silently resolve to `internal_message_id=None` downstream: wrong provenance in the audit log at step 5's fence check, and no dedup against a rerun of the same hallucinated citation at step 7 (`message_id=None` never collides).
2. **Entity resolution.** Singleton task Рєњ `get_or_create_entity('_self')`. Multi-entity: exact `normalize_key` match on the known-entities roster; else best `difflib` similarity Рђћ `>= 0.6` auto-matches; `[0.4, 0.6)` routes to the closest existing entity but **forces `pending_review`** (`pending_reason="near_duplicate_entity"`, `proposed_entity=` the LLM's raw entity string) regardless of the LLM's `is_new_entity` flag (a near-duplicate needs a human, not an auto-mint); `< 0.4` mints a **`PendingCreate` sentinel** (deferred Рђћ see Files) if `is_new_entity` was set, else drops (no entity, no flag Рєњ nothing to do). This is the first guard in the chain, so its `pending_reason` (when set) can never itself be overwritten by a later step Рђћ but per the paragraph above, no later step is allowed to overwrite it either way.
3. **Stage legality.** Only for `field=="stage"` **and only if no earlier guard already claimed `pending_reason`**: current stage already terminal Рєњ forces pending (`pending_reason="terminal_locked"`; only users move terminal entities out); a backward move (`all_stages.index(new) < index(current)`) Рєњ forces pending (`pending_reason="backward_move"`). Forward/lateral/skip moves are unrestricted (schema-level rule; this step is the validator's own backward/terminal gate).
4. **Evidence.** Whitespace-normalized `evidence_quote` must be an exact substring of the whitespace-normalized full thread text Рђћ else dropped outright (fail-closed; the cheapest hallucination guard, no pending-review fallback for a fabricated quote).
5. **Correction fences.** If the entity (real, not `PendingCreate` Рђћ a not-yet-created entity has no prior corrections to fence against by construction) has a `latest_applied_user_event`: the proposal's evidence message's `gmail_internal_date` must be strictly newer (ms-epoch, naive-as-UTC guard Рђћ see Files) than that event's `created_at`, else forces pending Рђћ `pending_reason="fence_blocked"`, but **only if `pending_reason` is still unset** from an earlier step.
6. **No-op.** `entity.state.get(field) == new_value` Рєњ dropped (not even reviewed Рђћ nothing changed).
7. **Idempotency.** SELECT-first check (`find_event_for_message_field(task_id, internal_message_id, field)`) Рђћ an existing row Рєњ dropped as a duplicate. The partial unique index `uq_task_event_msg_field` is only the race backstop for a genuinely concurrent insert on a **migrated** DB; it is absent from the `create_all` test fixture (raw SQL in the migration, not a SQLAlchemy `Index()` on the model) Рђћ this SELECT-first path is what tests actually exercise, and the index itself is verified by review, not a unit test.
8. **Confidence gate + deferred mint.** Only now Рђћ after every guard above passed Рђћ does a `PendingCreate` sentinel actually get materialized into a real `TaskStateEntity` row (`get_or_create_entity`), because both outcomes below need a real `entity_id`. Then: `not pending_reason and confidence >= settings.task_apply_confidence` (default 75) Рєњ `status="applied"` (`apply_event` folds it into `entity.state`, bumps `task.version`); else Рєњ `status="pending_review"`, and if no earlier step claimed a `pending_reason`, this branch defaults it to `"low_confidence"` (a plain confidence-gate deferral, nothing else forced it) Рђћ this is the ONLY step that can produce that particular reason. Both branches insert via `_append_event_guarded`, which wraps the flush in a `db.begin_nested()` SAVEPOINT and swallows `IntegrityError` as the same idempotency race backstop mentioned in step 7 (scoped to just this one insert, so a genuine race on one proposal can't poison every other proposal already flushed-but-uncommitted in the same `validate_and_stage` batch).

## Decision points & gotchas

- **Sticky user links.** `repo.upsert_link`'s one rule: an existing `origin='user'` row is never silently downgraded by an `origin='llm'` call (no-op, returns `None`). This is ALL the protection a user's explicit attach/detach decision needs against a later automatic reclassify or backfill re-run Рђћ there's deliberately no separate "is this link user-owned?" guard anywhere else.
- **Correction fences are strictly-newer, ms-epoch, naive-as-UTC-guarded.** `transitions._to_ms_epoch` treats a naive `created_at` as UTC before comparing Рђћ needed because SQLite (the test dialect) doesn't round-trip `tzinfo` through commit+`expire_on_commit`+reload, so a reloaded row's `created_at` can come back naive even though every write is `datetime.now(timezone.utc)`. No-op on Postgres (real `TIMESTAMPTZ`). The fence itself: after ANY user-origin applied event on an entity, only evidence **strictly** newer than that event may move it again Рђћ a message with `gmail_internal_date` equal to or older than the fence forces `pending_review`.
- **Refold semantics: user wins created_at ties.** `refold_entity` sorts an entity's applied events by `(created_at, 0 if origin=='llm' else 1)` before folding Рђћ so on an exact-timestamp tie, the llm-origin event folds in first and the user-origin event folds in **last**, winning. Fields with no surviving applied event (e.g. its one applied event got reverted/rejected) are dropped from `state` entirely, except `stage`, which always gets an explicit `None` default so the board can rely on the key existing.
- **Approval re-dates the event so user decisions survive refolds.** `POST .../events/{id}/approve` sets `event.created_at = datetime.now(timezone.utc)` immediately before `apply_event` Рђћ `refold_entity`'s own fold key is `created_at` ("assertion time," per the bullet above), so an approved event that was originally proposed well before other events later applied to the same entity would otherwise re-sort back to its old position on any future refold (a revert/detach/merge touching that entity) and lose the fold to a newer-but-since-approved-over applied event, silently undoing the user's explicit approval with no error and no signal. Re-dating means the approved event jumps to the top of the feed Рђћ its `created_at` now reads as the approval time, not the original proposal time; the original proposal time isn't lost, just no longer on this row directly Рђћ it's recoverable via a `message_id` join back to the source `InboxMessage.internal_date`. Gmail provenance fields (`message_id`/`gmail_message_id`/`evidence_quote`) are untouched Рђћ only the fold-ordering key moves.
- **Rejecting a minted new-entity pending deletes the orphaned empty entity.** Step 8 of the validator mints a real `TaskStateEntity` row even for a `pending_review` outcome (both the applied and pending branches need a real `entity_id` on the event they write) Рђћ if the rejected event was that entity's ONLY history and nothing was ever folded into its `state`, rejecting it would otherwise strand an empty, permanently-visible board row with no path to ever remove it. `reject_event` guards on `event.entity_id is not None` and calls `repo.delete_entity_if_orphaned(task_id=..., entity_id=event.entity_id, excluding_event_id=event.id)`, which hard-deletes the entity only if zero OTHER events reference it and `entity.state` carries no non-null values (`{}` and refold's own `{"stage": None}` default both count as empty). An entity with other history (an applied event, another still-pending proposal, a manual state edit) survives untouched. This doesn't bump `task.version` or need a dedicated SSE nudge beyond the `pending_count` change `_publish_task_updated` already carries Рђћ the client's `TaskDetail` already refetches the board on reject, so a vanished entity is simply absent from that refetch (see CLIENT_INDEX for the one not-explicitly-covered scenario: a drawer/deep-link open on that exact entity id at the moment of reject).
- **Confidence scales 0-100.** `TASK_APPLY_CONFIDENCE=75` (step 8's auto-apply gate) and `TASK_LINK_CONFIDENCE=60` (auto-link gate in `_write_task_links` and `backfill_task`'s tracker branch) are both `config.py` defaults, independently tunable. Note the two prompts' confidence handling differs: `extract_transition._parse_one` **drops the whole proposal item** if confidence isn't an exact int in `[0,100]`; `triage_thread.parse_response` **clamps** an out-of-range numeric confidence into `[0,100]` instead (only drops for non-numeric/bool/missing) Рђћ an LLM off-by-a-hair on a tracker score still counts, but a malformed extraction proposal doesn't get a guessed confidence.
- **Difflib entity similarity, NOT `pg_trgm`.** `ENTITY_MATCH_THRESHOLD=0.6` / `ENTITY_PENDING_THRESHOLD=0.4` are pure-Python `difflib.SequenceMatcher.ratio()` on `normalize_key`-normalized strings Рђћ a deliberate deviation from the spec's `pg_trgm` mention, so entity resolution is dialect-independent and unit-testable on the SQLite `create_all` fixture, which has no Postgres extensions at all.
- **The partial unique index only exists in migration-built DBs.** `uq_task_event_msg_field` (`task_id, message_id, field` WHERE `message_id IS NOT NULL`) is `op.execute` raw SQL in `0007_tasks.py`, not a SQLAlchemy `Index()` attached to the `TaskEvent` model Рђћ so `Base.metadata.create_all()` (the test conftest's fixture setup) never creates it. Step 7's SELECT-first check (`find_event_for_message_field`) is therefore the **only** guard the test suite exercises; the index itself is a race backstop for a genuinely concurrent duplicate insert on Postgres, verified by code review rather than a unit test (per `transitions._append_event_guarded`'s own comment).
- **Version bumps on every PATCH, attach, detach, and delete Рђћ but NOT on reject or criteria mutations.** `bump_version` is called by `apply_event`, `refold_entity`, PATCH's own explicit `bump_version` call, `attach_thread`, `detach_thread`, and `delete_task` (all unconditionally). `reject_event` does **not** bump Рђћ its immediate `task_updated` publish carries whatever version the task already had. **Confirmed post-2B:** `attach_thread`/`detach_thread`'s new `append_example` call (Phase 2B Task 2) mutates `task.criteria` in the same request but never touches `task.version` Рђћ a criteria-only change is invisible to `version`-based change detection entirely; only `pending_count` moving (see next bullet) or a later `bump_version`-calling operation would surface it. A client relying on "every `task_updated` implies a version change" would miss all of these paths Рђћ this is exactly why the 2B client's `TasksProvider` treats `pending_count` as an equally load-bearing second signal (see CLIENT_INDEX).
- **Detach auto-rejects that thread's still-pending events (Phase 2B Task 1), **and** detach bumps version unconditionally.** `DELETE /tasks/{id}/threads/{thread_id}` flips every `list_pending_events_for_thread` result to `status="rejected"` before its existing revert/refold logic runs (see api/tasks.py lines 473-476). This is deliberate Рђћ a detached thread's proposals citing evidence from a thread the task no longer tracks must not remain approvable. Detach also calls `bump_version` unconditionally (line 506), even if there were zero applied events to revert (so `refold_entity` never runs and would have bumped nothing). Rejected-via-detach events are indistinguishable on the wire from a manual reject (`status="rejected"`, no separate reason/marker) Рђћ `pending_reason` is preserved from before the flip (it was already set when the event was staged as pending_review) rather than cleared or overwritten.
- **Pending-reason vocabulary is fixed at 5 values, `proposed_entity` fires exactly once.** `pending_reason` is written by the validator's own guard chain only (`near_duplicate_entity`/`backward_move`/`terminal_locked`/`fence_blocked`/`low_confidence`) Рђћ the API layer never writes it (the one manual-state-edit event a user creates via `POST .../entities/{id}/state` is `status="applied"` immediately, so it never gets a `pending_reason` at all). `proposed_entity` is non-null in exactly one case (`pending_reason=="near_duplicate_entity"`) Рђћ for every other pending event (and every applied/rejected/reverted event) it's `None`, even if the underlying proposal DID reference an entity name (that name is simply discarded once resolution succeeds normally).
- **`extract_transition._parse_one`'s float-confidence clamp only helps a numeric-but-wrong-type response Рђћ a `bool` or truly out-of-range value still drops the whole proposal item.** Mirrors `triage_thread.parse_response`'s existing clamp (documented above), but the two prompts' contracts are otherwise unchanged: an out-of-range **int** confidence (e.g. `150`) is still dropped outright by `extract_transition`, only a **float** (`82.5`) gets rounded + clamped. `triage_thread` clamps both float AND out-of-range int alike Рђћ the two parsers are still NOT identical, just closer than before.
- **Learning-loop example cap is a silent FIFO eviction, and `_neutralize_tags` only protects the criteria text, not the classify prompt's own escaping (there is none Рђћ it never needed any).** `cap_examples` drops the OLDEST blocks once a criteria string exceeds `EXAMPLE_CAP=30` (positives+nearmisses combined, not per-tag) Рђћ a task that's been corrected 40 times over its lifetime silently loses its first 10 corrections' worth of criteria signal with no user-visible warning; only the newest 30 survive. `_neutralize_tags` exists solely so an email's own literal `<positive>`/`<nearmiss>`-shaped substrings can't be mistaken for real block markup by `cap_examples`'s regex (or by a human/LLM reader of the raw criteria text) Рђћ it does nothing to protect against prompt injection via the *content* of an example (the classify prompt already treats all criteria text as untrusted-but-literal, same as before this feature existed).
- **The 422-detail contract: every schema/manual-edit/merge validation failure is `HTTPException(422, str(exc))` Рђћ a bare string, never a structured body.** `PATCH /tasks/{id}` (state_schema re-validation, additive-change violations), `POST .../entities/{id}/state` (coerce_value failures), and `POST .../entities/{id}/merge` (self-merge) all raise this way; FastAPI serializes it as `{"detail": "<the string>"}`. The 2B client's `throwWithDetail` helper (`lib/api.ts`) is built specifically around this shape Рђћ see CLIENT_INDEX for the client-side contract and its one carve-out (a list-shaped `detail`, e.g. from pydantic body-level validation, falls back to a generic message instead of trying to render an array).
- **Backfill's "matched" count excludes already-attached-below-threshold threads Рђћ a landmine for a future re-backfill endpoint.** In `backfill_task` phase 1, a candidate whose THIS-run confidence falls below `TASK_LINK_CONFIDENCE` is skipped via `continue` **before** `upsert_link` is ever called Рђћ it is never added to `matched_thread_ids`, even if that thread already has an attached link from a previous run. Phase 2's extraction candidate set (`matched_threads`) is derived solely from `matched_thread_ids`, so such a thread is excluded from re-extraction THIS run too, despite still being attached in the DB. A future "re-run backfill" endpoint that assumes `matched` РЅѕ "everything currently attached" would silently skip these.
- **Triage replaced classify on the sync path (one call, dual-write); `classify()` itself is deleted (Phase 4).** `gmail_sync._triage_batch` (partial/full/extend) and `workers/task_engine_tasks._run_bucket_backfill`/`_run_tracker_backfill` (via `backfill_task`) call `llm.classify.triage()` Рђћ one Haiku call per thread returns both the bucket pick AND tracker relevance (D2: no doubled LLM volume vs. running two separate calls). The dual-write on the sync path: bucket_id feeds the existing upsert as before; every `(task_id, confidence) >= TASK_LINK_CONFIDENCE` tuple gets `upsert_link(origin="llm", state="attached")` in the same transaction. `classify()` Рђћ the pre-triage, bucket-only classifier `triage()` replaced back in Phase 2A Рђћ carried zero production call sites from that point on (kept only as a tested building block; `triage_thread`'s bucket-section rendering still delegates to `classify_thread` so the zero-tracker `triage()` case matches what `classify()` used to produce byte-for-byte) and is now gone outright; `triage()` is the only classify-stage entrypoint left in the codebase.
- **Propose retry symmetry: a 2-call cap, not "retry only on schema-invalid."** `propose_task_draft` spends **at most one retry total** per draft, no matter which of the two failure modes the first attempt hits (unparseable JSON Рђћ including what a transient API error degrades to, since `call_messages` returns `""` on any exception Рђћ vs. parseable-but-schema-invalid). Critically: if attempt 1 is unparseable and the retry comes back parseable-but-schema-invalid, that schema failure does **not** get its own second retry Рђћ it falls straight to `_FALLBACK_SCHEMA_DICT` rather than compounding into 3 LLM calls. At most 2 propose calls ever happen for one draft.
- **`GET /api/tasks/draft/{draft_id}` is the one 403 in an otherwise 404-only router.** Every other ownership check in this file (`_require_owned_task`, `_require_owned_event`) returns 404 for both "not yours" and "doesn't exist" (no enumeration split, matching `api/buckets.py`). The draft-poll endpoint instead returns 403 "not your draft" for a wrong-owner draft_id Рђћ an intentional asymmetry: draft ids are server-generated but pre-task-creation (per `draft_cache.py`'s own docstring, "we don't want to assume it's a secret"), a different threat model than a real task's id.
- **`stage_counts` key order is server-controlled and NOT alphabetical (Phase 3).** `_stage_counts` orders keys schema-stage-order-first (only when `state_schema` parses via `validate_schema`), then any observed stage values not in the schema (e.g. a stage a since-edited schema no longer declares), then a trailing `"(no stage)"` bucket for entities with `state.get("stage") is None` Рђћ stages with zero entities are omitted, so an empty task's `stage_counts` is `{}`. A schema-less task, or one whose `state_schema` fails `validate_schema` (hand-edited/migration-corrupted Рђћ the same failure mode `process_task_updates`'s per-task isolation already tolerates), falls back to observed order only. The client (`lib/api.ts`'s `TaskSummary.stage_counts` type comment, `HudPage.stageHistogramLabel`) must preserve this insertion order via plain `Object.entries` Рђћ sorting it would put `"(no stage)"` in the wrong place and scramble the schema's own stage sequence.
- **`GET /api/reviews`/`GET /api/activity` are the one place cross-task data leaves task-scoped ownership checks (Phase 3) Рђћ treat any change here as a security boundary, not a convenience query.** Every other read/write in this router is scoped by `get_owned_task`'s single-task 404 check; these two routes instead join directly on `Task.user_id == user.id` and `Task.is_deleted == False` (`repo.py`'s own module comment makes this explicit) to aggregate across ALL of a user's tasks in one query Рђћ a mistake here (e.g. a missing `Task.user_id` filter) would leak one user's email-derived event text/quotes to another. Both routes clamp `limit` in Python (`max(min(...))`) rather than reject an out-of-range value Рђћ a deliberate departure from `GET /tasks/{id}/events`'s own `limit` param, which is a declarative `Query(ge=1, le=200)` and DOES 422 on a violation; don't assume the two `limit` params share validation behavior.
- **Bucket unification is an ID-preserving migration, not a re-key (Phase 4).** `0009_bucket_unification.py` copies every `buckets` row into `tasks` under the SAME `id`, so `InboxThread.bucket_id` values are never rewritten anywhere Рђћ only the column's FK target changes (Postgres-only; SQLite never enforced it). This is why the migration is a single raw-SQL `INSERT ... SELECT` with no id-remapping step and no data migration needed on `inbox_threads` itself. A "default bucket" (`user_id IS NULL`) is now indistinguishable, at the row level, from any other `kind='bucket'` task except by that one column Рђћ there is no separate "is this a default" flag.
- **`GET /api/tasks` with no `kind` param is now an explicit tracker-only filter, not an accidental one.** Before Phase 4, `list_tasks` had nothing else to filter Рђћ every row already was a tracker. Post-unification, the same endpoint's default (`kind=None` Рєњ `list_tasks(kind="tracker")` in `api/tasks.py`) is load-bearing: without it, `GET /api/tasks` would return bucket rows too and buckets would leak into the HUD's task grid (`HudPage`'s task cards, `TasksProvider`'s list) alongside real trackers. Any future change to this default must preserve the tracker-only behavior explicitly, not rely on "there's nothing else in the table" being true anymore.
- **Bucket-kind tasks bump `version` and publish `task_updated` only through `/api/tasks` (never the shim).** The client mutates buckets exclusively through `PATCH/DELETE /api/tasks/{id}` (via `useBuckets`' `rename`/`softDelete` calling `patchTask`/`deleteTask`), which unconditionally bump version and publish on every successful call, tracker OR bucket (nothing in `_require_owned_task`/those two routes filters by `kind`). The `/api/buckets` shim (`api/buckets.py`'s `patch_bucket`/`delete_bucket`, thin wrappers over `bucket_repo.rename`/`bucket_repo.soft_delete`) exists only for stale pre-deploy tabs and is deleted in Phase 5 Рђћ it is unreachable from the current client. Don't assume any backward-looking code meant to read bucket mutations from `task_updated` consumers (e.g. monitoring dashboards or other consumers) will see them if they still target the shim path rather than the real routes.
- **A bucket's own criteria can never be edited or grown after creation via any current route.** `_PatchBody` in `api/buckets.py` is name-only (matching the old bucket shape); `_CreateTaskBody`'s learning-loop `add_example` machinery (attach/detach appending to `criteria`) is wired only through `POST/DELETE /tasks/{id}/threads`, which nothing calls for a bucket-kind task (buckets have no `ThreadsPanel`/attach-detach UI). A bucket's `criteria` is therefore fixed at creation time (from the goalРєњdraft wizard's confirmed positives/negatives) for its entire lifetime Рђћ the only way its threads get re-evaluated is a full `_run_bucket_backfill` re-run, and there is currently no endpoint that re-triggers one after creation (creating a NEW bucket re-triages every EXISTING bucket's candidate pool too, since `_run_bucket_backfill` triages against the full active set Рђћ but an existing bucket's own criteria text never changes).
- **Trigger changed: bucket creation now runs `backfill_task`, not the deleted `reclassify_user_inbox`.** Before Phase 4, `POST /api/buckets` fired `workers.tasks.reclassify_user_inbox` (a bespoke retriage-everything Celery task, ~110s for a large inbox). That task and its `_reclassify_all`/`_inline_reload` helpers are gone; both `POST /api/buckets` (the shim) and `POST /api/tasks` with `kind="bucket"` now enqueue the same `backfill_task` (kind='bucket' branch) that a new tracker's creation always used. Functionally similar (retriage the user's stored history against the full classify-relevant set) but the publish contract differs Рђћ see the bullet above and WORKERS_INDEX/INBOX_SYNC_INDEX for the full before/after.
