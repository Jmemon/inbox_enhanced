# Tasks Index

> Scope: Phase 2A task engine — user-defined "trackers" that watch inbox threads
> for progress toward a goal. EPS (Entity-Pipeline Schema) contract
> (`task_engine/schema.py`); CRUD/append-only-event repo (`task_engine/repo.py`,
> never commits); the pure 8-step extraction validator (`task_engine/
> transitions.py`, no LLM/DB-write side effects of its own); the one-LLM-call
> orchestrator (`task_engine/engine.py`); goal→draft cache (`task_engine/
> draft_cache.py`); criteria-text grammar shared with buckets (`task_engine/
> criteria.py`); the triage LLM prompt that fused bucket-classify + tracker-
> relevance into one call (`llm/prompts/triage_thread.py`, replacing `classify()`
> on the sync path — D2, no doubled LLM volume) plus the extraction/propose
> prompts (`llm/prompts/{extract_transition,propose_task}.py`); the decoupled
> Celery module (`workers/task_engine_tasks.py`: process_task_updates,
> extract_for_thread, propose_task_draft, backfill_task); the full HTTP surface
> (`api/tasks.py`: goal→draft, tracker CRUD, board/events feed, attach/detach,
> approve/reject/revert corrections, manual state edit, entity merge); and the
> `task_draft_ready`/`task_updated`/`task_backfill_progress` SSE vocabulary.
> Postgres tables `tasks`/`task_thread_links`/`task_state_entities`/`task_events`
> (migration `0007_tasks.py`). Cross-refs: WORKERS_INDEX (celery/LLM plumbing
> this reuses), INBOX_SYNC_INDEX (the sync path that feeds `process_task_updates`).
>
> **Phase 2B additions** (server-side; the client surface built on top of
> this API lives in CLIENT_INDEX): pending-review **provenance** — every
> `pending_review` `TaskEvent` now records WHY (`pending_reason` — a 5-value
> enum set by the validator's own guard chain: `near_duplicate_entity`/
> `backward_move`/`terminal_locked`/`fence_blocked`/`low_confidence`, first
> guard to fire wins) and, for the near-duplicate case, WHAT the LLM actually
> proposed (`proposed_entity`, its verbatim entity string) — new nullable
> `task_events` columns (migration `0008_pending_reason.py`), serialized by
> `api/tasks.py`'s `_serialize_event` and surfaced by the client's review
> tray. Detaching a thread (`DELETE /tasks/{id}/threads/{thread_id}`) now
> also auto-rejects that thread's still-`pending_review` events (they were
> never folded into any `entity.state`, so there's nothing to refold — just
> a status flip) so a removed thread's unreviewed proposals can't be
> approved later. `extract_transition.parse_response` now accepts a
> **float** confidence (rounds + clamps to `[0,100]`, parity with
> `triage_thread`'s existing clamp) instead of dropping the whole proposal
> item over a formatting quirk. The spec §4.6 **learning loop**
> (`task_engine/criteria.py`'s new `append_example`/`cap_examples`/
> `EXAMPLE_CAP=30`, `_neutralize_tags` injection guard): an explicit
> attach/detach (`add_example: bool` body/query param, default `True`) feeds
> the thread's most-recent message back into the task's own `criteria` text
> as one more tagged `<positive>`/`<nearmiss>` block, FIFO-capped at 30
> examples total; `task_engine.repo.recent_user_events` + `engine.
> extract_for_pair` additionally surface the task's 5 most recent human
> corrections directly in the extraction prompt itself (`extract_transition.
> build_user_message`'s new `user_corrections` param) so the LLM is
> discouraged from proposing to relitigate one, not just fenced out after
> the fact by step 5. All of the above 422s (`state_schema`/manual-edit/
> merge validation failures) follow one contract — `HTTPException(422,
> str(exc))`, a plain-string FastAPI `detail` body — which the 2B client
> reads via a shared `throwWithDetail` helper (`lib/api.ts`) rather than a
> generic status message.

## Routes / Tasks / Entrypoints

### HTTP API (`app.api.tasks`, all `/api/tasks*`)

| Route | Status | Behavior |
|-------|--------|----------|

## The 8-step validator, in order (`transitions._process_one`)

Every proposal from `extract_transition.parse_response` runs through exactly these steps, in this order, and lands in exactly one of applied / pending_review / dropped. **Phase 2B Task 1** threads a `pending_reason: str|None` through steps 2/3/5/8 (replacing the old bare `forced_pending: bool`) — whichever guard is the FIRST to force pending wins and stamps its own reason string; every later guard's own force-pending branch is gated `if pending_reason is None` so it never overwrites an earlier reason. `proposed_entity` (the LLM's verbatim entity string) is set only alongside step 2's `near_duplicate_entity` reason. Both fields are written to the `pending_review` `TaskEvent` row only (`None`/`None` on `applied`).

1. **Shape.** `field=="stage"` → value must be in `schema.all_stages()`; else `schema.attr(field)` must resolve and `coerce_value` must succeed. Failure → dropped (malformed output is noise, never even reviewed).
   - **Step 1b — message reference (foreign-message-id guard).** `message_id` must resolve to an actual message in `message_by_gmail_id` (the just-fetched thread's own messages) — otherwise dropped before entity resolution even runs. Without this, a hallucinated/omitted `message_id` would silently resolve to `internal_message_id=None` downstream: wrong provenance in the audit log at step 5's fence check, and no dedup against a rerun of the same hallucinated citation at step 7 (`message_id=None` never collides).
2. **Entity resolution.** Singleton task → `get_or_create_entity('_self')`. Multi-entity: exact `normalize_key` match on the known-entities roster; else best `difflib` similarity — `>= 0.6` auto-matches; `[0.4, 0.6)` routes to the closest existing entity but **forces `pending_review`** (`pending_reason="near_duplicate_entity"`, `proposed_entity=` the LLM's raw entity string) regardless of the LLM's `is_new_entity` flag (a near-duplicate needs a human, not an auto-mint); `< 0.4` mints a **`PendingCreate` sentinel** (deferred — see Files) if `is_new_entity` was set, else drops (no entity, no flag → nothing to do). This is the first guard in the chain, so its `pending_reason` (when set) can never itself be overwritten by a later step — but per the paragraph above, no later step is allowed to overwrite it either way.
3. **Stage legality.** Only for `field=="stage"` **and only if no earlier guard already claimed `pending_reason`**: current stage already terminal → forces pending (`pending_reason="terminal_locked"`; only users move terminal entities out); a backward move (`all_stages.index(new) < index(current)`) → forces pending (`pending_reason="backward_move"`). Forward/lateral/skip moves are unrestricted (schema-level rule; this step is the validator's own backward/terminal gate).
4. **Evidence.** Whitespace-normalized `evidence_quote` must be an exact substring of the whitespace-normalized full thread text — else dropped outright (fail-closed; the cheapest hallucination guard, no pending-review fallback for a fabricated quote).
5. **Correction fences.** If the entity (real, not `PendingCreate` — a not-yet-created entity has no prior corrections to fence against by construction) has a `latest_applied_user_event`: the proposal's evidence message's `gmail_internal_date` must be strictly newer (ms-epoch, naive-as-UTC guard — see Files) than that event's `created_at`, else forces pending — `pending_reason="fence_blocked"`, but **only if `pending_reason` is still unset** from an earlier step.
6. **No-op.** `entity.state.get(field) == new_value` → dropped (not even reviewed — nothing changed).
7. **Idempotency.** SELECT-first check (`find_event_for_message_field(task_id, internal_message_id, field)`) — an existing row → dropped as a duplicate. The partial unique index `uq_task_event_msg_field` is only the race backstop for a genuinely concurrent insert on a **migrated** DB; it is absent from the `create_all` test fixture (raw SQL in the migration, not a SQLAlchemy `Index()` on the model) — this SELECT-first path is what tests actually exercise, and the index itself is verified by review, not a unit test.
8. **Confidence gate + deferred mint.** Only now — after every guard above passed — does a `PendingCreate` sentinel actually get materialized into a real `TaskStateEntity` row (`get_or_create_entity`), because both outcomes below need a real `entity_id`. Then: `not pending_reason and confidence >= settings.task_apply_confidence` (default 75) → `status="applied"` (`apply_event` folds it into `entity.state`, bumps `task.version`); else → `status="pending_review"`, and if no earlier step claimed a `pending_reason`, this branch defaults it to `"low_confidence"` (a plain confidence-gate deferral, nothing else forced it) — this is the ONLY step that can produce that particular reason. Both branches insert via `_append_event_guarded`, which wraps the flush in a `db.begin_nested()` SAVEPOINT and swallows `IntegrityError` as the same idempotency race backstop mentioned in step 7 (scoped to just this one insert, so a genuine race on one proposal can't poison every other proposal already flushed-but-uncommitted in the same `validate_and_stage` batch).

## Decision points & gotchas

- **Sticky user links.** `repo.upsert_link`'s one rule: an existing `origin='user'` row is never silently downgraded by an `origin='llm'` call (no-op, returns `None`). This is ALL the protection a user's explicit attach/detach decision needs against a later automatic reclassify or backfill re-run — there's deliberately no separate "is this link user-owned?" guard anywhere else.
- **Correction fences are strictly-newer, ms-epoch, naive-as-UTC-guarded.** `transitions._to_ms_epoch` treats a naive `created_at` as UTC before comparing — needed because SQLite (the test dialect) doesn't round-trip `tzinfo` through commit+`expire_on_commit`+reload, so a reloaded row's `created_at` can come back naive even though every write is `datetime.now(timezone.utc)`. No-op on Postgres (real `TIMESTAMPTZ`). The fence itself: after ANY user-origin applied event on an entity, only evidence **strictly** newer than that event may move it again — a message with `gmail_internal_date` equal to or older than the fence forces `pending_review`.
- **Refold semantics: user wins created_at ties.** `refold_entity` sorts an entity's applied events by `(created_at, 0 if origin=='llm' else 1)` before folding — so on an exact-timestamp tie, the llm-origin event folds in first and the user-origin event folds in **last**, winning. Fields with no surviving applied event (e.g. its one applied event got reverted/rejected) are dropped from `state` entirely, except `stage`, which always gets an explicit `None` default so the board can rely on the key existing.
- **Confidence scales 0-100.** `TASK_APPLY_CONFIDENCE=75` (step 8's auto-apply gate) and `TASK_LINK_CONFIDENCE=60` (auto-link gate in `_write_task_links`/`_reclassify_all`/`backfill_task`) are both `config.py` defaults, independently tunable. Note the two prompts' confidence handling differs: `extract_transition._parse_one` **drops the whole proposal item** if confidence isn't an exact int in `[0,100]`; `triage_thread.parse_response` **clamps** an out-of-range numeric confidence into `[0,100]` instead (only drops for non-numeric/bool/missing) — an LLM off-by-a-hair on a tracker score still counts, but a malformed extraction proposal doesn't get a guessed confidence.
- **Difflib entity similarity, NOT `pg_trgm`.** `ENTITY_MATCH_THRESHOLD=0.6` / `ENTITY_PENDING_THRESHOLD=0.4` are pure-Python `difflib.SequenceMatcher.ratio()` on `normalize_key`-normalized strings — a deliberate deviation from the spec's `pg_trgm` mention, so entity resolution is dialect-independent and unit-testable on the SQLite `create_all` fixture, which has no Postgres extensions at all.
- **The partial unique index only exists in migration-built DBs.** `uq_task_event_msg_field` (`task_id, message_id, field` WHERE `message_id IS NOT NULL`) is `op.execute` raw SQL in `0007_tasks.py`, not a SQLAlchemy `Index()` attached to the `TaskEvent` model — so `Base.metadata.create_all()` (the test conftest's fixture setup) never creates it. Step 7's SELECT-first check (`find_event_for_message_field`) is therefore the **only** guard the test suite exercises; the index itself is a race backstop for a genuinely concurrent duplicate insert on Postgres, verified by code review rather than a unit test (per `transitions._append_event_guarded`'s own comment).
- **Version bumps on every PATCH, attach, detach, and delete — but NOT on reject or criteria mutations.** `bump_version` is called by `apply_event`, `refold_entity`, PATCH's own explicit `bump_version` call, `attach_thread`, `detach_thread`, and `delete_task` (all unconditionally). `reject_event` does **not** bump — its immediate `task_updated` publish carries whatever version the task already had. **Confirmed post-2B:** `attach_thread`/`detach_thread`'s new `append_example` call (Phase 2B Task 2) mutates `task.criteria` in the same request but never touches `task.version` — a criteria-only change is invisible to `version`-based change detection entirely; only `pending_count` moving (see next bullet) or a later `bump_version`-calling operation would surface it. A client relying on "every `task_updated` implies a version change" would miss all of these paths — this is exactly why the 2B client's `TasksProvider` treats `pending_count` as an equally load-bearing second signal (see CLIENT_INDEX).
- **Detach auto-rejects that thread's still-pending events (Phase 2B Task 1), **and** detach bumps version unconditionally.** `DELETE /tasks/{id}/threads/{thread_id}` flips every `list_pending_events_for_thread` result to `status="rejected"` before its existing revert/refold logic runs (see api/tasks.py lines 473-476). This is deliberate — a detached thread's proposals citing evidence from a thread the task no longer tracks must not remain approvable. Detach also calls `bump_version` unconditionally (line 506), even if there were zero applied events to revert (so `refold_entity` never runs and would have bumped nothing). Rejected-via-detach events are indistinguishable on the wire from a manual reject (`status="rejected"`, no separate reason/marker) — `pending_reason` is preserved from before the flip (it was already set when the event was staged as pending_review) rather than cleared or overwritten.
- **Pending-reason vocabulary is fixed at 5 values, `proposed_entity` fires exactly once.** `pending_reason` is written by the validator's own guard chain only (`near_duplicate_entity`/`backward_move`/`terminal_locked`/`fence_blocked`/`low_confidence`) — the API layer never writes it (the one manual-state-edit event a user creates via `POST .../entities/{id}/state` is `status="applied"` immediately, so it never gets a `pending_reason` at all). `proposed_entity` is non-null in exactly one case (`pending_reason=="near_duplicate_entity"`) — for every other pending event (and every applied/rejected/reverted event) it's `None`, even if the underlying proposal DID reference an entity name (that name is simply discarded once resolution succeeds normally).
- **`extract_transition._parse_one`'s float-confidence clamp only helps a numeric-but-wrong-type response — a `bool` or truly out-of-range value still drops the whole proposal item.** Mirrors `triage_thread.parse_response`'s existing clamp (documented above), but the two prompts' contracts are otherwise unchanged: an out-of-range **int** confidence (e.g. `150`) is still dropped outright by `extract_transition`, only a **float** (`82.5`) gets rounded + clamped. `triage_thread` clamps both float AND out-of-range int alike — the two parsers are still NOT identical, just closer than before.
- **Learning-loop example cap is a silent FIFO eviction, and `_neutralize_tags` only protects the criteria text, not the classify prompt's own escaping (there is none — it never needed any).** `cap_examples` drops the OLDEST blocks once a criteria string exceeds `EXAMPLE_CAP=30` (positives+nearmisses combined, not per-tag) — a task that's been corrected 40 times over its lifetime silently loses its first 10 corrections' worth of criteria signal with no user-visible warning; only the newest 30 survive. `_neutralize_tags` exists solely so an email's own literal `<positive>`/`<nearmiss>`-shaped substrings can't be mistaken for real block markup by `cap_examples`'s regex (or by a human/LLM reader of the raw criteria text) — it does nothing to protect against prompt injection via the *content* of an example (the classify prompt already treats all criteria text as untrusted-but-literal, same as before this feature existed).
- **The 422-detail contract: every schema/manual-edit/merge validation failure is `HTTPException(422, str(exc))` — a bare string, never a structured body.** `PATCH /tasks/{id}` (state_schema re-validation, additive-change violations), `POST .../entities/{id}/state` (coerce_value failures), and `POST .../entities/{id}/merge` (self-merge) all raise this way; FastAPI serializes it as `{"detail": "<the string>"}`. The 2B client's `throwWithDetail` helper (`lib/api.ts`) is built specifically around this shape — see CLIENT_INDEX for the client-side contract and its one carve-out (a list-shaped `detail`, e.g. from pydantic body-level validation, falls back to a generic message instead of trying to render an array).
- **Backfill's "matched" count excludes already-attached-below-threshold threads — a landmine for a future re-backfill endpoint.** In `backfill_task` phase 1, a candidate whose THIS-run confidence falls below `TASK_LINK_CONFIDENCE` is skipped via `continue` **before** `upsert_link` is ever called — it is never added to `matched_thread_ids`, even if that thread already has an attached link from a previous run. Phase 2's extraction candidate set (`matched_threads`) is derived solely from `matched_thread_ids`, so such a thread is excluded from re-extraction THIS run too, despite still being attached in the DB. A future "re-run backfill" endpoint that assumes `matched` ≈ "everything currently attached" would silently skip these.
- **Triage replaces classify on the sync path (one call, dual-write); `classify()` has no production callers left.** `gmail_sync._triage_batch` (partial/full/extend) and `workers/tasks._reclassify_all` call `llm.classify.triage()` — one Haiku call per thread returns both the bucket pick AND tracker relevance (D2: no doubled LLM volume vs. running two separate calls). The dual-write: bucket_id feeds the existing upsert as before; every `(task_id, confidence) >= TASK_LINK_CONFIDENCE` tuple gets `upsert_link(origin="llm", state="attached")` in the same transaction. `classify()` itself is behaviorally unchanged (same signature, own test file) but its docstring now states the deprecation explicitly: zero production call sites as of Phase 2A, Phase 4 removes it — kept meanwhile as a tested building block (`triage_thread`'s bucket-section rendering is delegated to `classify_thread` specifically so the zero-tracker case still matches it byte-for-byte), not because a live production caller depends on it.
- **Propose retry symmetry: a 2-call cap, not "retry only on schema-invalid."** `propose_task_draft` spends **at most one retry total** per draft, no matter which of the two failure modes the first attempt hits (unparseable JSON — including what a transient API error degrades to, since `call_messages` returns `""` on any exception — vs. parseable-but-schema-invalid). Critically: if attempt 1 is unparseable and the retry comes back parseable-but-schema-invalid, that schema failure does **not** get its own second retry — it falls straight to `_FALLBACK_SCHEMA_DICT` rather than compounding into 3 LLM calls. At most 2 propose calls ever happen for one draft.
- **`GET /api/tasks/draft/{draft_id}` is the one 403 in an otherwise 404-only router.** Every other ownership check in this file (`_require_owned_task`, `_require_owned_event`) returns 404 for both "not yours" and "doesn't exist" (no enumeration split, matching `api/buckets.py`). The draft-poll endpoint instead returns 403 "not your draft" for a wrong-owner draft_id — an intentional asymmetry: draft ids are server-generated but pre-task-creation (per `draft_cache.py`'s own docstring, "we don't want to assume it's a secret"), a different threat model than a real task's id.
