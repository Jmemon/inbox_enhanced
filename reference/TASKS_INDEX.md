> for progress toward a goal. EPS (Entity-Pipeline Schema) contract

## The 8-step validator, in order (`transitions._process_one`)

Every proposal from `extract_transition.parse_response` runs through exactly these steps, in this order, and lands in exactly one of applied / pending_review / dropped:

1. **Shape.** `field=="stage"` → value must be in `schema.all_stages()`; else `schema.attr(field)` must resolve and `coerce_value` must succeed. Failure → dropped (malformed output is noise, never even reviewed).
   - **Step 1b — message reference (foreign-message-id guard).** `message_id` must resolve to an actual message in `message_by_gmail_id` (the just-fetched thread's own messages) — otherwise dropped before entity resolution even runs. Without this, a hallucinated/omitted `message_id` would silently resolve to `internal_message_id=None` downstream: wrong provenance in the audit log at step 5's fence check, and no dedup against a rerun of the same hallucinated citation at step 7 (`message_id=None` never collides).
2. **Entity resolution.** Singleton task → `get_or_create_entity('_self')`. Multi-entity: exact `normalize_key` match on the known-entities roster; else best `difflib` similarity — `>= 0.6` auto-matches; `[0.4, 0.6)` routes to the closest existing entity but **forces `pending_review`** regardless of the LLM's `is_new_entity` flag (a near-duplicate needs a human, not an auto-mint); `< 0.4` mints a **`PendingCreate` sentinel** (deferred — see Files) if `is_new_entity` was set, else drops (no entity, no flag → nothing to do).
3. **Stage legality.** Only for `field=="stage"`: current stage already terminal → forces pending (only users move terminal entities out); a backward move (`all_stages.index(new) < index(current)`) → forces pending. Forward/lateral/skip moves are unrestricted (schema-level rule; this step is the validator's own backward/terminal gate).
4. **Evidence.** Whitespace-normalized `evidence_quote` must be an exact substring of the whitespace-normalized full thread text — else dropped outright (fail-closed; the cheapest hallucination guard, no pending-review fallback for a fabricated quote).
5. **Correction fences.** If the entity (real, not `PendingCreate` — a not-yet-created entity has no prior corrections to fence against by construction) has a `latest_applied_user_event`: the proposal's evidence message's `gmail_internal_date` must be strictly newer (ms-epoch, naive-as-UTC guard — see Files) than that event's `created_at`, else forces pending.
6. **No-op.** `entity.state.get(field) == new_value` → dropped (not even reviewed — nothing changed).
7. **Idempotency.** SELECT-first check (`find_event_for_message_field(task_id, internal_message_id, field)`) — an existing row → dropped as a duplicate. The partial unique index `uq_task_event_msg_field` is only the race backstop for a genuinely concurrent insert on a **migrated** DB; it is absent from the `create_all` test fixture (raw SQL in the migration, not a SQLAlchemy `Index()` on the model) — this SELECT-first path is what tests actually exercise, and the index itself is verified by review, not a unit test.
8. **Confidence gate + deferred mint.** Only now — after every guard above passed — does a `PendingCreate` sentinel actually get materialized into a real `TaskStateEntity` row (`get_or_create_entity`), because both outcomes below need a real `entity_id`. Then: `not forced_pending and confidence >= settings.task_apply_confidence` (default 75) → `status="applied"` (`apply_event` folds it into `entity.state`, bumps `task.version`); else → `status="pending_review"`. Both branches insert via `_append_event_guarded`, which wraps the flush in a `db.begin_nested()` SAVEPOINT and swallows `IntegrityError` as the same idempotency race backstop mentioned in step 7 (scoped to just this one insert, so a genuine race on one proposal can't poison every other proposal already flushed-but-uncommitted in the same `validate_and_stage` batch).

## Decision points & gotchas

- **Sticky user links.** `repo.upsert_link`'s one rule: an existing `origin='user'` row is never silently downgraded by an `origin='llm'` call (no-op, returns `None`). This is ALL the protection a user's explicit attach/detach decision needs against a later automatic reclassify or backfill re-run — there's deliberately no separate "is this link user-owned?" guard anywhere else.
- **Correction fences are strictly-newer, ms-epoch, naive-as-UTC-guarded.** `transitions._to_ms_epoch` treats a naive `created_at` as UTC before comparing — needed because SQLite (the test dialect) doesn't round-trip `tzinfo` through commit+`expire_on_commit`+reload, so a reloaded row's `created_at` can come back naive even though every write is `datetime.now(timezone.utc)`. No-op on Postgres (real `TIMESTAMPTZ`). The fence itself: after ANY user-origin applied event on an entity, only evidence **strictly** newer than that event may move it again — a message with `gmail_internal_date` equal to or older than the fence forces `pending_review`.
- **Refold semantics: user wins created_at ties.** `refold_entity` sorts an entity's applied events by `(created_at, 0 if origin=='llm' else 1)` before folding — so on an exact-timestamp tie, the llm-origin event folds in first and the user-origin event folds in **last**, winning. Fields with no surviving applied event (e.g. its one applied event got reverted/rejected) are dropped from `state` entirely, except `stage`, which always gets an explicit `None` default so the board can rely on the key existing.
- **Confidence scales 0-100.** `TASK_APPLY_CONFIDENCE=75` (step 8's auto-apply gate) and `TASK_LINK_CONFIDENCE=60` (auto-link gate in `_write_task_links`/`_reclassify_all`/`backfill_task`) are both `config.py` defaults, independently tunable. Note the two prompts' confidence handling differs: `extract_transition._parse_one` **drops the whole proposal item** if confidence isn't an exact int in `[0,100]`; `triage_thread.parse_response` **clamps** an out-of-range numeric confidence into `[0,100]` instead (only drops for non-numeric/bool/missing) — an LLM off-by-a-hair on a tracker score still counts, but a malformed extraction proposal doesn't get a guessed confidence.
- **Difflib entity similarity, NOT `pg_trgm`.** `ENTITY_MATCH_THRESHOLD=0.6` / `ENTITY_PENDING_THRESHOLD=0.4` are pure-Python `difflib.SequenceMatcher.ratio()` on `normalize_key`-normalized strings — a deliberate deviation from the spec's `pg_trgm` mention, so entity resolution is dialect-independent and unit-testable on the SQLite `create_all` fixture, which has no Postgres extensions at all.
- **The partial unique index only exists in migration-built DBs.** `uq_task_event_msg_field` (`task_id, message_id, field` WHERE `message_id IS NOT NULL`) is `op.execute` raw SQL in `0007_tasks.py`, not a SQLAlchemy `Index()` attached to the `TaskEvent` model — so `Base.metadata.create_all()` (the test conftest's fixture setup) never creates it. Step 7's SELECT-first check (`find_event_for_message_field`) is therefore the **only** guard the test suite exercises; the index itself is a race backstop for a genuinely concurrent duplicate insert on Postgres, verified by code review rather than a unit test (per `transitions._append_event_guarded`'s own comment).
- **Version bumps on every PATCH but NOT on reject/attach/DELETE.** `bump_version` is called by `apply_event`, `refold_entity`, and PATCH's own explicit `bump_version` call (added specifically so a name/status-only PATCH — which touches neither — still produces a version the 2B client's version-gap "refetch only if newer" SSE logic will act on). `reject_event`, `attach_thread`, and `delete_task` (soft-delete) call **none** of those — their immediate `task_updated` publish carries whatever version the task already had. (`detach_thread` is the middle case: it bumps only if it actually reverted ≥1 applied event whose entity refolded; a detach of a thread with zero applied events bumps nothing.) A client relying on "every `task_updated` implies a version change" would miss these three paths.
- **Backfill's "matched" count excludes already-attached-below-threshold threads — a landmine for a future re-backfill endpoint.** In `backfill_task` phase 1, a candidate whose THIS-run confidence falls below `TASK_LINK_CONFIDENCE` is skipped via `continue` **before** `upsert_link` is ever called — it is never added to `matched_thread_ids`, even if that thread already has an attached link from a previous run. Phase 2's extraction candidate set (`matched_threads`) is derived solely from `matched_thread_ids`, so such a thread is excluded from re-extraction THIS run too, despite still being attached in the DB. A future "re-run backfill" endpoint that assumes `matched` ≈ "everything currently attached" would silently skip these.
- **Triage replaces classify on the sync path (one call, dual-write); `classify()` has no production callers left.** `gmail_sync._triage_batch` (partial/full/extend) and `workers/tasks._reclassify_all` call `llm.classify.triage()` — one Haiku call per thread returns both the bucket pick AND tracker relevance (D2: no doubled LLM volume vs. running two separate calls). The dual-write: bucket_id feeds the existing upsert as before; every `(task_id, confidence) >= TASK_LINK_CONFIDENCE` tuple gets `upsert_link(origin="llm", state="attached")` in the same transaction. `classify()` itself is untouched (same signature, own test file) but as of this stamp is called by nothing in `server/app` — kept as a tested building block (`triage_thread`'s bucket-section rendering is delegated to `classify_thread` specifically so the zero-tracker case still matches it byte-for-byte), not because a live production caller depends on it.
- **Propose retry symmetry: a 2-call cap, not "retry only on schema-invalid."** `propose_task_draft` spends **at most one retry total** per draft, no matter which of the two failure modes the first attempt hits (unparseable JSON — including what a transient API error degrades to, since `call_messages` returns `""` on any exception — vs. parseable-but-schema-invalid). Critically: if attempt 1 is unparseable and the retry comes back parseable-but-schema-invalid, that schema failure does **not** get its own second retry — it falls straight to `_FALLBACK_SCHEMA_DICT` rather than compounding into 3 LLM calls. At most 2 propose calls ever happen for one draft.
- **`GET /api/tasks/draft/{draft_id}` is the one 403 in an otherwise 404-only router.** Every other ownership check in this file (`_require_owned_task`, `_require_owned_event`) returns 404 for both "not yours" and "doesn't exist" (no enumeration split, matching `api/buckets.py`). The draft-poll endpoint instead returns 403 "not your draft" for a wrong-owner draft_id — an intentional asymmetry: draft ids are server-generated but pre-task-creation (per `draft_cache.py`'s own docstring, "we don't want to assume it's a secret"), a different threat model than a real task's id.
