# Client Index

> Scope: React 19 + Vite browser SPA (client/src) — react-router-dom routing (`Gate`→`BrowserRouter`, `AppShell` layout route, `/` HudPage, `/inbox` InboxPage, `/tasks/:taskId` TaskDetail), App/AuthProvider, lib/api fetch wrappers (+ `searchInbox`, `getSyncStatus`, and Phase 2B's full task-engine fetcher set) + lib/sse EventSource singleton (now also carrying the three task SSE events), `state/InboxProvider.tsx` — a shell-level context (`useInboxStore()`) composing `useBuckets`/`useInbox`/`useInboxSse` **once** for the life of the authed session so `/` and `/inbox` read and write one live store instead of each route remounting its own copy (its permanent `useInboxSse` subscription also keeps the SSE singleton pinned open, superseding the old `AppShell` no-op pin), `state/TasksProvider.tsx` (**new**, Phase 2B) — the analogous shell-level task store (`useTasksStore()`), nested inside `InboxProvider` in `AppShell`, driving both the HUD's task cards and `/tasks/:taskId`'s detail page off one shared list+per-task-detail cache and one SSE subscription, shared debounced inbox search (`pages/search/`) used by both inbox-facing routes, HudPage's store-derived bucket counts + sync-freshness strip + (**new**) task cards + new-task entry point, inbox list + pagination + reload, bucket filter/new/view modals, and the full task-engine UI (`pages/task/`: `NewTaskWizard`+`SchemaEditor` goal→draft→create flow, `TaskDetail` composition root, `PipelineBoard`, `EntityDrawer`, `ReviewFeed`, `ThreadsPanel`); the Browser↔API fetch+SSE realtime layer (§1.6, §2.1, §2.11, §3.4).

| client/src/state/TasksProvider.tsx | `TasksProvider`/`useTasksStore()` (**new**, Phase 2B). Mounted once by `AppShell`, nested inside `InboxProvider` — the analogous shared task store for the life of the authed session (same rationale as `InboxProvider`: survive route navigation between the HUD and `/tasks/:taskId` instead of remounting/re-subscribing SSE per route). State: `tasks: Task[]` (list, no `state_schema`) + `byId` (memoized), `details: Record<id, TaskDetail>` (per-task cache, populated lazily by `loadDetail`), `backfill: Record<id, {scanned, matched, done}>` (live progress, from `task_backfill_progress`). `refresh()` → `GET /api/tasks`. `loadDetail(id)` → `GET /api/tasks/{id}`; on a 404 (detected via an `Error.message.includes('404')` substring match — `getJSON`'s only structured error shape is `{kind:'unauthorized'}` for 401s, there's no structured 404 shape) evicts the id from `details` and triggers a list `refresh()`, returning `null` rather than rethrowing (the DELETE case); any other failure logs + rethrows. `createTask`/`patchTask`/`deleteTask` wrap their `lib/api.ts` calls, updating `details`/`tasks` optimistically off the response then always calling `refresh()`. `taskFromDetail(detail)` strips `state_schema` back off a `TaskDetail` to produce the list-shaped `Task` — so one `GET .../{id}` response satisfies both the detail cache AND keeps the `tasks` list item current, no second fetch. | client/src/lib/sse.ts | EventSource singleton. Exports `subscribeSse(handler)→unsub`, types `PreviewExample`/`SseDataEvent`/`SseConnEvent`/`SseEvent`. One `_es=EventSource('/api/sse',{withCredentials:true})`; `_handlers:Set`; closes when the last handler unsubscribes. ## Decision points & gotchas
- **Pagination + auto-extend** (useInbox): `useEffect` fires `requestExtend` when `page >= pageCount-1`, not while a filter is active (filter shrinks pageCount artificially), not when `more===false` or `extendInFlight`. `lastExtendAtLength` ref blocks re-firing at the same `idLayer.length` so a 0-result extend can't tight-loop; re-arms when new ids grow the length. Extend cursor = smallest `recent_message.internal_date` across `idLayer`.
- **Extend watchdog**: `setTimeout(90_000)`; if `extend_complete` never arrives, force-reset `extendInFlight` + clear no-progress guard (no auto-retry — surfaces server bugs). Cleared on event arrival or POST failure.
- **LWW gate** (`applyThreadUpdates`): accept incoming thread only if `recent_message.internal_date >= lastInternalDate[id]` (default 0). `fetchAndReplace` **resets** (not merges) `lastInternalDate` so a thread aged out of the latest-200 window can't keep a stale gate value.
- **Snapshot/replay ordering** (useInboxSse): on `_open`, buffer `threads_updated` until `snapshot()` resolves, then drain in order — prevents an early SSE event being overwritten by a later snapshot. `cancelled` flag guards against re-`_open` racing a prior lifecycle. `_error` clears `ready` so subsequent events buffer again.
- **resync vs snapshot**: `snapshot` toggles `loading` (flashes "loading…"); `resync` (ReloadButton, watchdogs, page-clamp recovery) does NOT — keeps the list on screen. `resync` collapses `idLayer` to latest 200, so a page-clamp `useEffect` snaps `page` back to `pageCount` to avoid a blank extended-history page.
- **Optimistic vs server-confirmed**: nothing optimistic — `useBuckets` mutations refetch (`GET /api/buckets`) before UI updates; inbox rows always come from server via fetch/batch.
- **Draft preview dual-resolution** (NewBucketModal): SSE fast path + 5s poll fallback (poll starts at 5s to let SSE win the ~40s scoring window). `appliedRef` per `draft_id` makes apply idempotent regardless of arrival order; `gone`(404 TTL-expired) just stops — user re-runs "find examples"/"more examples" (each re-issues `postBucketDraftPreview` with `exclude_thread_ids=seenIds`).
- **One shared `useBuckets`/`useInbox` instance for the whole authed session, not per route** (**updated** — supersedes the old per-route-instance gotcha): `InboxProvider` (mounted once by `AppShell`) is the only place `useBuckets`/`useInbox`/`useInboxSse` are called; `HudPage` and `InboxPage` both consume the same instances via `useInboxStore()`. Creating/renaming/deleting a bucket on `/inbox` now live-updates `HudPage`'s bucket cards immediately (no more "refresh only on next mount"), and navigating between `/` and `/inbox` no longer remounts-and-refetches the inbox snapshot — `inbox.loading` only flashes true on the first-ever mount per session. Within `InboxPage`'s subtree, `NewBucketModal`/`ViewBucketsModal` still take callbacks from the store's `buckets` object rather than calling `useBuckets()` themselves — a second instance anywhere would refresh nobody's rendered list, leaving toolbar/filter stale.
- **StaticFiles / dev-proxy split**: prod — Vite builds into `server/app/static/` (`VITE_OUT_DIR` override), served by FastAPI `StaticFiles` + SPA catch-all (same origin, no proxy). Dev — Vite dev server `:5173` proxies `/auth` and `/api` to `http://localhost:8000` (`vite.config.ts`); `same-origin`/`withCredentials` cookies work because the proxy keeps one origin.
- **401 handling**: `getJSON`/`getThreadsBatch` throw `{kind:'unauthorized'}`; `useAuth.refresh` maps any error to `anon`. Other hooks (`useInbox`, `useBuckets`) do not specially route 401 — they log/swallow.
- **Archived-thread eviction, not a flag** (useInbox, **new**): `applyThreadUpdates` splits LWW-accepted threads into archived vs live; archived ids are deleted outright from `displayLayer`/`idLayer`/`lastInternalDate` — there's no rendered "archived" state, the row just disappears on the next SSE-driven batch fetch. This mirrors the server's reconciling archive/un-archive (see INBOX_SYNC_INDEX) but the client only ever sees the one-way "thread left the inbox" transition; an un-archive re-adds the thread as an ordinary live merge (it's just `is_archived:false` on the next accepted update).
- **Search race guard, not cancellation** (`useInboxSearch`, shared by `HudPage`/`InboxPage`): the 300ms debounce timer is cleared per-keystroke, but an in-flight `searchInbox` fetch is never aborted — `searchSeq` just discards its result if a newer query (or a clear) has since superseded it (`seq !== searchSeq.current`). Clearing the box bumps `searchSeq` too, so a slow stale response can't repopulate results after the user backs out.
- **Search bypasses `useInbox` entirely (both routes)**: `results` is a flat one-shot array from `GET /api/search`, not merged into `idLayer`/`displayLayer`. While search is active, the shared `useInbox` instance's pagination/auto-extend/bucket filter keep running in the background but are invisible until the query is cleared and `<InboxList>` switches back to `inbox.pageThreads` (`InboxPage`) or the store-derived `recent`/bucket-count views (`HudPage`). **Updated**: `HudPage` now does have a live `useInbox` instance (the shared one, via `useInboxStore()`) but still reads it only for `idLayer`/`displayLayer`, never `page`/pagination — its search simply swaps `<InboxList>`'s source the same way it always did.
- **The SSE-pinning gotcha is gone structurally, not by convention** (**rewritten** — supersedes the old "AppShell pins the SSE singleton open" entry): `lib/sse.ts`'s `EventSource` is reference-counted by handler count — `_es` closes when the last `subscribeSse` handler unsubscribes and reopens fresh on the next subscribe. Previously, `AppShell` had to hold a dedicated no-op handler because the two routes' own subscriptions (`HudPage`'s direct `subscribeSse` call, `InboxPage`'s `useInboxSse`) each mounted/unmounted with their route, so a cross-route navigation could transiently drop the handler count to zero between the outgoing route's cleanup and the incoming route's mount — closing and reopening the connection (plus an `active_users` deregister/re-register blip) on every navigation. Now `InboxProvider`'s single `useInboxSse` subscription lives at the `AppShell` layer for the entire authed session — it mounts once and only unmounts on sign-out (`AppShell` unmount → `InboxProvider` unmount → last unsub → close). Neither route mounts or unmounts its own SSE handler anymore, so the subscriber count never transiently hits zero during `/`↔`/inbox` navigation: the close/reopen churn risk is eliminated by construction, not papered over by a pin.
- **SSE error-streak → 401-only session recheck** (**new**, `AppShell` + `useAuth.recheckSession`): an expired session cookie makes every SSE reconnect fail with a fresh 401 forever — nothing about the stream alone distinguishes "the connection is down" from "the session is dead." `AppShell` counts consecutive `_error` events (reset by any `_open`); at 3 or more in a row, throttled to once per 30s, it calls `recheckSession()` — a `useAuth` method distinct from mount-time `refresh()` but sharing its `fetchMe()` helper. `recheckSession` flips auth state to `'anon'` **only** when the failure is a definitive 401 (`{kind:'unauthorized'}`, `lib/api.ts`'s shape for a real 401); any other failure (network outage, 5xx, timeout) leaves state untouched and the SSE backoff keeps retrying — a backend outage must not force-logout a session that's still valid. This deliberately differs from `refresh()`'s mount-time catch-all (any failure → `anon`), which is correct there because a failed initial load has no session to preserve. A real 401 flips `Gate` to `anon` → `<Login>` replaces `<AppShell>`/`<InboxProvider>` in the same commit, unsubscribing the last SSE handler and closing the stream.
- **HudPage refresh cadence — status-only now** (**updated**): `HudPage` no longer fetches its own inbox snapshot or listens for `threads_updated`/`_open` — the shared store (`InboxProvider`'s `useInboxSse`→`useInbox`) keeps `idLayer`/`displayLayer` current independently of which route is mounted, so `HudPage`'s "recently processed" list and bucket counts just re-render off the store's state (no debounce timer needed). `HudPage`'s own effect only drives sync-status cosmetics: an initial `getSyncStatus()` fetch, a 30s `refreshStatus` poll, and a 5s `setInterval` (`forceTick`) that re-renders `agoLabel` between polls — two intervals, no unsub, no debounce timer.
- **A task `version` gap alone is NOT a sufficient refetch trigger — `pending_count` is an equally load-bearing second signal** (**new**, Phase 2B; confirmed against server behavior in TASKS_INDEX): `TasksProvider`'s `task_updated` handler only calls `loadDetail(task_id)` when `e.version > known.version OR e.pending_count !== known.summary.pending_reviews`. This OR is required, not defensive boilerplate — per TASKS_INDEX's gotchas, several real mutations publish `task_updated` **without** bumping `version` at all: `reject_event`, `attach_thread`/`detach_thread`'s criteria-only `append_example` write (Phase 2B Task 2), and detach's new auto-reject-pending-events step (Phase 2B Task 1) all move `pending_count` (reject/auto-reject) or mutate the task in a way invisible to `version` (criteria) while leaving `version` untouched. A client that checked `version` alone would silently miss all of these — the review-queue badge (`task.summary.pending_reviews`, rendered on both the HUD's task cards and implicitly via `TaskDetail`'s refetch-cascade effect) would go stale exactly on a reject.
- **The learning loop is entirely server-side and invisible to the client** (**new**, Phase 2B Task 2): `Task`/`TaskDetail`'s TypeScript shape (`lib/api.ts`) has no `criteria` field at all — `append_example`'s tagged `<positive>`/`<nearmiss>` blocks (and the FIFO cap at `EXAMPLE_CAP=30`) live purely in a Postgres column the client never fetches. The "teach the task" checkbox (`ThreadsPanel`'s `AttachedRow`/`SearchResultRow`, default checked, wired to `attachThread`/`detachThread`'s `addExample` param) is the only client-visible trace of this feature — there is no UI anywhere that shows the accumulated example text, its current count, or when the FIFO cap starts evicting old corrections.
- **`EntityDrawer`'s per-entity history is bounded by `TaskDetail`'s events fetch, not by entity** (**new**, Phase 2B — verified gotcha): `TaskDetail.refetchEvents` calls `getTaskEvents(taskId)` with no `limit`/`page` args, so the server default (`GET /api/tasks/{id}/events` → `limit=50`, page 1, newest-first across the **whole task**) is what both `ReviewFeed` and `EntityDrawer` filter client-side. `EntityDrawer`'s `entityEvents` (`events.filter(e => e.entity_id === entity.id)`) can therefore under-represent a busy entity's true history on a task with many entities/events: once the task has accumulated more than 50 events total, an entity whose own events have aged out of that newest-50 window shows an incomplete (or empty) "history" section with no indication anything was truncated — there's no "load more"/pagination affordance in the drawer, even though `getTaskEvents` itself supports `page`/`limit`. `ReviewFeed`'s "Recent activity" list further slices this same already-capped 50 down to 30 non-pending entries.
