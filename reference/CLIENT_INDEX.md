> **Phase 4 additions** (bucket unification — buckets are now `tasks(kind='bucket')` server-side, see TASKS_INDEX): `useBuckets()` (`pages/buckets/useBuckets.tsx`) is now **task-backed** — its `refresh` calls `getTasks({kind:'bucket'})` (`GET /api/tasks?kind=bucket`) instead of a dedicated bucket-list fetch, and maps each `Task` row into the same `Bucket` shape (`id`/`name`/`criteria`/`is_default`) every consumer already expected; `rename`/`softDelete` now call `patchTask`/`deleteTask` instead of bucket-specific fetchers, and its own `create` is **deleted** (bucket creation moved to the task wizard — see next). `lib/api.ts`'s dedicated bucket fetchers (`createBucket`, `patchBucket`, `deleteBucket`, `postBucketDraftPreview`, `getBucketDraftPreview`) and `lib/sse.ts`'s `bucket_draft_preview` `SseDataEvent` member are all **deleted** — `Bucket`/`BucketExampleIn` survive as pure client-side types (the mapped shape of a `GET /api/tasks?kind=bucket` list item; `BucketExampleIn` reused as `createTask`'s example element). `pages/buckets/NewBucketModal.tsx` is **deleted outright** — bucket creation now goes through `pages/task/NewTaskWizard.tsx`, which gains a `kind?: 'tracker' | 'bucket'` prop (default `'tracker'`): bucket mode skips the `<SchemaEditor>` step entirely (`state_schema` always sent as `null`) and takes an `onCreated?: (taskId) => void` callback in place of tracker mode's post-backfill `navigate(/tasks/{id})` (there's no per-bucket detail page to navigate to). `InboxPage.tsx` renders `<NewTaskWizard kind="bucket" onCreated={() => buckets.refresh()} onClose={...}>` where `NewBucketModal` used to render; its own `createWithWatchdog` (the 60s+150s reclassify-completion watchdog) is **deleted** — bucket backfill's `threads_updated` publish is deterministic on completion (unlike the old fire-and-forget `reclassify_user_inbox` it replaced, see INBOX_SYNC_INDEX), so nothing needs to compensate for a missed SSE event anymore.

> **Phase 4.5 additions** (jobs surface, spec `specs/005_jobs_surface/design.md` — replaces every auto-opening, screen-blocking wizard wait with a user-initiated header chip + slide-over panel; "nothing ever pops up on its own again"): **new** `state/JobsProvider.tsx` (`JobsProvider`/`useJobsStore()`, exported `isJobTerminal(job)` helper — `dismissed_at !== null || stage === 'done' || stage === 'failed'`) — mounted once in `AppShell`, **inside** `InboxProvider`/`TasksProvider` (it calls `useInboxStore()` to reach `buckets.refresh`, so it must nest there, not above). Fetches `GET /api/jobs` (active-only) on mount and on every SSE `job_updated`/`_open`; a `POLL_INTERVAL_MS=15_000` `setInterval` runs ONLY while `jobs.some(j => !isJobTerminal(j))` is true, torn down and recreated as that condition flips (not just on mount/unmount) — the moment the last job finishes, no interval is left ticking. On each `refresh()`, diffs the new fetch against a `prevJobsRef` (a ref, not state — write-only) by id: a `kind==='creation' && task_kind==='bucket' && stage==='done'` job whose PREVIOUS stage wasn't `'done'` fires `buckets.refresh()` (via `useInboxStore()`) — a transition-diff, not a level-check, because `list_jobs` keeps a terminal job in the active list for 7 days and a level-check would re-refresh the bucket list every 15s poll for that whole window. Exposes `{jobs, refresh, startCreation(goal, taskKind), confirmJob(id, body), dismissJob(id)}`, each mutation awaiting its API call then `refresh()`ing. **New** `jobs/JobsChip.tsx` (`JobsChip({onReview})`, rendered in `AppShell`'s header) — renders nothing when `jobs.length === 0` (the store's fetch is already active-only, so zero really means nothing to show); a spinner (`anyRunning = jobs.some(j => !isJobTerminal(j))`) and a blue dot (`anyNeedsUser = jobs.some(j => j.needs_user)`, `#3b82f6`) inside a `Jobs {N}` button; owns its own `open` boolean and renders `<JobsPanel>` as a sibling when open, rather than lifting that state to `AppShell`. **New** `jobs/JobsPanel.tsx` (`JobsPanel({onClose, onReview})`) — a fixed right slide-over (`position:fixed`, transparent full-viewport backdrop that closes only the panel on click, `z-index` above content — overlay, never blocking); cards sorted newest-`created_at`-first (not `updated_at`, so a progress tick never reshuffles the list); per-card body/actions keyed on `(job.kind, job.stage)` — creation: `proposing`→"drafting proposal…"/no actions, `draft_ready`→"ready for review"/**[Review]** (calls `onReview(job)`, wired to `AppShell`'s `reviewJob` state), `backfilling`→progress bar (`scanned/total` + `matched`)/no actions, `done`→"tracker created"+`Open task` link (tracker, has `task_id`) or "bucket live"+dismiss (bucket, no per-bucket detail page), `failed`→error text+dismiss; delete_retriage: `running`→progress bar/no actions, `done`→"`{matched}` threads reclassified"+dismiss, `failed`→error+dismiss. Title is `job.payload?.proposal?.name ?? job.goal` for creation jobs, a fixed `"Reclassifying orphaned threads"` string for delete_retriage (which carries no name/goal of its own). `AppShell.tsx`: provider nesting is now `InboxProvider > TasksProvider > JobsProvider`, and the persistent header (previously a sibling above all three providers) **moved inside** this nesting so `JobsChip`, rendered in it, can call `useJobsStore()`; `AppShell` itself owns a new `reviewJob: Job | null` state (set by `JobsChip`'s `onReview` prop, which is `JobsPanel`'s own onReview forwarded up) and renders a review-mode `<NewTaskWizard reviewJob={reviewJob} onClose={() => setReviewJob(null)}>` as a sibling of `<Outlet/>`, inside `JobsProvider`. `pages/task/NewTaskWizard.tsx` is **restructured to exactly two user-initiated moments**, never anything that opens or persists on its own: **start** (`kind` prop, no `reviewJob`) — a goal-only form step; submitting calls `startCreation(goal, kind)` then `onClose()` **immediately** — no in-modal wait, no draft SSE/poll machinery at all (the old `draftId`/`appliedRef`/`pending` step and its own `subscribeSse` subscription for `task_draft_ready` are gone entirely — `NewTaskWizard` no longer subscribes to SSE or imports `useEffect`). **Review** (`reviewJob: Job` prop set) — opens DIRECTLY at the review step, seeded from `reviewJob.payload.proposal`/`.positives`/`.near_misses` (`effectiveKind = reviewJob.task_kind ?? 'tracker'`, trusting the job's own kind over any caller-supplied `kind` prop); submitting calls `confirmJob(reviewJob.id, body)` then `onClose()` **immediately** — no `creating` step, no `navigatedRef` backfill-watch effect, no `useNavigate` call anywhere in the file. Both moments' failures (`startError`/`createError`) render inline via a shared `errorBannerStyle` and leave the modal open, editable. The post-create "open task"/"bucket live" affordance now lives entirely in `JobsPanel`'s own done-card (see above) — the wizard itself never watches backfill progress or navigates. `InboxPage.tsx`'s bucket-creation wizard mount drops its `onCreated` prop entirely (start-mode only; `JobsProvider`'s own bucket-refresh diff, above, replaces it). `pages/task/SchemaEditor.tsx` gains a **`PipelineChipList`** component (replacing `ChipList` for `pipeline.stages` only — `pipeline.terminal` is unchanged, still the plain `ChipList`, since terminal stages carry no meaningful order): renders `→` arrow separators between chips (a styled `<span>`, not part of either chip's own box) and supports reordering two ways — HTML5 drag-and-drop (`draggable`/`onDragStart`/`onDragOver`/`onDrop`, tracking `dragIndex`/`overIndex` local state) for mouse users, and a keyboard-accessible ◀/▶ button pair per chip (disabled at the array's ends) as the required a11y fallback (drag-and-drop alone isn't operable without a pointer) — both paths funnel through one shared `move(from, to)` helper. A `pendingFocusRef`+`moveBtnRefs` pair makes focus follow the moved chip after a ◀/▶ click (a `useEffect` keyed on `items` re-focuses the target button post-reorder, or the opposite button if the target is now disabled at an array end, so focus never drops to `<body>`). `pages/buckets/ViewBucketsModal.tsx`: the delete-confirmation block moves from rendering inline inside whichever bucket's own `<li>` (below its criteria block, which could push the confirm/cancel buttons below the fold for a bucket further down the list) to rendering **once**, directly under the modal's `<h3>buckets</h3>` header, driven by a derived `confirmingBucket = buckets.find(b => b.id === confirming)` lookup — always visible the instant "delete" is clicked, regardless of list position. `lib/api.ts`: **deleted** `TaskDraftProposal`/`TaskDraftPoll` types and `postTaskDraft`/`getTaskDraft` fetchers entirely; **new** `Job`/`JobPayload` types (mirroring `api/jobs.py`'s `_serialize_job` field-for-field — `payload` is the server's **nested** `{proposal: {name, description, state_schema, keyword_probes}, positives, near_misses}` shape, deliberately NOT a flat merge of the old `TaskDraftProposal` + examples, and carries no `criteria` key) and fetchers `createJob(goal, taskKind)` (202, `POST /api/jobs`), `getJobs(active=true)` (`GET /api/jobs` or `?active=0`), `getJob(id)`, `confirmJob(id, body: ConfirmJobBody)` (`POST /api/jobs/{id}/confirm`, `ConfirmJobBody` = every `_CreateTaskBody` field except `kind`/`goal`), `dismissJob(id)` (204, idempotent). `lib/sse.ts`: `SseDataEvent`'s `task_draft_ready` member is **deleted**, replaced by `job_updated {job_id}` — a pure nudge (never a row), published after every job-row commit, telling `JobsProvider` to refetch `GET /api/jobs`.

## Routes / Tasks / Entrypoints
- **Routing** (**new**, react-router-dom ^7) — `App.tsx`'s `Gate` renders `<BrowserRouter><Routes>` once authed (`loading`/`anon` still render `<Splash>`/`<Login>` outside the router entirely). One layout route, `<AppShell>`, wraps both leaf routes via `<Outlet>`:

  | Path | Element | Notes |
  |------|---------|-------|
  | `/` | `HudPage` | default landing — snapshot + sync-freshness strip + bucket cards + (**new**, Phase 2B) task cards |
  | `/inbox` | `InboxPage` | full inbox list/search/buckets (ex-`Home`) |

- **AppShell** — the layout route: persistent header (title, `NavLink`s for `/` and `/inbox`, user email, sign-out) wrapping `<Outlet>` in `<InboxProvider>` (the old no-op SSE pin is gone, superseded by the provider's own permanent subscription, see gotchas). `AppShell` also runs its own `subscribeSse` purely to count `_error`/`_open` streaks and trigger `useAuth().recheckSession()` after 3+ consecutive errors (throttled 30s) — see gotchas for the 401-only semantics. **Updated** (Phase 2B): `<InboxProvider>` now also wraps a nested `<TasksProvider>` around the same `<Outlet>`, so `/tasks/:taskId` (and every other route) can call `useTasksStore()`. **Updated** (Phase 4.5): nesting gains a third, innermost `<JobsProvider>` (`<InboxProvider><TasksProvider><JobsProvider>`), and the header itself **moves inside** all three so the `<JobsChip>` it renders can call `useJobsStore()`. `AppShell` also now owns a `reviewJob: Job | null` state feeding a conditionally-mounted review-mode `<NewTaskWizard>` — see that file's row and JobsProvider/JobsChip/JobsPanel below.

- **InboxProvider / useInboxStore** (`state/InboxProvider.tsx`) — the shared inbox store, mounted once by `AppShell` for the life of the authed session. Composes `useBuckets`+`useInbox`+`useInboxSse` (all three unmodified) and exposes `{buckets, inbox, filterSelection, setFilterSelection}` via context; `HudPage`, `InboxPage`, and `TaskDetail` (for `buckets` only, to label attached threads' bucket pills) all call `useInboxStore()` instead of instantiating their own hooks. - **TasksProvider / useTasksStore** (**new**, Phase 2B, `state/TasksProvider.tsx`) — the analogous shared task store, mounted once by `AppShell` (nested inside `InboxProvider`) for the life of the authed session. Owns the task list (`tasks`/`byId`), a lazily-populated per-task detail cache (`details`, read via `getDetail`/`loadDetail`), and live backfill progress (`backfill`). ## Decision points & gotchas
- **Pagination + auto-extend** (useInbox): `useEffect` fires `requestExtend` when `page >= pageCount-1`, not while a filter is active (filter shrinks pageCount artificially), not when `more===false` or `extendInFlight`. `lastExtendAtLength` ref blocks re-firing at the same `idLayer.length` so a 0-result extend can't tight-loop; re-arms when new ids grow the length. Extend cursor = smallest `recent_message.internal_date` across `idLayer`.
- **Extend watchdog**: `setTimeout(90_000)`; if `extend_complete` never arrives, force-reset `extendInFlight` + clear no-progress guard (no auto-retry — surfaces server bugs). Cleared on event arrival or POST failure.
- **LWW gate** (`applyThreadUpdates`): accept incoming thread only if `recent_message.internal_date >= lastInternalDate[id]` (default 0). `fetchAndReplace` **resets** (not merges) `lastInternalDate` so a thread aged out of the latest-200 window can't keep a stale gate value.
- **Snapshot/replay ordering** (useInboxSse): on `_open`, buffer `threads_updated` until `snapshot()` resolves, then drain in order — prevents an early SSE event being overwritten by a later snapshot. `cancelled` flag guards against re-`_open` racing a prior lifecycle. `_error` clears `ready` so subsequent events buffer again.
- **resync vs snapshot**: `snapshot` toggles `loading` (flashes "loading…"); `resync` (ReloadButton, watchdogs, page-clamp recovery) does NOT — keeps the list on screen. `resync` collapses `idLayer` to latest 200, so a page-clamp `useEffect` snaps `page` back to `pageCount` to avoid a blank extended-history page.
- **Optimistic vs server-confirmed**: nothing optimistic — `useBuckets` mutations refetch (`GET /api/buckets`) before UI updates; inbox rows always come from server via fetch/batch.
- **The old per-wizard SSE-fast-path + poll-fallback dual-resolution dance is gone entirely (Phase 4.5) — there is no goal→draft dance left in `NewTaskWizard` at all.** Prior phases' entry here described `NewTaskWizard`'s `appliedRef`-per-`draft_id` idempotency guard racing an SSE `task_draft_ready` event against a 5s poll fallback, with a `gone` (redis-TTL-expired) state surfaced via `draftError`. None of that exists anymore: the wizard's start moment fires `startCreation` and closes immediately, with no result to wait for in the same mount; its review moment opens ALREADY seeded from a `draft_ready` job's `payload` (passed in as the `reviewJob` prop by whoever clicked `JobsPanel`'s `[Review]`), so there's nothing to resolve at wizard-open time either. The equivalent "did this happen yet" polling now lives one level up, in `JobsProvider`'s list-level `GET /api/jobs` fetch (SSE-nudged + 15s-polled) — see that file's own row for the mechanism that replaced this one.
- **One shared `useBuckets`/`useInbox` instance for the whole authed session, not per route**: `InboxProvider` (mounted once by `AppShell`) is the only place `useBuckets`/`useInbox`/`useInboxSse` are called; `HudPage` and `InboxPage` both consume the same instances via `useInboxStore()`. Renaming/deleting a bucket on `/inbox` live-updates `HudPage`'s bucket cards immediately (no "refresh only on next mount"), and navigating between `/` and `/inbox` no longer remounts-and-refetches the inbox snapshot — `inbox.loading` only flashes true on the first-ever mount per session. Within `InboxPage`'s subtree, `ViewBucketsModal` still takes callbacks from the store's `buckets` object rather than calling `useBuckets()` itself — a second instance anywhere would refresh nobody's rendered list, leaving toolbar/filter stale. **Phase 4**: bucket *creation* no longer goes through this store at all. **Phase 4.5**: nor does it go through `TasksProvider`/`createTask` anymore either — `NewTaskWizard kind="bucket"` now calls `useJobsStore().startCreation`/`confirmJob` (see `state/JobsProvider.tsx`'s own row), and `buckets.refresh()` fires from `JobsProvider`'s own bucket-creation-done transition diff, not any wizard callback prop.
- **StaticFiles / dev-proxy split**: prod — Vite builds into `server/app/static/` (`VITE_OUT_DIR` override), served by FastAPI `StaticFiles` + SPA catch-all (same origin, no proxy). Dev — Vite dev server `:5173` proxies `/auth` and `/api` to `http://localhost:8000` (`vite.config.ts`); `same-origin`/`withCredentials` cookies work because the proxy keeps one origin.
- **401 handling**: `getJSON`/`getThreadsBatch` throw `{kind:'unauthorized'}`; `useAuth.refresh` maps any error to `anon`. Other hooks (`useInbox`, `useBuckets`) do not specially route 401 — they log/swallow.
- **Archived-thread eviction, not a flag** (useInbox, **new**): `applyThreadUpdates` splits LWW-accepted threads into archived vs live; archived ids are deleted outright from `displayLayer`/`idLayer`/`lastInternalDate` — there's no rendered "archived" state, the row just disappears on the next SSE-driven batch fetch. This mirrors the server's reconciling archive/un-archive (see INBOX_SYNC_INDEX) but the client only ever sees the one-way "thread left the inbox" transition; an un-archive re-adds the thread as an ordinary live merge (it's just `is_archived:false` on the next accepted update).
- **Search race guard, not cancellation** (`useInboxSearch`, shared by `HudPage`/`InboxPage`): the 300ms debounce timer is cleared per-keystroke, but an in-flight `searchInbox` fetch is never aborted — `searchSeq` just discards its result if a newer query (or a clear) has since superseded it (`seq !== searchSeq.current`). Clearing the box bumps `searchSeq` too, so a slow stale response can't repopulate results after the user backs out.
- **Search bypasses `useInbox` entirely (both routes)**: `results` is a flat one-shot array from `GET /api/search`, not merged into `idLayer`/`displayLayer`. While search is active, the shared `useInbox` instance's pagination/auto-extend/bucket filter keep running in the background but are invisible until the query is cleared and `<InboxList>` switches back to `inbox.pageThreads` (`InboxPage`) or the store-derived `recent`/bucket-count views (`HudPage`). **Updated**: `HudPage` now does have a live `useInbox` instance (the shared one, via `useInboxStore()`) but still reads it only for `idLayer`/`displayLayer`, never `page`/pagination — its search simply swaps `<InboxList>`'s source the same way it always did.
- **The SSE-pinning gotcha is gone structurally, not by convention** (**rewritten** — supersedes the old "AppShell pins the SSE singleton open" entry): `lib/sse.ts`'s `EventSource` is reference-counted by handler count — `_es` closes when the last `subscribeSse` handler unsubscribes and reopens fresh on the next subscribe. Previously, `AppShell` had to hold a dedicated no-op handler because the two routes' own subscriptions (`HudPage`'s direct `subscribeSse` call, `InboxPage`'s `useInboxSse`) each mounted/unmounted with their route, so a cross-route navigation could transiently drop the handler count to zero between the outgoing route's cleanup and the incoming route's mount — closing and reopening the connection (plus an `active_users` deregister/re-register blip) on every navigation. Now `InboxProvider`'s single `useInboxSse` subscription lives at the `AppShell` layer for the entire authed session — it mounts once and only unmounts on sign-out (`AppShell` unmount → `InboxProvider` unmount → last unsub → close). Neither route mounts or unmounts its own SSE handler anymore, so the subscriber count never transiently hits zero during `/`↔`/inbox` navigation: the close/reopen churn risk is eliminated by construction, not papered over by a pin.
- **SSE error-streak → 401-only session recheck** (**new**, `AppShell` + `useAuth.recheckSession`): an expired session cookie makes every SSE reconnect fail with a fresh 401 forever — nothing about the stream alone distinguishes "the connection is down" from "the session is dead." `AppShell` counts consecutive `_error` events (reset by any `_open`); at 3 or more in a row, throttled to once per 30s, it calls `recheckSession()` — a `useAuth` method distinct from mount-time `refresh()` but sharing its `fetchMe()` helper. `recheckSession` flips auth state to `'anon'` **only** when the failure is a definitive 401 (`{kind:'unauthorized'}`, `lib/api.ts`'s shape for a real 401); any other failure (network outage, 5xx, timeout) leaves state untouched and the SSE backoff keeps retrying — a backend outage must not force-logout a session that's still valid. This deliberately differs from `refresh()`'s mount-time catch-all (any failure → `anon`), which is correct there because a failed initial load has no session to preserve. A real 401 flips `Gate` to `anon` → `<Login>` replaces `<AppShell>`/`<InboxProvider>` in the same commit, unsubscribing the last SSE handler and closing the stream.
- **HudPage refresh cadence — status-only now** (**updated**): `HudPage` no longer fetches its own inbox snapshot or listens for `threads_updated`/`_open` — the shared store (`InboxProvider`'s `useInboxSse`→`useInbox`) keeps `idLayer`/`displayLayer` current independently of which route is mounted, so `HudPage`'s "recently processed" list and bucket counts just re-render off the store's state (no debounce timer needed). `HudPage`'s own effect only drives sync-status cosmetics: an initial `getSyncStatus()` fetch, a 30s `refreshStatus` poll, and a 5s `setInterval` (`forceTick`) that re-renders `agoLabel` between polls — two intervals, no unsub, no debounce timer.
- **A task `version` gap alone is NOT a sufficient refetch trigger — `pending_count` is an equally load-bearing second signal** (**new**, Phase 2B; confirmed against server behavior in TASKS_INDEX): `TasksProvider`'s `task_updated` handler only calls `loadDetail(task_id)` when `e.version > known.version OR e.pending_count !== known.summary.pending_reviews`. This OR is required, not defensive boilerplate — per TASKS_INDEX's gotchas, several real mutations publish `task_updated` **without** bumping `version` at all: `reject_event`, `attach_thread`/`detach_thread`'s criteria-only `append_example` write (Phase 2B Task 2), and detach's new auto-reject-pending-events step (Phase 2B Task 1) all move `pending_count` (reject/auto-reject) or mutate the task in a way invisible to `version` (criteria) while leaving `version` untouched. A client that checked `version` alone would silently miss all of these — the review-queue badge (`task.summary.pending_reviews`, rendered on both the HUD's task cards and implicitly via `TaskDetail`'s refetch-cascade effect) would go stale exactly on a reject.
- **The learning loop is entirely server-side and invisible to the client** (**new**, Phase 2B Task 2): `Task`/`TaskDetail`'s TypeScript shape (`lib/api.ts`) has no `criteria` field at all — `append_example`'s tagged `<positive>`/`<nearmiss>` blocks (and the FIFO cap at `EXAMPLE_CAP=30`) live purely in a Postgres column the client never fetches. The "teach the task" checkbox (`ThreadsPanel`'s `AttachedRow`/`SearchResultRow`, default checked, wired to `attachThread`/`detachThread`'s `addExample` param) is the only client-visible trace of this feature — there is no UI anywhere that shows the accumulated example text, its current count, or when the FIFO cap starts evicting old corrections.
- **`EntityDrawer`'s per-entity history is bounded by `TaskDetail`'s events fetch, not by entity** (**new**, Phase 2B — verified gotcha): `TaskDetail.refetchEvents` calls `getTaskEvents(taskId)` with no `limit`/`page` args, so the server default (`GET /api/tasks/{id}/events` → `limit=50`, page 1, newest-first across the **whole task**) is what both `ReviewFeed` and `EntityDrawer` filter client-side. `EntityDrawer`'s `entityEvents` (`events.filter(e => e.entity_id === entity.id)`) can therefore under-represent a busy entity's true history on a task with many entities/events: once the task has accumulated more than 50 events total, an entity whose own events have aged out of that newest-50 window shows an incomplete (or empty) "history" section with no indication anything was truncated — there's no "load more"/pagination affordance in the drawer, even though `getTaskEvents` itself supports `page`/`limit`. `ReviewFeed`'s "Recent activity" list further slices this same already-capped 50 down to 30 non-pending entries.
- **`ReviewTray`/`ActivityTicker` are self-owned-data components, deliberately outside `TasksProvider`'s cache** (**new**, Phase 3): both fetch `getReviews()`/`getActivity()` directly on mount rather than being fed by `useTasksStore()` — the cross-task aggregate feed has exactly one consumer each, so there's no cache to share. They still READ `useTasksStore()`'s `tasks` list, but only to derive a summed scalar (`totalPending` for `ReviewTray`; `totalVersion` AND `totalPending` for `ActivityTicker`) that tells them "something changed somewhere, go refetch" — they never read `tasks[].summary`/`.version` for display, and approving/rejecting from inside either component never touches `TasksProvider`'s state directly (that store still only updates via its own SSE `task_updated` handler). A change made inside a task's own `ReviewFeed`/`EntityDrawer` is what surfaces here indirectly, through the same SSE path.
- **Why `ActivityTicker` sums TWO signals (`totalVersion` and `totalPending`) but `ReviewTray` sums only one (`totalPending`)** (**new**, Phase 3): `ReviewTray`'s feed is exactly the pending-review set, and the one thing that ever moves an event into or out of it (a new LLM proposal, or an approve/reject resolving one) always moves `pending_reviews` — one signal suffices. `ActivityTicker`'s feed is the OTHER set (applied/rejected/reverted) — a `reject` produces a brand-new item in THIS feed (`status="rejected"`), but per TASKS_INDEX's `bump_version` contract, `reject_event` is the one mutation that does **not** bump `task.version` ("server publishes the pending_count change only by design," per the component's own comment) — so `totalVersion` alone would never notice a reject happened and the ticker would silently miss showing it. `totalPending` (the same signal `ReviewTray` already uses) catches rejects; `totalVersion` catches everything else (applies/edits/approves). Both are kept as separate `useEffect` dependencies rather than combined into one arithmetic sum, because an approve's `version +1` and `pending_reviews -1` would otherwise net to zero and silently skip a needed refetch.
- **`JobsProvider` must mount inside `InboxProvider` — it is not an independent shell-level store like `TasksProvider`** (**new**, Phase 4.5): unlike `TasksProvider` (which reads nothing from `InboxProvider`), `JobsProvider` calls `useInboxStore()` directly to reach `buckets.refresh` for its bucket-creation-done hand-off. `AppShell`'s nesting (`InboxProvider > TasksProvider > JobsProvider`) is load-bearing for this reason, not just a style convention — moving `JobsProvider` outside `InboxProvider` would crash on `useInboxStore()`'s own "must be used within a provider" throw.
- **`JobsProvider`'s 15s poll start/stops with the jobs list itself, not with mount/unmount — the interval-leak guard is the effect's own cleanup running on every `hasActive` flip** (**new**, Phase 4.5): the gating condition (`hasActive = jobs.some(j => !isJobTerminal(j))`) is a `useEffect` dependency, so React tears down the previous interval and (if still applicable) starts a fresh one on every render where `hasActive` changed — the load-bearing property is that this teardown fires the MOMENT the last non-terminal job resolves (dismissed or reaches `done`/`failed`), not just on the provider's own unmount (which never happens for the life of the authed session). A naive `useEffect(() => { setInterval(...) }, [])` (mount-once) would instead poll forever regardless of job state.
- **Bucket-creation completion is detected by a transition diff (`prevJobsRef`), not a level check on the jobs list** (**new**, Phase 4.5): `list_jobs`'s server-side active window keeps a `done`/`failed` job visible for 7 days (TASKS_INDEX), so a job that reached `done` stays in `JobsProvider`'s `jobs` array for a week. If `refresh()` fired `buckets.refresh()` every time it saw ANY `kind='creation' && task_kind='bucket' && stage='done'` job in the list, every 15s poll tick (or every `job_updated` nudge from an unrelated job) would re-refresh the bucket list for that entire window. Instead, `prevJobsRef` (a ref, deliberately not state) holds the PREVIOUS fetch, and `refresh()` only fires `buckets.refresh()` when a matching job's stage differs from what it was in that previous snapshot (`!prev || prev.stage !== 'done'`) — a one-time edge trigger, not a recurring level check.
- **The wizard's two moments never overlap within one mount, and neither one waits on anything server-side anymore** (**new**, Phase 4.5 — the central behavioral change of the restructure): `NewTaskWizard`'s `step` is decided ONCE at mount (`reviewJob` present → `'review'`, absent → `'form'`) and has no setter — there is no code path where a single mounted instance transitions between the two steps, unlike the old `form→pending→review→creating` machine. Both `startCreation` (start moment) and `confirmJob` (review moment) call `onClose()` in the same handler that awaits them, synchronously with the API call's resolution — the wizard is never left open waiting on a Celery task, an SSE event, or a poll. All of that (draft-ready detection, backfill progress, the eventual "open task"/"bucket live" affordance) is `JobsPanel`'s job now, entirely decoupled from whichever wizard instance (if any) is currently mounted — closing the wizard, navigating away, or even a full page reload during the wait loses nothing, since none of that state ever lived in the wizard to begin with.
- **`GET /api/tasks/{id}/draft` and `POST /api/tasks/draft` have zero client call sites left; `POST /api/tasks` has exactly zero client call sites left (Phase 4.5)** — both are still live server routes (`POST /api/tasks` explicitly documented server-side as "remains unchanged for API/test use," per TASKS_INDEX), but nothing under `client/src` calls either directly anymore. Any future client feature that needs to create a task/bucket without going through the jobs surface (e.g. a hypothetical bulk-import tool) would need to add a new `lib/api.ts` fetcher for `POST /api/tasks` from scratch — it isn't lying around unused, it's fully absent from this file.
- **Actions/rules tracker-only gating lives at the call site, not inside the components (Phase 5)**: neither `RulesSection` nor `TaskActionsPanel` reads `task.kind` itself — `TaskDetail.tsx` repeats `{detail.kind === 'tracker' && <Component .../>}` at two separate render points. A hypothetical third mount point for either component (there is none today) would need to add its own identical guard; nothing enforces this centrally.
- **`deleteRule` is the one Phase 5 fetcher not using `throwWithDetail`**: every sibling (`createRule`, `patchRule`, `approveAction`, `rejectAction`, `undoAction`) surfaces a server 422/409 `detail` string verbatim; `deleteRule` throws a bare `Error` on a non-204 response instead. Harmless in practice (the route is an idempotent soft-delete that only 404s on ownership, never 422s), but inconsistent with its siblings.
- **`action_updated` is handled filtered vs. unfiltered depending on scope (Phase 5)**: the HUD's `ReviewTray`/`ActivityTicker` refetch on ANY `action_updated` event with no `task_id` check (correct — they aggregate every task's actions); `TaskActionsPanel` filters to `e.task_id === taskId` (correct — a single task's panel refetching on an unrelated task's action would be pointless work, though harmless if it didn't).
