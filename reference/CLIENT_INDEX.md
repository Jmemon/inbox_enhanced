# Client Index

> Scope: React 19 + Vite browser SPA (client/src) — react-router-dom routing (`Gate`→`BrowserRouter`, `AppShell` layout route, `/` HudPage, `/inbox` InboxPage), App/AuthProvider, lib/api fetch wrappers (+ `searchInbox`, `getSyncStatus`) + lib/sse EventSource singleton, `state/InboxProvider.tsx` — a shell-level context (`useInboxStore()`) composing `useBuckets`/`useInbox`/`useInboxSse` **once** for the life of the authed session so `/` and `/inbox` read and write one live store instead of each route remounting its own copy (its permanent `useInboxSse` subscription also keeps the SSE singleton pinned open, superseding the old `AppShell` no-op pin), shared debounced inbox search (`pages/search/`) used by both routes, HudPage's store-derived bucket counts + sync-freshness strip, inbox list + pagination + reload, bucket filter/new/view modals; the Browser↔API fetch+SSE realtime layer (§1.6, §2.1, §2.11, §3.4).

## Routes / Tasks / Entrypoints
- **Routing** (**new**, react-router-dom ^7) — `App.tsx`'s `Gate` renders `<BrowserRouter><Routes>` once authed (`loading`/`anon` still render `<Splash>`/`<Login>` outside the router entirely). One layout route, `<AppShell>`, wraps both leaf routes via `<Outlet>`:

  | Path | Element | Notes |
  |------|---------|-------|
  | `/` | `HudPage` | default landing — snapshot + sync-freshness strip + bucket cards |
  | `/inbox` | `InboxPage` | full inbox list/search/buckets (ex-`Home`) |
  | `*` | `<Navigate to="/" replace>` | catch-all redirect |

- **AppShell** — the layout route: persistent header (title, `NavLink`s for `/` and `/inbox`, user email, sign-out) wrapping `<Outlet>` in `<InboxProvider>` (**updated** — the SSE pin used to be a dedicated no-op `useEffect` here; that's gone, superseded by the provider's own permanent subscription, see gotchas).
- **useAuth** — owns auth context (`loading|authed|anon`). Bootstrap `useEffect`→`/auth/me`. `signOut`→`/auth/logout`.
- **InboxProvider / useInboxStore** (**new**, `state/InboxProvider.tsx`) — the shared inbox store, mounted once by `AppShell` for the life of the authed session. Composes `useBuckets`+`useInbox`+`useInboxSse` (all three unmodified) and exposes `{buckets, inbox, filterSelection, setFilterSelection}` via context; `HudPage` and `InboxPage` both call `useInboxStore()` instead of instantiating their own hooks. Sign-out still closes the SSE stream: `AppShell` unmounts → `InboxProvider` unmounts → `useInboxSse`'s last unsub fires → `lib/sse.ts` closes `_es`.
- **useBuckets** — **updated**: single shared instance owned by `InboxProvider`, not one per route. `HudPage` reads `buckets.buckets`/`buckets.byId` (bucket-count cards); `InboxPage` additionally uses `buckets.create`/`rename`/`softDelete` (each mutates then `refresh()` via `GET /api/buckets`). Because it's one instance, a bucket created/renamed/deleted on `/inbox` is immediately reflected in `HudPage`'s cards without navigating back to remount it.
- **useInbox** — **updated**: single shared instance owned by `InboxProvider` (previously mounted only by `InboxPage`): `idLayer:string[]` (ordered ids) + `displayLayer:Record<id,InboxThread>` (rows), `page`, `more:boolean|null`, `extendInFlight`. Provides `snapshot` (loading toggle), `resync` (silent), `applyThreadUpdates(ids)` (LWW-gates, then partitions accepted threads into archived-evict vs live-merge), `requestExtend`, `hydrateCurrentPage`, derived `pageThreads`/`pageCount`/`filteredIdLayer`. `HudPage` reads `idLayer`/`displayLayer` directly (for its "recently processed" list and bucket counts) but never touches `page`/pagination/filter — those stay `InboxPage`-only concerns even though the underlying hook instance is shared.
- **useInboxSse** — **updated**: single permanent subscription owned by `InboxProvider` (previously mounted only by `InboxPage`, plus a separate `AppShell` no-op pin): `_open`→`snapshot()`, then drains buffered `threads_updated` via `onApply`; live `threads_updated` apply directly once ready. Because this subscription lives for the whole authed session, the inbox stays current even while the user sits on the HUD, not the inbox list.
- **Search** (`pages/search/`, shared location) — `useInboxSearch()`: debounced (300ms) search box used identically by both `HudPage` and `InboxPage`: `query`/`results`/`error` state + `searchSeq` ref (monotonic token, discards stale out-of-order responses) driving `searchInbox(q)`→`GET /api/search`. Non-null `results` replaces the mounted page's `<InboxList>` source; clearing the query (or an empty trimmed value) restores that page's normal view. Independent of `useInbox` — does not touch `idLayer`/`displayLayer`/pagination/filter. Unaffected by the `InboxProvider` refactor.
- **Modals** — `NewBucketModal` (form→pending→review draft flow), `ViewBucketsModal` (rename/soft-delete). Both shown conditionally from `InboxPage`.
- **Key components** — `AppShell` (persistent chrome + provider mount point), `InboxProvider`/`useInboxStore` (shared state), `HudPage` / `InboxPage` (route composition roots, now store consumers), `SecondaryHeader` (toolbar), `InboxList`, `Pagination`, `ReloadButton`, `FilterByBucketDropdown`.

  - `POST /api/threads/batch {thread_ids}` (`getThreadsBatch`) — SSE replay / hydrate / extend apply.
  - `GET /api/threads/{id}` (`getThread`) — exported, **no current call site**.
  - `POST /api/inbox/refresh` (`requestRefresh`, 202) — ReloadButton.
  - `POST /api/inbox/extend {before_internal_date}` (`postInboxExtend`, 202) — useInbox auto-extend.
  - `GET /api/buckets`, `POST /api/buckets`, `PATCH /api/buckets/{id}`, `DELETE /api/buckets/{id}` (204) — useBuckets, one shared instance owned by `InboxProvider`.
  - `POST /api/buckets/draft/preview {name,description,exclude_thread_ids}` (202→`{draft_id}`) + `GET /api/buckets/draft/preview/{draft_id}` (202 pending / 200 ready / 404 gone) — NewBucketModal.
- **SSE — lib/sse.ts** `EventSource('/api/sse')`. Data events consumed: `threads_updated{thread_ids}` (useInboxSse→useInbox.applyThreadUpdates), `extend_complete{thread_ids,more}` (useInbox), `bucket_draft_preview{draft_id,positives,near_misses}` (NewBucketModal). Synthetic conn events `_open`/`_error` broadcast to all handlers. Server `: keepalive` frames (5s) are non-JSON and ignored by `onmessage` try/catch.
- **Client-side state**: pagination cursor `page` + derived `pageCount` (PAGE_SIZE=50); `more` (history exhaustion); `extendInFlight`; refs `lastInternalDate` (LWW gate per id), `lastExtendAtLength` (no-progress guard), `extendWatchdog` (90s timer) — all owned by the one `InboxProvider`-instantiated `useInbox`, not `InboxPage`; NewBucketModal `appliedRef` (per-draft idempotency); `InboxPage` reclassify `setTimeout`s (60s+150s); shared `useInboxSearch` `query`/`results`/`error` state + `searchSeq` ref (monotonic token, guards stale async responses), used by both `HudPage` and `InboxPage`; `HudPage`'s own `status` state + 30s `refreshStatus` poll and cosmetic 5s ago-ticker `setInterval` (`forceTick`) — **updated**: the old `refreshTimer` (2s SSE-refresh debounce) and `snapshot` state are gone, since the inbox list is no longer `HudPage`'s own fetch.
- **fetch vs SSE**: snapshot/hydrate/mutations are fetch; new/changed threads, extend completion, and draft-preview results are pushed over SSE (with HTTP polling fallback only for draft preview).

## Data flows / cross-subsystem touchpoints
- `Browser ──[GET /auth/me, Cookie:session]──> API` — auth bootstrap.
- `Browser ──[GET /api/inbox?limit=200]──> API ──[JSON InboxPage]──> Browser` — snapshot/resync top-N view.
- `Browser ──[GET /api/sse, text/event-stream]──> API` — long-lived; frames `threads_updated`/`extend_complete`/`bucket_draft_preview` + `: keepalive`/5s.
- `Browser ──[POST /api/threads/batch {thread_ids:[…]}]──> API ──[JSON {threads}]──> Browser` — SSE replay/hydrate path (1 round trip for ≤200 ids).
- `Browser ──[POST /api/inbox/refresh]──> API` (202) — kicks a worker Gmail poll; results return later as SSE `threads_updated`. ReloadButton also resyncs locally.
- `Browser ──[POST /api/inbox/extend {before_internal_date}]──> API` (202) — worker extends Gmail history; completion arrives as SSE `extend_complete`.
- `Browser ──[POST /api/buckets/draft/preview]──> API ──[{draft_id}]──> Browser`; result via SSE `bucket_draft_preview` OR polled `GET /api/buckets/draft/preview/{draft_id}` (redis cache, 600s TTL).
- `Browser ──[POST /api/buckets]──> API` enqueues `reclassify_user_inbox`; completion *should* arrive as `threads_updated` but `InboxPage` schedules 60s+150s `resync()` watchdogs to compensate.
- `Browser ──[GET /api/search?q=]──> API ──[JSON InboxPage]──> Browser` — shared `useInboxSearch()` debounced (300ms) box (`HudPage` + `InboxPage`); plain synchronous request/response, no SSE involved; `searchSeq` discards a response if a newer query has since superseded it; results replace the mounted page's `<InboxList>` source until the query is cleared.
- `Browser ──[GET /api/sync/status]──> API ──[JSON {last_synced_at, has_cursor}]──> Browser` — `HudPage`'s own `refreshStatus()` fetches this (mount + 30s poll) to drive the freshness strip; plain synchronous request/response; independent of the shared inbox store.
- **Intra-tab (§2.11)**: `InboxProvider`'s single `useInboxSse` and `NewBucketModal` subscribe the shared `lib/sse.ts` `EventSource` via the `_handlers` Set — one socket fans out to the few React subscribers that remain (**updated** — previously each of `useInboxSse`/`HudPage`/`AppShell`'s no-op held their own handle; now the inbox side is one shared subscription for the whole authed session).
- **SSE→state mutation**: `threads_updated`→`getThreadsBatch(ids)`→LWW-gate→merge `displayLayer` + re-sort `idLayer` desc by `internal_date`. `extend_complete`→`setMore`+clear watchdog+apply ids. `bucket_draft_preview`→push examples + flip modal to `review`.

## Decision points & gotchas
- **Pagination + auto-extend** (useInbox): `useEffect` fires `requestExtend` when `page >= pageCount-1`, not while a filter is active (filter shrinks pageCount artificially), not when `more===false` or `extendInFlight`. `lastExtendAtLength` ref blocks re-firing at the same `idLayer.length` so a 0-result extend can't tight-loop; re-arms when new ids grow the length. Extend cursor = smallest `recent_message.internal_date` across `idLayer`.
- **Extend watchdog**: `setTimeout(90_000)`; if `extend_complete` never arrives, force-reset `extendInFlight` + clear no-progress guard (no auto-retry — surfaces server bugs). Cleared on event arrival or POST failure.
- **LWW gate** (`applyThreadUpdates`): accept incoming thread only if `recent_message.internal_date >= lastInternalDate[id]` (default 0). `fetchAndReplace` **resets** (not merges) `lastInternalDate` so a thread aged out of the latest-200 window can't keep a stale gate value.
- **Snapshot/replay ordering** (useInboxSse): on `_open`, buffer `threads_updated` until `snapshot()` resolves, then drain in order — prevents an early SSE event being overwritten by a later snapshot. `cancelled` flag guards against re-`_open` racing a prior lifecycle. `_error` clears `ready` so subsequent events buffer again.
- **resync vs snapshot**: `snapshot` toggles `loading` (flashes "loading…"); `resync` (ReloadButton, watchdogs, page-clamp recovery) does NOT — keeps the list on screen. `resync` collapses `idLayer` to latest 200, so a page-clamp `useEffect` snaps `page` back to `pageCount` to avoid a blank extended-history page.
- **Optimistic vs server-confirmed**: nothing optimistic — `useBuckets` mutations refetch (`GET /api/buckets`) before UI updates; inbox rows always come from server via fetch/batch.
- **Draft preview dual-resolution** (NewBucketModal): SSE fast path + 5s poll fallback (poll starts at 5s to let SSE win the ~40s scoring window). `appliedRef` per `draft_id` makes apply idempotent regardless of arrival order; `gone`(404 TTL-expired) just stops — user re-runs "find examples"/"more examples" (each re-issues `postBucketDraftPreview` with `exclude_thread_ids=seenIds`).
- **One shared `useBuckets`/`useInbox` instance for the whole authed session, not per route** (**updated** — supersedes the old per-route-instance gotcha): `InboxProvider` (mounted once by `AppShell`) is the only place `useBuckets`/`useInbox`/`useInboxSse` are called; `HudPage` and `InboxPage` both consume the same instances via `useInboxStore()`. Creating/renaming/deleting a bucket on `/inbox` now live-updates `HudPage`'s bucket cards immediately (no more "refresh only on next mount"), and navigating between `/` and `/inbox` no longer remounts-and-refetches the inbox snapshot — `inbox.loading` only flashes true on the first-ever mount per session. Within `InboxPage`'s subtree, `NewBucketModal`/`ViewBucketsModal` still take callbacks from the store's `buckets` object rather than calling `useBuckets()` themselves — a second instance anywhere would refresh nobody's rendered list, leaving toolbar/filter stale.
- **StaticFiles / dev-proxy split**: prod — Vite builds into `server/app/static/` (`VITE_OUT_DIR` override), served by FastAPI `StaticFiles` + SPA catch-all (same origin, no proxy). Dev — Vite dev server `:5173` proxies `/auth` and `/api` to `http://localhost:8000` (`vite.config.ts`); `same-origin`/`withCredentials` cookies work because the proxy keeps one origin.
- **401 handling**: `getJSON`/`getThreadsBatch` throw `{kind:'unauthorized'}`; `useAuth.refresh` maps any error to `anon`. Other hooks (`useInbox`, `useBuckets`) do not specially route 401 — they log/swallow.
- **Archived-thread eviction, not a flag** (useInbox, **new**): `applyThreadUpdates` splits LWW-accepted threads into archived vs live; archived ids are deleted outright from `displayLayer`/`idLayer`/`lastInternalDate` — there's no rendered "archived" state, the row just disappears on the next SSE-driven batch fetch. This mirrors the server's reconciling archive/un-archive (see INBOX_SYNC_INDEX) but the client only ever sees the one-way "thread left the inbox" transition; an un-archive re-adds the thread as an ordinary live merge (it's just `is_archived:false` on the next accepted update).
- **Search race guard, not cancellation** (`useInboxSearch`, shared by `HudPage`/`InboxPage`): the 300ms debounce timer is cleared per-keystroke, but an in-flight `searchInbox` fetch is never aborted — `searchSeq` just discards its result if a newer query (or a clear) has since superseded it (`seq !== searchSeq.current`). Clearing the box bumps `searchSeq` too, so a slow stale response can't repopulate results after the user backs out.
- **Search bypasses `useInbox` entirely (both routes)**: `results` is a flat one-shot array from `GET /api/search`, not merged into `idLayer`/`displayLayer`. While search is active, the shared `useInbox` instance's pagination/auto-extend/bucket filter keep running in the background but are invisible until the query is cleared and `<InboxList>` switches back to `inbox.pageThreads` (`InboxPage`) or the store-derived `recent`/bucket-count views (`HudPage`). **Updated**: `HudPage` now does have a live `useInbox` instance (the shared one, via `useInboxStore()`) but still reads it only for `idLayer`/`displayLayer`, never `page`/pagination — its search simply swaps `<InboxList>`'s source the same way it always did.
- **The SSE-pinning gotcha is gone structurally, not by convention** (**rewritten** — supersedes the old "AppShell pins the SSE singleton open" entry): `lib/sse.ts`'s `EventSource` is reference-counted by handler count — `_es` closes when the last `subscribeSse` handler unsubscribes and reopens fresh on the next subscribe. Previously, `AppShell` had to hold a dedicated no-op handler because the two routes' own subscriptions (`HudPage`'s direct `subscribeSse` call, `InboxPage`'s `useInboxSse`) each mounted/unmounted with their route, so a cross-route navigation could transiently drop the handler count to zero between the outgoing route's cleanup and the incoming route's mount — closing and reopening the connection (plus an `active_users` deregister/re-register blip) on every navigation. Now `InboxProvider`'s single `useInboxSse` subscription lives at the `AppShell` layer for the entire authed session — it mounts once and only unmounts on sign-out (`AppShell` unmount → `InboxProvider` unmount → last unsub → close). Neither route mounts or unmounts its own SSE handler anymore, so the subscriber count never transiently hits zero during `/`↔`/inbox` navigation: the close/reopen churn risk is eliminated by construction, not papered over by a pin.
- **HudPage refresh cadence — status-only now** (**updated**): `HudPage` no longer fetches its own inbox snapshot or listens for `threads_updated`/`_open` — the shared store (`InboxProvider`'s `useInboxSse`→`useInbox`) keeps `idLayer`/`displayLayer` current independently of which route is mounted, so `HudPage`'s "recently processed" list and bucket counts just re-render off the store's state (no debounce timer needed). `HudPage`'s own effect only drives sync-status cosmetics: an initial `getSyncStatus()` fetch, a 30s `refreshStatus` poll, and a 5s `setInterval` (`forceTick`) that re-renders `agoLabel` between polls — two intervals, no unsub, no debounce timer.
