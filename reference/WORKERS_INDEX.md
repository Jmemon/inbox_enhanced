# Workers Index

> Scope: Celery worker + beat subsystem ‚Äî celery_app factory (broker/backend=REDIS_URL, task_always_eager test mode), a 30s beat tick **plus** (**Phase 3**, HUD inversion) a second, hourly beat tick (`enqueue_tracker_owner_polls`) that polls tracker owners with no open SSE tab ‚Äî the 30s tick alone only reaches `active_users` ‚Äî tasks (enqueue_polls, enqueue_tracker_owner_polls, poll_new_messages, full_sync_inbox_task, extend_inbox_history_task), `last_sync:{uid}` freshness marker written on 5 sync-completion sites (read by `GET /api/sync/status`; **Phase 4**: was 6 ‚Äî `reclassify_user_inbox`, a former sync task, is deleted along with `draft_preview_bucket`, see gotchas), gmail_sync orchestration entrypoint (reconciling full sync, widened-historyTypes partial sync) driven by `_triage_batch` (Phase 2A: bucket pick + tracker relevance in one LLM call, dual-write into `inbox_threads.bucket_id` + `task_thread_links`, replacing the old classify-only `_classify_batch`), `partial_sync_inbox`/`full_sync_inbox` returning `(all_ids, content_ids)` and `extend_inbox_history` returning `(ids, more, new_link_ids)` so extraction is only ever routed off actually-fetched content or freshly-written links, never a flag-only touch (unread/archive/soft-delete) ‚Äî see gotchas, the decoupled `workers/task_engine_tasks` Celery module (process_task_updates/extract_for_thread/propose_task_draft/backfill_task) it fans out to on 5 sync-completion hook sites (including `extend_inbox_history_task`, conditional on `new_link_ids`), per-user sync_lock, active_users zset gating, cache-before-publish, pubsub user:{uid} SSE fan-out, LLM via OpenRouter with per-call llm_calls metrics persistence (app/llm/metrics.py). **Phase 4** (bucket unification): buckets are now `tasks(kind='bucket')` (migration `0009_bucket_unification.py`, no separate `Bucket` model/table); `reclassify_user_inbox`/`draft_preview_bucket` and their `_inline_reload`/`_reclassify_all`/`_extend_inline` helpers are deleted, along with `app/inbox/preview_cache.py` and `llm/classify.py`'s legacy `classify()`; bucket creation (`POST /api/buckets`, now a one-release shim over `tasks(kind='bucket')`, or `POST /api/tasks` with `kind="bucket"`) enqueues the task engine's own `backfill_task` (`workers/task_engine_tasks.py`, its `kind='bucket'` branch) instead ‚Äî a pure reclassification pass over the user's stored history, no `task_thread_links`/`task_events`/extraction. **Phase 4.5** (jobs surface): the creation wizard's goal‚Üídraft flow and post-create backfill now optionally thread a `jobs`-table row through `workers/task_engine_tasks.py`'s `propose_task_draft`/`backfill_task` (a **new** `job_id` param on each), and a **new** Celery task, `retriage_deleted_bucket`, re-triages a deleted bucket's orphaned threads ‚Äî the second lock-free `InboxThread.bucket_id` writer this module now carries alongside `_run_bucket_backfill`. All three failure paths now write a job row's `stage="failed"`+`error` via a shared `_record_job_failure` helper (fresh session) instead of leaving a client stranded on an unfinished SSE-only flow ‚Äî full jobs-domain detail (repo/API/payload shape/gotchas) lives in TASKS_INDEX.md, this doc covers only the worker-side plumbing. `tests/conftest.py` now defaults `CELERY_TASK_ALWAYS_EAGER=1` at its very top (an `os.environ.setdefault`, before any other import) rather than relying on each test file/CI env to set it ‚Äî see gotchas. Task-engine deep detail (schema/repo/transitions/API/backfill kind-branching) lives in TASKS_INDEX.md ‚Äî this doc covers only the worker-side plumbing it reuses.

## Routes / Tasks / Entrypoints

### Celery tasks (all `app.workers.tasks.*`)
| Task (name) | Signature | Trigger | Behavior (one line) |
|------|-----------|---------|---------|
| `enqueue_polls` | `()` | beat tick (30s) | `active_users.purge_expired()` then `poll_new_messages.apply_async([uid], countdown=0)` per active uid. Fan-out only. |
| `enqueue_tracker_owner_polls` | `()` | beat tick (hourly, **Phase 3**) | One `DISTINCT` query for `Task.user_id` where `kind="tracker"`, `status="active"`, `is_deleted=False`, `user_id IS NOT NULL`, `state_schema IS NOT NULL`; early-return if none qualify. `active_users.purge_expired()` then excludes any uid already in `active_users` (already covered by the 30s path). Each remaining uid gets `poll_new_messages.apply_async([uid], countdown=zlib.crc32(uid.encode()) % 3600)` ‚Äî a deterministic (not the process-salted builtin `hash()`) hourly-offset spread, so the same user always lands at the same point in the hour rather than every tracker owner's poll firing in the same instant. Fan-out only ‚Äî feeds the same `poll_new_messages` the 30s path does. |
| `poll_new_messages` | `(user_id)` | beat fan-out; API `/sse` kickoff (cursor exists), `/inbox/refresh` (cursor exists) | acquire `sync_lock` (else skip); no cursor‚Üí`full_sync_inbox`; else `fetch_history_records`‚Üí404=`HistoryGoneError`‚Üífull; empty‚Üísilent (no publish); records‚Üí`partial_sync_inbox`; publish `threads_updated` with `all_ids`. | `full_sync_inbox_task` | `(user_id)` | API `/sse` kickoff (no cursor), `/inbox/refresh` (no cursor) | lock‚Üí`gmail_sync.full_sync_inbox` (reconciling upsert: archives/un-archives, never deletes)‚Üípublish `threads_updated` with `all_ids`‚Üí**Phase 2A**: `process_task_updates.apply_async` with `content_ids` if non-empty (reconcile-archived ids are flag-only and excluded)‚Üímarks `last_sync:{uid}`. |

**Phase 4: `draft_preview_bucket` and `reclassify_user_inbox` ‚Äî the two rows this table used to carry here ‚Äî are both deleted.** `draft_preview_bucket` (`POST /buckets/draft/preview`) is superseded by the task engine's own goal‚Üídraft flow (`POST /api/tasks/draft`, `TASKS_INDEX.md`); `reclassify_user_inbox` (`POST /api/buckets`, custom bucket created) is superseded by `workers/task_engine_tasks.py`'s `backfill_task` ‚Äî bucket creation (buckets are now `tasks(kind='bucket')`) enqueues that task's `kind='bucket'` branch (`_run_bucket_backfill`) instead, which triages the user's stored history against the full active bucket set and writes only `InboxThread.bucket_id` changes (no `sync_lock`, no `task_thread_links`, no extraction) ‚Äî full behavior ‚Üí `TASKS_INDEX.md`.

### gmail_sync orchestration entrypoints (called by tasks, not Celery tasks themselves)
- `fetch_history_records(gmail, *, start_history_id)` ‚Üí `(records, latest_history_id)`; `users.history.list(historyTypes=["messageAdded","messageDeleted","labelAdded","labelRemoved"], labelId="INBOX")`; 404‚Üí`HistoryGoneError`.
- `partial_sync_inbox(db, *, user, history_records=None, new_history_id=None)` ‚Üí `(all_ids, content_ids)`; incremental writer over 4 record shapes: `messagesAdded` (fetch+**triage**+upsert, plus dual-write `task_thread_links` ‚Äî the only shape that lands in `content_ids`), `messagesDeleted` (soft-delete `is_deleted` + `recompute_thread_pointers` + archive-when-empty ‚Äî flag-only, `all_ids` only), `labelsAdded`/`labelsRemoved` INBOX (archive/un-archive, or ingest an unseen thread on INBOX-added ‚Äî ingest triggers a fetch and so lands in `content_ids`; archive/un-archive of an already-held thread is flag-only) and UNREAD (flip `is_unread` ‚Äî flag-only); per-thread try/except tolerates deleted-thread 404s so cursor advances.
- `full_sync_inbox(db, *, user)` ‚Üí `(all_ids, content_ids)`; **reconciling upsert, not a wipe**: `threads.list(maxResults=200, labelIds=["INBOX"])` ‚Üí per-stub get ‚Üí **triage** (bucket + tracker relevance) ‚Üí upsert (dual-write `task_thread_links`) ‚Äî every listed/upserted thread (new or reappearing) lands in `content_ids` ‚Üí un-archive any stored thread the listing returns (already counted via the upsert above) ‚Üí archive stored non-archived threads inside the listing's activity window but absent from it (window floor guarded against `recent_internal_date=0` messageless entries ‚Äî flag-only, `all_ids` only, excluded from `content_ids`) ‚Üí advance cursor to max history_id. `clear_user_inbox` is NOT called here (account-deletion-only, currently uncalled anywhere).
- `extend_inbox_history(db, *, user, before_internal_date_ms)` ‚Üí `(ids, more, new_link_ids)`; `threads.list(q=before:<unix_secs>, labelIds=["INBOX"])`; also **triage**s + dual-writes `task_thread_links` for each fetched thread; `new_link_ids` is the subset `_write_task_links` actually linked this call (used by `extend_inbox_history_task` to enqueue `process_task_updates`); does NOT touch `gmail_last_history_id`; caller owns lock.

## Decision points & gotchas
- **sync_lock serialization**: poll / full / extend acquire `sync_lock:{uid}` (NX, 600s TTL) to avoid racing the `(user_id, gmail_id)` unique constraint, and all three skip silently if contended. **Phase 4**: `reclassify_user_inbox` (the one caller that used to `self.retry(countdown=30)` instead of skipping) and `draft_preview_bucket`/`_extend_inline` (the task and helper that used to grab the lock transiently) are all deleted ‚Äî bucket creation's replacement (`backfill_task`'s bucket branch) never touches this lock at all, so every remaining acquirer shares the same "skip if held" behavior.
- **active-user gating (the 30s path only ‚Äî Phase 3 adds a second, hourly path)**: `enqueue_polls`'s 30s tick only polls uids in the `active_users` zset (added by `/sse` on first connect, TTL 60s, refreshed) ‚Äî it purges expired entries before listing, so disconnected users stop being polled by THIS path. A user with at least one active, non-deleted, schema-bearing tracker is no longer stranded when they close every tab, though: `enqueue_tracker_owner_polls` (hourly beat tick, **Phase 3**) polls every such owner NOT already in `active_users`, spread across the hour via a deterministic `crc32(uid) % 3600` countdown so they don't all fire in the same instant. A user with zero trackers is still only polled while a tab is open.
- **ack semantics**: `task_acks_late=False` (ack on receipt, not completion) ‚Äî periodic poll is best-effort; a worker crash mid-task loses that run (next beat tick re-polls).
- **commit-before-publish ordering, now a Postgres row instead of a Redis cache**: this module's own writer tasks don't own a cache anymore (**Phase 4**: `draft_preview_bucket`'s `preview_cache.store_result`-before-`_publish` pattern is gone along with the task and `app/inbox/preview_cache.py` itself; **Phase 4.5**: the task engine's OWN successor pattern ‚Äî `draft_cache.store_result`-before-`task_draft_ready` ‚Äî is gone too, `draft_cache.py` deleted outright) ‚Äî the same ordering rule now lives entirely in `workers/task_engine_tasks.py`'s `propose_task_draft` (`jobs_repo.set_payload`+`update_stage("draft_ready")`, commit, THEN publish `job_updated`, see TASKS_INDEX). There's no `mark_pending`-at-enqueue-time step anymore either ‚Äî `POST /api/jobs` (the current enqueue site) just `create_job`s the row (already visible to `GET /api/jobs` the instant it's committed, in whatever `stage` `create_job` gave it), so there's no "cache not written yet" 404 window to guard against in the first place.
- **full_sync_inbox is reconciling, not destructive**: it re-lists the latest 200 `labelIds=["INBOX"]` threads and upserts (archives/un-archives based on presence in that listing) ‚Äî it never deletes rows. The old wipe-then-repopulate approach was removed because Phase 2 task tables FK onto `inbox_threads.id` and `HistoryGoneError` recovery (which calls this fn) must not be destructive. `clear_user_inbox` (hard delete) still exists in `inbox_repo` but is account-deletion-only and has zero call sites in the sync/worker path.
- **HistoryGoneError / cursor expiry**: `fetch_history_records` 404 (cursor older than ~30-day window) ‚Üí fall back to `full_sync_inbox` (reconciling upsert, see above ‚Äî NOT a wipe). Same fallback in `poll_new_messages` (**Phase 4**: `_inline_reload`, a former second caller of this fallback, is deleted along with `reclassify_user_inbox`). `extend_inbox_history` deliberately leaves cursor untouched (must stay anchored at newest message).
- **widened historyTypes**: partial sync now also mirrors Gmail-side archive (`labelsAdded`/`labelsRemoved` INBOX), soft-delete (`messagesDeleted` ‚Üí `is_deleted` + pointer recompute + archive-when-empty), and read state (`UNREAD` label ‚Üí `InboxMessage.is_unread`) ‚Äî not just new messages. A thread with INBOX freshly added that this worker has never stored is ingested on the spot rather than dropped.
- **Reclassification and scoring paths are Postgres-only, no Gmail refetch.** `_score_all` (now called only by `propose_task_draft`) calls `inbox_repo.load_parsed_threads` to rebuild `ParsedThread`s from stored `body_text`/`labels` rows instead of looping `gmail.threads().get()` per candidate, so that scoring path is bound by LLM concurrency (`LLM_CONCURRENCY`), not by sequential Gmail API round-trips. **Phase 4**: `_reclassify_all`, this bullet's other original caller, is deleted along with `reclassify_user_inbox` ‚Äî a bucket's own reclassify-on-creation (`_run_bucket_backfill`, `workers/task_engine_tasks.py`) follows the identical Postgres-only `load_parsed_threads` pattern in its own module instead (TASKS_INDEX), so a large bucket backfill is likewise LLM-bound, not Gmail-API-bound.
- **noise filter**: empty history records ‚Üí `poll_new_messages` returns without publishing (no `subscribers=0` churn).
- **score thresholds**: positives `score >= 7`, near-misses `4 <= score <= 6` ‚Äî `workers/tasks.py`'s `POSITIVE_THRESHOLD`/`NEAR_MISS_LOW`/`NEAR_MISS_HIGH` constants, kept alive solely because `task_engine_tasks.propose_task_draft` still imports them (top-3/top-3 was `draft_preview_bucket`'s own cut, now deleted; `propose_task_draft` applies the same score bands but takes its own top-`PROPOSE_TOP_N=5` ‚Äî see TASKS_INDEX).
- **INBOX scoping**: `labelId="INBOX"` (singular, history.list) / `labelIds=["INBOX"]` (plural, threads.list) ‚Äî without it SENT/DRAFTS leak into the inbox table.
- **beat single-replica**: beat service `numReplicas=1` (multiple beats multiply Gmail fan-out). Beat owns no app code beyond `beat_schedule.py`.
- **eager mode**: `CELERY_TASK_ALWAYS_EAGER=1` (tests) runs tasks synchronously in-process with `task_eager_propagates=True`; `SessionLocal` is module-rebindable for in-memory engines. **Phase 4.5**: `server/tests/conftest.py` now sets this env var itself, via `os.environ.setdefault("CELERY_TASK_ALWAYS_EAGER", "1")` as the very first executable line (before `REDIS_URL`/`DATABASE_URL`/etc.) ‚Äî `celery_app` freezes this value at its own first import, and pytest collects test files alphabetically, so conftest (which always loads first) must set it before any test module's import chain can reach `celery_app`. `setdefault` (not a plain assignment) so an externally-set value ‚Äî e.g. a CI env var ‚Äî still wins; this closes an import-order hazard where a test file that happened to import something celery-adjacent before this env var was set would freeze eager mode off for the whole process.
- **alembic NOT run by worker/beat**: Dockerfile chains `alembic upgrade head` before uvicorn, but worker/beat override CMD ‚Äî only the API container migrates. Migration `0006_data_floor` dialect-guards its FTS DDL (generated tsvector columns, GIN indexes, `pg_trgm`) behind a Postgres check so it still applies cleanly against the SQLite test engine (which just skips that block).
- **fixed countdown=0**: `enqueue_polls` enqueues with `countdown=0` for test determinism (the docstring notes prod could randomize a 0‚Äì10s spread).
- **llm_calls is fire-and-forget metrics, not a source of truth for control flow**: `record_call` swallows every exception and never raises ‚Äî a Postgres outage degrades to "no metrics row," never to a failed classify/score/preview call. Success vs. error rows are distinguished by content-extraction succeeding (`resp.choices[0].message.content`) BEFORE the metrics call ‚Äî a malformed 200 response records as `outcome="error"`, not a phantom success. The `record_call` write happens AFTER `call_messages` releases `Semaphore(LLM_CONCURRENCY)`, not inside it ‚Äî only the OpenRouter round-trip itself is concurrency-bounded.
- **`last_sync:{uid}` TTL is a self-cleaning ceiling, not a freshness expiry**: `mark()` sets a 90-day TTL refreshed on every call, so it never lapses for a user who keeps syncing ‚Äî it only exists so an abandoned/deleted account's marker doesn't live in Redis forever. Don't confuse this with "how fresh is the sync" (that's `GET /api/sync/status`'s `last_synced_at` computation on the client, not a Redis expiry).
- **`last_sync:{uid}` tracks sync attempts, not sync changes**: marked on every successful `poll_new_messages`/`full_sync_inbox_task` completion ‚Äî including `poll_new_messages`'s 0-history-records no-op exit ("a successful check IS a sync, even with nothing new," per the inline comment) and its `HistoryGoneError`-recovery full-sync path. So `GET /api/sync/status`'s `last_synced_at` means "last time this user's inbox was successfully checked against Gmail," not "last time something changed." `extend_inbox_history_task` (pulls older history on demand, not a periodic sync) deliberately doesn't mark it ‚Äî doing so would make an on-demand action look like a fresh periodic sync tick on the HUD. **Phase 4**: `reclassify_user_inbox` (a former marker) and `draft_preview_bucket` (a former non-marker, for the same never-touches-the-cursor reason) are both deleted; bucket creation's replacement (`backfill_task`'s bucket branch) isn't a sync-completion event at all and correctly never marks it either.
- **`_triage_batch` replaces `_classify_batch` everywhere on the sync path (Phase 2A)**: one Haiku call per thread now returns both the bucket pick AND tracker relevance (`llm.classify.triage()`, D2 ‚Äî no doubled LLM volume vs. two separate calls), and every one of `partial_sync_inbox`/`full_sync_inbox`/`extend_inbox_history` dual-writes: the bucket half feeds the existing `InboxThread.bucket_id` upsert exactly as before, the tracker half (`>= TASK_LINK_CONFIDENCE`) upserts `origin='llm'` `task_thread_links` via `_write_task_links`. With zero active trackers the bucket output is byte-for-byte identical to the old classify-only path (`triage_thread`'s prompt delegates its bucket section to `classify_thread`). Full validator/confidence detail ‚Üí `TASKS_INDEX.md`.
- **flag-only touches never feed extraction (Fix, was a cost leak)**: `partial_sync_inbox`/`full_sync_inbox` return `(all_ids, content_ids)` ‚Äî `all_ids` (published via `threads_updated`) includes flag-only flips (unread, archive/unarchive, soft-delete, reconcile-archive), but `content_ids` (routed to `process_task_updates`) is only the subset actually (re)fetched from Gmail this round. Before this fix, callers enqueued extraction off the same merged id list they published, so e.g. reading/archiving an already-tracked email fired a Sonnet extraction call per active tracker that deduped against unchanged evidence only AFTER the LLM cost was spent ‚Äî and a `HistoryGoneError`-triggered `full_sync_inbox` (up to 200 threads) amplified that waste up to 200√ó.
- **`extend_inbox_history_task` DOES trigger extraction, but only for freshly-linked threads (Fix)**: `extend_inbox_history` still calls `_triage_batch`/`_write_task_links` for every thread it fetches (so a tracker can pick up an old thread pulled in via "load more history"), and now also returns `new_link_ids` ‚Äî the subset `_write_task_links` actually linked this call. `extend_inbox_history_task` enqueues `process_task_updates.apply_async([uid, new_link_ids])` when non-empty. (Before this fix, extend never enqueued `process_task_updates` at all ‚Äî any thread it linked sat attached-but-unextracted until an unrelated future sync tick re-touched it.) **Phase 4**: `draft_preview_bucket`, the bullet this used to point at as "the only sync-adjacent task that never enqueues it," is deleted ‚Äî its bucket-kind replacement (`backfill_task`'s bucket branch, `workers/task_engine_tasks.py`) inherits that same "never enqueues `process_task_updates`" property, for the same reason: no `task_thread_links`/tracker awareness of any kind (see TASKS_INDEX).
