# Workers Index

> Scope: Celery worker + beat subsystem — celery_app factory (broker/backend=REDIS_URL, task_always_eager test mode), 30s beat tick, tasks (enqueue_polls, poll_new_messages, full_sync_inbox_task, extend_inbox_history_task, draft_preview_bucket, reclassify_user_inbox), `last_sync:{uid}` freshness marker written on 6 sync-completion sites (read by `GET /api/sync/status`), gmail_sync orchestration entrypoint (reconciling full sync, widened-historyTypes partial sync) now driven by `_triage_batch` (Phase 2A: bucket pick + tracker relevance in one LLM call, dual-write into `inbox_threads.bucket_id` + `task_thread_links`, replacing the old classify-only `_classify_batch`), `partial_sync_inbox`/`full_sync_inbox` returning `(all_ids, content_ids)` and `extend_inbox_history` returning `(ids, more, new_link_ids)` so extraction is only ever routed off actually-fetched content or freshly-written links, never a flag-only touch (unread/archive/soft-delete) — see gotchas, the decoupled `workers/task_engine_tasks` Celery module (process_task_updates/extract_for_thread/propose_task_draft/backfill_task) it fans out to on 6 sync-completion hook sites (including `extend_inbox_history_task`, conditional on `new_link_ids`), per-user sync_lock, active_users zset gating, cache-before-publish, pubsub user:{uid} SSE fan-out, LLM via OpenRouter with per-call llm_calls metrics persistence (app/llm/metrics.py). Task-engine deep detail (schema/repo/transitions/API) lives in TASKS_INDEX.md — this doc covers only the worker-side plumbing it reuses.

| server/app/llm/client.py | `call_messages(*, model, system, user, max_tokens=1024, stage="unknown", user_id=None, task_id=None) -> str` — one `AsyncOpenAI.chat.completions.create` call under the shared `Semaphore(LLM_CONCURRENCY)` on the dedicated `llm-loop` background thread; `extra_body={"usage":{"include": True}}` asks OpenRouter to attach cost + cached-token counts to `resp.usage`. `task_id` (**Phase 2A**: threaded through by the extraction stage only — `propose`/`classify` stages pass `user_id` alone, since a propose call has no task yet and a classify/triage call covers many tasks at once) flows straight to the `llm_calls` row for cost attribution. | server/app/llm/metrics.py | `record_call(*, stage, model, user_id=None, task_id=None, input_tokens=None, output_tokens=None, cache_read_tokens=None, cache_write_tokens=None, cost_usd=None, ttft_ms=None, duration_ms, outcome)` — writes one `LlmCall` row. Deliberately fire-and-forget: own short-lived `SessionLocal()` (module-attr, monkeypatchable like `workers/tasks.py`), wraps the whole body in `try/except Exception: log.exception(...)` — a metrics-write failure must never fail (or retry) the LLM call it's recording. | server/app/llm/classify.py | `classify(threads, buckets, current_bucket_ids, *, user_id=None) -> list[str \| None]` — one call per thread in parallel via `asyncio.gather` under the semaphore (through `client.run_in_loop`); output order matches input order. `_classify_one` passes `stage="classify", user_id=user_id` into `call_messages` for metrics attribution; no-fit response falls through to `current_bucket_id` (keeps existing assignment, `None` for new threads). **Phase 2A** adds `triage(threads, buckets, trackers, current_bucket_ids, *, user_id=None) -> list[tuple[str \| None, list[tuple[str, int]]]]` — the call that replaces `classify()` on the sync path (D2, no doubled LLM volume): one call per thread returns both the bucket pick and `[(task_id, confidence 0-100), ...]` tracker relevance, still `stage="classify"` (same cost accounting). `classify()` itself is unchanged but as of this stamp has no production callers left — `gmail_sync._triage_batch`/`tasks._reclassify_all` both call `triage()` instead; full detail (validator, confidence thresholds, dual-write) → `TASKS_INDEX.md`. |
| server/app/db/models.py | `LlmCall` (`server/app/db/models.py`, table `llm_calls`) — `id, user_id, task_id, stage, model, input_tokens, output_tokens, cache_read_tokens, cache_write_tokens, cost_usd, ttft_ms, duration_ms, outcome, created_at`; indexed on `user_id` and `created_at`. ## Routes / Tasks / Entrypoints

### Celery tasks (all `app.workers.tasks.*`)
| Task (name) | Signature | Trigger | Behavior (one line) |
|------|-----------|---------|---------|
| `enqueue_polls` | `()` | beat tick (30s) | `active_users.purge_expired()` then `poll_new_messages.apply_async([uid], countdown=0)` per active uid. Fan-out only. |
| `poll_new_messages` | `(user_id)` | beat fan-out; API `/sse` kickoff (cursor exists), `/inbox/refresh` (cursor exists) | acquire `sync_lock` (else skip); no cursor→`full_sync_inbox`; else `fetch_history_records`→404=`HistoryGoneError`→full; empty→silent (no publish); records→`partial_sync_inbox`; publish `threads_updated` with `all_ids`. Marks `last_sync:{uid}` on all 4 exit paths — no-cursor-full, 404-fallback-full, 0-records, partial-complete. **Phase 2A**: the 3 exit paths that DO publish (no-cursor-full, 404-fallback-full, partial) also fire `task_engine_tasks.process_task_updates.apply_async([user_id, content_ids])` whenever `content_ids` is non-empty (after the `threads_updated` publish, before `last_sync.mark`) — a flag-only sync (unread flip, archive/unarchive, soft-delete) publishes `all_ids` but never enqueues extraction. |
| `full_sync_inbox_task` | `(user_id)` | API `/sse` kickoff (no cursor), `/inbox/refresh` (no cursor) | lock→`gmail_sync.full_sync_inbox` (reconciling upsert: archives/un-archives, never deletes)→publish `threads_updated` with `all_ids`→**Phase 2A**: `process_task_updates.apply_async` with `content_ids` if non-empty (reconcile-archived ids are flag-only and excluded)→marks `last_sync:{uid}`. |
| `extend_inbox_history_task` | `(user_id, before_internal_date_ms)` | API `POST /inbox/extend` (202) | lock→`gmail_sync.extend_inbox_history`→publish `extend_complete {thread_ids, more}` (more=True when 200 stubs returned). Does **not** mark `last_sync`. **Phase 2A**: DOES enqueue `process_task_updates.apply_async` with `new_link_ids` (threads `extend_inbox_history` just linked to a tracker) when non-empty — fixed from an earlier version that dual-wrote `task_thread_links` here but never triggered extraction at all (see gotchas). |
| `draft_preview_bucket` | `(user_id, draft_id, name, description, exclude_thread_ids=None)` | API `POST /buckets/draft/preview` (202) | read ≤100 candidates; pool<100→`_extend_inline` (takes lock); `_score_all` (Postgres bodies via `load_parsed_threads` + parallel LLM — **no Gmail refetch**); top-3 ≥7 / top-3 4–6; **`preview_cache.store_result` BEFORE** publish `bucket_draft_preview`. NO sync_lock held for the whole task. Does **not** mark `last_sync` nor enqueue `process_task_updates` — read-only scoring, never touches `inbox_threads`/task links (see gotchas). |
| `reclassify_user_inbox` | `(self, user_id)` `bind=True, max_retries=3` | API `POST /buckets` (custom bucket created) | lock (contended→`self.retry(countdown=30)`); `_inline_reload`(partial/full, returns `(all_ids, content_ids)`) + `_reclassify_all` (Postgres bodies via `load_parsed_threads`, **no Gmail refetch** — `triage()` w/ current bucket as stability hint, write only changed bucket assignments + upsert every `task_thread_links` hit `>= TASK_LINK_CONFIDENCE`, returns bucket-changed ids UNION freshly-linked ids); publish union of `all_ids ∪ reclassified` via `threads_updated`→**Phase 2A**: `process_task_updates.apply_async` with the union of `content_ids ∪ reclassified` if non-empty (narrower than the publish set — a flag-only inline-reload touch never enqueues extraction on its own)→marks `last_sync:{uid}`. |

### gmail_sync orchestration entrypoints (called by tasks, not Celery tasks themselves)
- `fetch_history_records(gmail, *, start_history_id)` → `(records, latest_history_id)`; `users.history.list(historyTypes=["messageAdded","messageDeleted","labelAdded","labelRemoved"], labelId="INBOX")`; 404→`HistoryGoneError`.
- `partial_sync_inbox(db, *, user, history_records=None, new_history_id=None)` → `(all_ids, content_ids)`; incremental writer over 4 record shapes: `messagesAdded` (fetch+**triage**+upsert, plus dual-write `task_thread_links` — the only shape that lands in `content_ids`), `messagesDeleted` (soft-delete `is_deleted` + `recompute_thread_pointers` + archive-when-empty — flag-only, `all_ids` only), `labelsAdded`/`labelsRemoved` INBOX (archive/un-archive, or ingest an unseen thread on INBOX-added — ingest triggers a fetch and so lands in `content_ids`; archive/un-archive of an already-held thread is flag-only) and UNREAD (flip `is_unread` — flag-only); per-thread try/except tolerates deleted-thread 404s so cursor advances.
- `full_sync_inbox(db, *, user)` → `(all_ids, content_ids)`; **reconciling upsert, not a wipe**: `threads.list(maxResults=200, labelIds=["INBOX"])` → per-stub get → **triage** (bucket + tracker relevance) → upsert (dual-write `task_thread_links`) — every listed/upserted thread (new or reappearing) lands in `content_ids` → un-archive any stored thread the listing returns (already counted via the upsert above) → archive stored non-archived threads inside the listing's activity window but absent from it (window floor guarded against `recent_internal_date=0` messageless entries — flag-only, `all_ids` only, excluded from `content_ids`) → advance cursor to max history_id. `clear_user_inbox` is NOT called here (account-deletion-only, currently uncalled anywhere).
- `extend_inbox_history(db, *, user, before_internal_date_ms)` → `(ids, more, new_link_ids)`; `threads.list(q=before:<unix_secs>, labelIds=["INBOX"])`; also **triage**s + dual-writes `task_thread_links` for each fetched thread; `new_link_ids` is the subset `_write_task_links` actually linked this call (used by `extend_inbox_history_task` to enqueue `process_task_updates`); does NOT touch `gmail_last_history_id`; caller owns lock.

```
poll_new_messages | full_sync_inbox_task | reclassify_user_inbox | extend_inbox_history_task    (Phase 2A)
  ──[if content_ids / new_link_ids non-empty — never a flag-only touch]──>
  apply_async process_task_updates(uid, thread_ids) ──> Worker.task_engine_tasks (own module, own txn)
    ──[per attached∩touched pair: 1 LLM extract call]──> task_events/task_state_entities
    ──[PUBLISH user:{uid} task_updated]──> Redis ──> SSE ──> Browser        (full detail → TASKS_INDEX.md)
```
Published events → browser over SSE: `threads_updated {thread_ids}` (poll/full/reclassify), `extend_complete {thread_ids, more}` (extend), `bucket_draft_preview {draft_id, positives, near_misses}` (preview), and — from the separate `task_engine_tasks` module — `task_draft_ready`/`task_updated`/`task_backfill_progress` (TASKS_INDEX.md). Browser then re-reads via `/api/threads/batch` (filters by `InboxThread.id` — why workers return internal ids, not gmail ids). Preview also pollable via `GET /api/buckets/draft/preview/{draft_id}` (preview cache fallback). Both `_reclassify_all` and `_score_all` skip the `Gmail v1 threads.get` step entirely — they read `inbox_repo.load_parsed_threads` (Postgres) instead, so reclassify/preview latency is now LLM-bound, not Gmail-API-bound.

## Decision points & gotchas
- **sync_lock serialization**: poll / full / extend / reclassify acquire `sync_lock:{uid}` (NX, 600s TTL) to avoid racing the `(user_id, gmail_id)` unique constraint. Contended → poll/full/extend skip silently; `reclassify_user_inbox` instead `self.retry(countdown=30)`. `draft_preview_bucket` does NOT hold it across the task — only `_extend_inline` grabs it transiently (skips if held).
- **active-user gating**: only uids in the `active_users` zset (added by `/sse` on first connect, TTL 60s, refreshed) get polled. `enqueue_polls` purges expired before listing — disconnected users stop being polled.
- **ack semantics**: `task_acks_late=False` (ack on receipt, not completion) — periodic poll is best-effort; a worker crash mid-task loses that run (next beat tick re-polls).
- **cache-before-publish ordering**: `draft_preview_bucket` calls `preview_cache.store_result` BEFORE `_publish` so a client polling between the two reads the ready result, not stale "pending". `mark_pending` is written by the API at enqueue time so the GET never 404s a fast poller.
- **full_sync_inbox is reconciling, not destructive**: it re-lists the latest 200 `labelIds=["INBOX"]` threads and upserts (archives/un-archives based on presence in that listing) — it never deletes rows. The old wipe-then-repopulate approach was removed because Phase 2 task tables FK onto `inbox_threads.id` and `HistoryGoneError` recovery (which calls this fn) must not be destructive. `clear_user_inbox` (hard delete) still exists in `inbox_repo` but is account-deletion-only and has zero call sites in the sync/worker path.
- **HistoryGoneError / cursor expiry**: `fetch_history_records` 404 (cursor older than ~30-day window) → fall back to `full_sync_inbox` (reconciling upsert, see above — NOT a wipe). Same fallback in `poll_new_messages` and `_inline_reload`. `extend_inbox_history` deliberately leaves cursor untouched (must stay anchored at newest message).
- **widened historyTypes**: partial sync now also mirrors Gmail-side archive (`labelsAdded`/`labelsRemoved` INBOX), soft-delete (`messagesDeleted` → `is_deleted` + pointer recompute + archive-when-empty), and read state (`UNREAD` label → `InboxMessage.is_unread`) — not just new messages. A thread with INBOX freshly added that this worker has never stored is ingested on the spot rather than dropped.
- **reclassify/score paths are Postgres-only, no Gmail refetch**: `_reclassify_all` and `_score_all` both call `inbox_repo.load_parsed_threads` to rebuild `ParsedThread`s from stored `body_text`/`labels` rows instead of looping `gmail.threads().get()` per candidate — a 200-thread reclassify is now bound by LLM concurrency (`LLM_CONCURRENCY`), not by 200 sequential Gmail API round-trips.
- **noise filter**: empty history records → `poll_new_messages` returns without publishing (no `subscribers=0` churn).
- **score thresholds (preview)**: positives `score >= 7` (top 3), near-misses `4 <= score <= 6` (top 3).
- **INBOX scoping**: `labelId="INBOX"` (singular, history.list) / `labelIds=["INBOX"]` (plural, threads.list) — without it SENT/DRAFTS leak into the inbox table.
- **beat single-replica**: beat service `numReplicas=1` (multiple beats multiply Gmail fan-out). Beat owns no app code beyond `beat_schedule.py`.
- **eager mode**: `CELERY_TASK_ALWAYS_EAGER=1` (tests) runs tasks synchronously in-process with `task_eager_propagates=True`; `SessionLocal` is module-rebindable for in-memory engines.
- **alembic NOT run by worker/beat**: Dockerfile chains `alembic upgrade head` before uvicorn, but worker/beat override CMD — only the API container migrates. Migration `0006_data_floor` dialect-guards its FTS DDL (generated tsvector columns, GIN indexes, `pg_trgm`) behind a Postgres check so it still applies cleanly against the SQLite test engine (which just skips that block).
- **fixed countdown=0**: `enqueue_polls` enqueues with `countdown=0` for test determinism (the docstring notes prod could randomize a 0–10s spread).
- **llm_calls is fire-and-forget metrics, not a source of truth for control flow**: `record_call` swallows every exception and never raises — a Postgres outage degrades to "no metrics row," never to a failed classify/score/preview call. Success vs. error rows are distinguished by content-extraction succeeding (`resp.choices[0].message.content`) BEFORE the metrics call — a malformed 200 response records as `outcome="error"`, not a phantom success. The `record_call` write happens AFTER `call_messages` releases `Semaphore(LLM_CONCURRENCY)`, not inside it — only the OpenRouter round-trip itself is concurrency-bounded.
- **`last_sync:{uid}` TTL is a self-cleaning ceiling, not a freshness expiry**: `mark()` sets a 90-day TTL refreshed on every call, so it never lapses for a user who keeps syncing — it only exists so an abandoned/deleted account's marker doesn't live in Redis forever. Don't confuse this with "how fresh is the sync" (that's `GET /api/sync/status`'s `last_synced_at` computation on the client, not a Redis expiry).
- **`last_sync:{uid}` tracks sync attempts, not sync changes** (**new**): marked on every successful `poll_new_messages`/`full_sync_inbox_task`/`reclassify_user_inbox` completion — including `poll_new_messages`'s 0-history-records no-op exit ("a successful check IS a sync, even with nothing new," per the inline comment) and its `HistoryGoneError`-recovery full-sync path. So `GET /api/sync/status`'s `last_synced_at` means "last time this user's inbox was successfully checked against Gmail," not "last time something changed." `extend_inbox_history_task` (pulls older history on demand, not a periodic sync) and `draft_preview_bucket` (scores/reads only, never touches the cursor) deliberately don't mark it — doing so would make an on-demand action look like a fresh periodic sync tick on the HUD.
- **`_triage_batch` replaces `_classify_batch` everywhere on the sync path (Phase 2A)**: one Haiku call per thread now returns both the bucket pick AND tracker relevance (`llm.classify.triage()`, D2 — no doubled LLM volume vs. two separate calls), and every one of `partial_sync_inbox`/`full_sync_inbox`/`extend_inbox_history` dual-writes: the bucket half feeds the existing `InboxThread.bucket_id` upsert exactly as before, the tracker half (`>= TASK_LINK_CONFIDENCE`) upserts `origin='llm'` `task_thread_links` via `_write_task_links`. With zero active trackers the bucket output is byte-for-byte identical to the old classify-only path (`triage_thread`'s prompt delegates its bucket section to `classify_thread`). Full validator/confidence detail → `TASKS_INDEX.md`.
- **flag-only touches never feed extraction (Fix, was a cost leak)**: `partial_sync_inbox`/`full_sync_inbox` return `(all_ids, content_ids)` — `all_ids` (published via `threads_updated`) includes flag-only flips (unread, archive/unarchive, soft-delete, reconcile-archive), but `content_ids` (routed to `process_task_updates`) is only the subset actually (re)fetched from Gmail this round. Before this fix, callers enqueued extraction off the same merged id list they published, so e.g. reading/archiving an already-tracked email fired a Sonnet extraction call per active tracker that deduped against unchanged evidence only AFTER the LLM cost was spent — and a `HistoryGoneError`-triggered `full_sync_inbox` (up to 200 threads) amplified that waste up to 200×.
- **`extend_inbox_history_task` DOES trigger extraction, but only for freshly-linked threads (Fix)**: `extend_inbox_history` still calls `_triage_batch`/`_write_task_links` for every thread it fetches (so a tracker can pick up an old thread pulled in via "load more history"), and now also returns `new_link_ids` — the subset `_write_task_links` actually linked this call. `extend_inbox_history_task` enqueues `process_task_updates.apply_async([uid, new_link_ids])` when non-empty. (Before this fix, extend never enqueued `process_task_updates` at all — any thread it linked sat attached-but-unextracted until an unrelated future sync tick re-touched it.) `draft_preview_bucket` remains the only sync-adjacent task that never enqueues it — see below.
- **`draft_preview_bucket` never touches task links at all**: it's a read-only scoring task (bucket drafts only) with no tracker awareness — it doesn't call `triage()`/`_write_task_links`/`process_task_updates`. Don't confuse it with `propose_task_draft` (the task-engine's own, separate goal→draft flow in `workers/task_engine_tasks.py`, see TASKS_INDEX) — same cache-before-publish shape, different subsystem, different Redis key prefix (`preview:{draft_id}` vs. `task_draft:{draft_id}`).
