# Inbox Sync Index

> Scope: three-way inbox synchronization — Gmail ↔ Postgres (`inbox_threads`/`inbox_messages`) ↔ browser. Full sync = reconciling upsert (never wipes; archives absent-from-window threads, un-archives listed ones); partial sync mirrors Gmail archive/soft-delete/unread via widened `historyTypes`; extend sync; the Gmail history cursor (`gmail_last_history_id`) + `HistoryGoneError`; the Celery poll/full/extend tasks (bucket+tracker classification via `triage()`, Phase 2A — see WORKERS_INDEX/TASKS_INDEX; **Phase 4**: the former 4th sync-adjacent task, `reclassify_user_inbox`, is deleted — buckets are now `tasks(kind='bucket')`, and creating one runs the task engine's own `backfill_task` instead, see gotchas); `sync_lock`/`active_users` Redis gates plus (**Phase 3**) a second, hourly `enqueue_tracker_owner_polls` beat entry that polls tracker owners with no open SSE tab — the 30s `active_users` path alone no longer decides who gets synced, see gotchas; the `last_sync:{uid}` freshness marker + `GET /api/sync/status`; the `user:{uid}` pubsub→SSE push path (poll/full/extend also fan out `process_task_updates` after publish — gated on content-touched / freshly-linked ids only, never flag-only touches; see gotchas); `GET /api/search` (Postgres FTS/ILIKE); the client `idLayer`/`displayLayer` LWW merge + auto-extend + archived-thread eviction + shared `useInboxSearch` (HudPage/InboxPage).

This is a cross-cutting doc: it threads the Worker, Data, Realtime, API, and Client
subsystems along the single axis of "keep the inbox in sync." For depth on any one
layer see its dedicated index (`WORKERS_INDEX.md`, `CLIENT_INDEX.md`; deeper still,
`ARCHITECTURE.md`).

## Routes / Tasks / Entrypoints

**HTTP (API process):**
- `GET /api/sse` — browser opens one per tab. First conn for a user → pubsub subscribe + `active_users.add` + kickoff sync task. Streams JSON `data:` frames + 5s keepalive (each keepalive also refreshes the `active_users` TTL).
- `GET /api/inbox?limit=&page=&include_archived=` — paginated list, most-recently-active first (`last_activity_at` desc); returns `{as_of, page, limit, threads}`. `limit` clamps 1–200, default 50; client fetches 200. Archived threads hidden unless `include_archived=true`.
- `GET /api/threads/{id}` — single thread (rare fallback); 404 if not owned.
- `POST /api/threads/batch {thread_ids}` — SSE-replay bulk fetch; ≤500 ids; foreign/unknown silently dropped.
- `POST /api/inbox/refresh` — 202; Reload button. Branches on cursor: present → `poll_new_messages`, absent → `full_sync_inbox_task`. Result arrives via SSE.
- `POST /api/inbox/extend {before_internal_date}` — 202; enqueues `extend_inbox_history_task`. Result via SSE `extend_complete`.
- `GET /api/search?q=&limit=&page=&include_archived=` — same response shape as `/api/inbox`; server-side text search (see `search_repo` in Files). Synchronous request/response, no SSE/task involved.
- `GET /api/sync/status` (**new**) — `{last_synced_at, has_cursor}`; reads `last_sync:{uid}` (Redis) + `User.gmail_last_history_id` (Postgres). Synchronous, no Celery/SSE. Powers the client HUD freshness strip.

**Celery tasks (worker process):**
- `enqueue_polls` — beat-fired (30s). `purge_expired()` → `list_active()` → `poll_new_messages.apply_async([uid], countdown=0)` per active uid.
- `enqueue_tracker_owner_polls` — beat-fired (hourly, **Phase 3**). DISTINCT query for active/non-deleted/schema-bearing tracker owners (`Task.user_id`) → `purge_expired()` → excludes any uid already in `active_users` → `poll_new_messages.apply_async([uid], countdown=zlib.crc32(uid.encode())%3600)` per remaining uid — keeps a tracker's pipeline advancing even when its owner has no open tab. Full detail → WORKERS_INDEX.md.
- `poll_new_messages(uid)` — `sync_lock.acquire` (silent return if held). No cursor → `full_sync_inbox`. Else `fetch_history_records`: 404 → `full_sync_inbox`; empty → silent (no publish); records → `partial_sync_inbox(records, new_id)`. Publishes `threads_updated` with `all_ids` (internal ids). **Phase 2A**: on the 3 exit paths that publish, also `apply_async`s `process_task_updates(uid, content_ids)` if `content_ids` is non-empty — a flag-only sync (unread flip/archive/soft-delete) publishes but never enqueues extraction. Marks `last_sync:{uid}` on all 4 exit paths (see WORKERS_INDEX).
- `full_sync_inbox_task(uid)` — locked full sync; SSE kickoff (no cursor) + refresh (no cursor). **Phase 2A**: `apply_async process_task_updates(uid, content_ids)` if non-empty (reconcile-archived ids are flag-only, excluded). Marks `last_sync:{uid}`.
- `extend_inbox_history_task(uid, before_ms)` — locked; `extend_inbox_history` (**Phase 2A**: also triages + dual-writes `task_thread_links` for every fetched thread); publishes `extend_complete {thread_ids, more}`; then `apply_async process_task_updates(uid, new_link_ids)` if any thread was freshly linked to a tracker — a thread pulled in via "load more history" that matches a tracker gets extracted right away instead of sitting attached-but-unextracted. Does **not** mark `last_sync` (on-demand history pull, not a periodic sync — see WORKERS_INDEX gotchas).
- **Phase 4: `reclassify_user_inbox` is deleted** — along with its `_inline_reload`/`_reclassify_all` helper privates. `POST /api/buckets` (bucket creation; buckets are now `tasks(kind='bucket')`) no longer fires a bespoke Gmail-adjacent reclassify task at all — it enqueues the task engine's own `backfill_task` (kind='bucket' branch, `workers/task_engine_tasks.py`) the exact same way a new tracker's creation does. That branch never touches `sync_lock`/Gmail/the history cursor (it only reads stored Postgres threads via `load_parsed_threads` and reclassifies `InboxThread.bucket_id`), so it's no longer one of this doc's Gmail-sync entrypoints at all — full detail (candidate pool, optimistic write guard, `threads_updated` publish) → `TASKS_INDEX.md`.
- `process_task_updates(uid, thread_ids)` / `extract_for_thread` / `propose_task_draft` / `backfill_task` — separate `workers/task_engine_tasks.py` module (own docstring: decoupled so LLM-latency-bound extraction can't compete with the Gmail-API-bound sync queue). Full behavior → `TASKS_INDEX.md`; this doc only tracks the enqueue edge from the sync tasks above.

**Beat:** `enqueue-polls-every-30s` → `enqueue_polls`; `poll-tracker-owners-hourly` (**Phase 3**) → `enqueue_tracker_owner_polls`. One replica only.

**Client hooks:** `useInboxSse` (snapshot+buffer lifecycle) drives `useInbox.applyThreadUpdates`; `useInbox` owns pagination + auto-extend + archived-thread eviction — both mounted only by `InboxPage` (**new** path — `Home.tsx` was deleted, moved to `pages/inbox/InboxPage.tsx`). The shared `useInboxSearch()` hook (`pages/search/`, **new** — used identically by `HudPage` and `InboxPage`, see CLIENT_INDEX) owns the search box (debounced 300ms, monotonic-token race guard) that swaps the mounted page's rendered list to `GET /api/search` results while a query is active.

## Data & state touched

**Postgres (system of record):**
- `inbox_threads` — write: thread upserts (full/partial/extend), `recent_message_id`/`last_activity_at` pointers (via `recompute_thread_pointers`), `is_archived` flips (partial-sync label events; full-sync reconciliation). `bucket_id` reassignment is no longer written by anything in this sync-path module (**Phase 4**: `reclassify_user_inbox`, the old in-place reclassify writer, is deleted) — it's now written only by the triage half of a fresh thread's initial classify (this module) or by `workers/task_engine_tasks.py`'s `_run_bucket_backfill` (a bucket's own backfill scan, own module, own `sync_lock`-free write guard — see TASKS_INDEX). Read: `list_threads` (`last_activity_at` sort, archived filter), `get_thread(s)`, `search_repo.search_threads`. Uq `(user_id, gmail_id)`. No deletes on the sync path.
- `inbox_messages` — write: message upserts (now include `body_text`/`labels`/derived `is_unread`); `is_deleted` soft-delete (partial sync `messagesDeleted`). `gmail_internal_date` (indexed) is the sort/LWW key; `gmail_history_id` feeds cursor advancement. Uq `(user_id, gmail_id)`. No deletes on the sync path (`clear_user_inbox` is account-deletion-only and currently uncalled).
- `task_thread_links` (**Phase 2A**) — write: `_write_task_links` upserts `origin='llm'` rows for every `triage()` hit `>= TASK_LINK_CONFIDENCE` (partial/full/extend all do this; sticky-rule protected against downgrading a user's own attach/detach). **Phase 4**: `reclassify_user_inbox`'s `_reclassify_all`, the fourth writer this row used to name, is deleted — a bucket's own creation-time backfill (`_run_bucket_backfill`) never writes `task_thread_links` at all (bucket-kind tasks carry no board/pipeline state to link threads into). Read: none from this sync-path subsystem — the task engine reads it (`TASKS_INDEX.md`).
- `llm_calls` — write: one row per LLM call via `app/llm/metrics.record_call` (fire-and-forget, own session, swallows errors). Read: none yet in the sync path (future ops/cost surfacing).
- `users.gmail_last_history_id` — write: `update_user_history_id` (partial = `historyId` from response; full = max history_id across ingested msgs). Read: `poll_new_messages`, `/api/inbox/refresh`, SSE kickoff to choose full vs partial. **Extend does NOT touch it.**

**Redis (3 roles, one instance):**
- `sync_lock:{uid}` — `SET NX EX 600`. Held by every writer task; releases in `finally`. TTL is the crash safety net.
- `active_users` (zset) — `ZADD` on first SSE conn + every 5s heartbeat (score = now+60); `ZREM` on last disconnect; `enqueue_polls` reads + purges. Gates *whom beat polls* (only users with a live SSE).
- `user:{uid}` (pubsub) — worker `PUBLISH`es; per-process dispatcher subscribes/routes to SSE queues. Transient; dropped if no subscriber.
- `last_sync:{uid}` — `SET ex=90d`, self-expiring, overwritten+refreshed in place (marker, not a log; the TTL is a self-cleaning ceiling for abandoned accounts, not a freshness expiry — refreshed on every `mark()`). Written by `last_sync.mark()` at **5** sync-completion sites (`poll_new_messages` ×4 exits, `full_sync_inbox_task` — **Phase 4**: was 6, `reclassify_user_inbox` deleted); **not** written by `extend_inbox_history_task` (nor, since Phase 4, by anything bucket-related — `draft_preview_bucket` is also deleted and bucket creation's replacement isn't a sync-completion event at all). Read by `GET /api/sync/status` (`last_sync.get()`) to power the HUD freshness strip.
- (broker/backend) — task messages.

**External:** Gmail v1 (`history.list` `historyTypes=["messageAdded","messageDeleted","labelAdded","labelRemoved"]` `labelId="INBOX"`; `threads.list maxResults=200 labelIds=["INBOX"]` ±`q=before:<secs>`; `threads.get format=full`); OpenRouter (OpenAI-compatible `chat.completions`, `LLM_CLASSIFY_MODEL` default `anthropic/claude-haiku-4-5`) for **triage** (**Phase 2A** — bucket pick + tracker relevance, one call per thread, `stage="classify"`), per-thread, via `_triage_batch` — see WORKERS for the client/metrics detail, TASKS_INDEX for the tracker-relevance half.

## Decision points & gotchas

- **Full sync is a reconciling upsert, never a wipe.** `full_sync_inbox` re-lists the latest 200 `labelIds=["INBOX"]` threads and upserts each; it never deletes rows (Phase 2 task tables FK onto `inbox_threads.id`, and `HistoryGoneError` recovery — which calls this function — must not be destructive). The listing is authoritative **both directions** within the window it covers: a stored thread the listing returns is un-archived even if a stale `is_archived=True` lingered; a stored, non-archived thread whose `last_activity_at` falls inside the window but which the listing no longer returns is archived (it left the inbox while the cursor was dead). Threads outside the window are left untouched — they're simply out of scope for this listing, not necessarily gone.
- **Reconcile-archive window guarded against messageless listing entries.** `parser.assemble_thread` returns `recent_internal_date=0` for a thread whose raw `messages` came back empty (a malformed/edge-case `threads.list` entry). The archive-window floor (`window_min`) is computed only from parsed threads with `recent_internal_date > 0` — if a stray `0` leaked in, `last_activity_at >= 0` would match nearly every stored thread and mass-archive the inbox in one sync. The listed-ids set used for un-archiving is still built from *all* parsed threads (including messageless ones), so a messageless listed thread is never itself archived.
- **`clear_user_inbox` is account-deletion-only.** No sync path calls it (grep confirms zero call sites outside its own definition as of this stamp); it exists for a future account-deletion flow. Do not wire it into any sync recovery path — that would defeat the reconciling-upsert guarantee above.
- **Widened `historyTypes` = four record shapes, not one.** `history.list` requests `messageAdded`, `messageDeleted`, `labelAdded`, `labelRemoved` (previously `messageAdded` only). `partial_sync_inbox` branches on all four: `messagesAdded` → fetch+**triage**+upsert (dual-write `task_thread_links`, Phase 2A), **then heal `is_archived` from the fetched labels** (see below); `messagesDeleted` → soft-delete (`InboxMessage.is_deleted=True`) + `recompute_thread_pointers` + archive-if-now-empty; `labelsAdded`/`labelsRemoved` with `INBOX` in the label set → flip `is_archived` (or ingest a never-seen thread if INBOX was just added to it); same for `UNREAD` → flip `InboxMessage.is_unread`. Threads/messages touched only by a flag flip never go through the upsert loop but are still merged into the returned `all_ids` so SSE consumers see the change — they are deliberately excluded from `content_ids`, so they never reach extraction (see the flag-only gotcha below).
- **`history.list` pagination is bounded, not single-page.** Gmail pages `history.list` at ~100 records; with four `historyTypes` in play a single poll interval can realistically exceed one page (bulk archive, a cursor that sat stale for a while). `fetch_history_records` now loops `nextPageToken`, accumulating records across pages and returning the *final* page's `historyId`, up to `MAX_HISTORY_PAGES=20` (~2000 records). If a `nextPageToken` still remains at that cap, it raises `HistoryGoneError` — same recovery path as a 404 (fall back to `full_sync_inbox`) — rather than silently dropping every record on unfetched pages, which is what the pre-fix single-page implementation did.
- **Mirror flags self-heal from live fetches, not only from full sync.** Two independent healing writes close the gap where a missed/dropped history record used to leave a stale flag until the next full sync: (1) `partial_sync_inbox`'s `messagesAdded` path (partial-sync-specific) derives `is_archived = not any("INBOX" in m.label_ids for m in parsed.messages)` from every `threads.get(format="full")` result and writes it immediately after the upsert — correct-by-construction because the flag-record loop (labelsAdded/Removed) runs *before* the fetch loop in the same call, so the fetched state is at-least-as-fresh; (2) `inbox_repo.upsert_message`'s update branch unconditionally sets `is_deleted = False` on every call — since full sync/extend/partial all route through `upsert_message` for any thread they actually fetch, a message returned by *any* of those live fetches definitionally exists, healing a spurious/duplicated `messagesDeleted` record instead of hiding the message forever.
- **Cursor expiry → full sync (reconciling, not destructive).** `history.list` 404 (`startHistoryId` past Gmail's ~30-day window) **or exceeding `MAX_HISTORY_PAGES` with a `nextPageToken` still pending** → `HistoryGoneError` → `full_sync_inbox` (see above — upserts + archive/un-archive reconciliation, never a wipe). No cursor at all → also full sync. Same branch in `poll_new_messages`, `/api/inbox/refresh`, and SSE kickoff — **Phase 4**: `_inline_reload`, the fourth caller of this same branch, is deleted along with `reclassify_user_inbox`.
- **Empty history = no publish.** `poll_new_messages` returns silently on 0 records — do NOT publish on noise, or every 30s tick wakes idle clients.
- **`sync_lock` prevents half-synced inboxes.** Concurrent SSE-kickoff + beat poll would both take the full-sync path and collide on uq `(user_id, gmail_id)`, rolling back the txn. All writers (`poll`, `full`, `extend`) acquire it and return silently if held. **Phase 4**: `reclassify_user_inbox` (the one caller that used to retry `countdown=30` up to 3× instead of returning silently) is deleted, along with the bucket-draft-preview flow's `_extend_inline` inline-lock helper — bucket creation's replacement (`backfill_task`'s bucket branch) never touches Gmail or this lock at all (see its own optimistic write guard, TASKS_INDEX), so every remaining `sync_lock` acquirer shares the same "return silently if held" behavior.
- **Cursor anchoring.** Partial sync advances the cursor to the response `historyId`; full sync to the max `gmail_history_id` across ingested messages; **extend must NOT touch the cursor** (it pulls *older* mail — moving the cursor back would make the next partial sync re-ingest or miss).
- **Per-thread `threads.get` is try/except in partial, extend, AND full sync.** A thread deleted (or otherwise unfetchable) between the history record/listing and the fetch errors; without the guard the whole task crashes — the cursor never advances (partial/extend) or the whole reconcile aborts (full) — and the 30s beat retries forever on the same dead thread. Full sync collects unfetchable stub ids into `failed_gmail_ids` and excludes them (alongside `listed_gmail_ids`) from the reconcile-archive candidate query, so a thread gmail still lists but that transiently failed to fetch this round is never wrongly archived as one that actually left the inbox.
- **Internal id, not gmail id, on the wire.** Sync fns return `InboxThread.id` (UUID hex). `threads_updated`/`extend_complete` carry these because `/api/threads/batch` filters by `InboxThread.id` — publishing `gmail_thread_id` would return zero rows.
- **Publish-after-commit; cache-before-publish (preview).** Sync commits to Postgres *before* `PUBLISH`, so the `batch` replay always finds the rows. (Draft-preview likewise `store_result` before publishing — see WORKERS.)
- **`INBOX` label scoping.** `history.list` uses singular `labelId="INBOX"`; `threads.list` uses plural `labelIds=["INBOX"]`. Without these, SENT/DRAFTS leak in and your own sent mail surfaces as inbox threads "from another address."
- **LWW merge + archive eviction on the client.** `applyThreadUpdates` accepts an incoming thread only if `recent_message.internal_date >= stored` (`lastInternalDate` ref), then splits accepted threads into archived vs. live: archived ids are deleted out of `displayLayer`/`idLayer`/the LWW gate map entirely (a thread that left the Gmail inbox disappears from the list rather than lingering with a stale flag); live threads merge + re-sort `idLayer` by date desc. `fetchAndReplace` *resets* (not merges) the gate map so a thread that fell out of the latest-200 window can't keep a stale value.
- **`load_parsed_threads` reconstructs bodies from Postgres, never Gmail.** `_score_all` (now called by the task engine's `propose_task_draft`, not a bucket draft preview — see TASKS_INDEX) calls it instead of looping `gmail.threads().get()` per candidate, keeping that scoring path purely LLM-bound, not Gmail-API-bound. **Phase 4**: `_reclassify_all`, this bullet's other original caller, is deleted along with `reclassify_user_inbox` — its Postgres-only, no-Gmail-refetch pattern lives on in `workers/task_engine_tasks.py`'s `_run_bucket_backfill`/`_run_tracker_backfill` instead (own module, own `load_parsed_threads` calls, full detail → TASKS_INDEX). Falls back `body_text → body_preview` for rows written before 0006 that haven't been re-touched by a sync yet; threads with zero non-deleted messages are skipped entirely.
- **`GET /api/search` is a plain synchronous read** — no Celery task, no lock, no publish. Postgres FTS (`websearch_to_tsquery`, ranked) on real deployments; `ILIKE` substring on SQLite (tests) since generated tsvector columns are Postgres-only (0006 dialect-guards that DDL).
- **Active-user gating (the 30s path only — Phase 3 adds a second, hourly path for tracker owners).** `enqueue_polls`'s 30s tick only polls users in `active_users`; the SSE heartbeat (5s) refreshes a 60s TTL, and `enqueue_polls` purges expired entries — so a dead API process can't strand a user as "active" on THIS path. (Note: `active_users.refresh` docstring says "every 20s" but the live cadence is the SSE `HEARTBEAT_SECONDS=5`.) A user with at least one active, non-deleted, schema-bearing tracker is no longer limited to this path: `enqueue_tracker_owner_polls` (hourly beat tick, **Phase 3**) polls every such owner NOT already in `active_users`, spread across the hour via a deterministic `crc32(uid) % 3600` countdown so they don't all land in the same instant — a tracker keeps advancing even after its owner closes every tab. A user with zero trackers is still only polled while a tab is open. Full detail → WORKERS_INDEX.md.
- **SSE delivery is best-effort.** Dispatcher `put_nowait` drops frames on a full queue (`QUEUE_MAXSIZE=100`), and `_publish` logs `subscribers=0` when nobody's listening (subscribe/unsubscribe churn during SSE flapping). The client compensates with the 90s extend watchdog and the `snapshot`-then-buffer-flush in `useInboxSse` — **Phase 4**: the old bucket-creation 60s+150s reclassify watchdog `resync()` timers are gone from `InboxPage.tsx` along with `reclassify_user_inbox` itself; bucket backfill's `threads_updated`/`task_backfill_progress` publishes are deterministic on completion, so nothing compensates for them the way the old fire-and-forget task needed.
- **Eager-mode tests.** `CELERY_TASK_ALWAYS_EAGER=1` runs tasks inline inside the test txn; `fakeredis` backs `sync_lock`/`active_users`/pubsub; `SessionLocal` in `tasks.py` is monkeypatched onto the in-memory engine.
- **`last_sync:{uid}` tracks sync attempts, not sync changes.** Marked on every successful `poll_new_messages`/`full_sync_inbox_task` completion — including the 0-history-records no-op exit — so `GET /api/sync/status`'s `last_synced_at` means "last time this user's inbox was successfully checked," not "last time something changed." `extend_inbox_history_task` deliberately doesn't mark it (on-demand history pull, not a periodic sync). **Phase 4**: `reclassify_user_inbox` (a former marker site) and `draft_preview_bucket` (a former non-marker) are both deleted — bucket creation's replacement, `backfill_task`'s bucket branch, isn't a sync-completion event at all and correctly never marks it either (full writer/reader detail in WORKERS_INDEX).
- **Sync feeds the task engine — content-touched/freshly-linked ids only, never flag-only touches** (Phase 2A, tightened by the final-review fixes). `poll_new_messages`/`full_sync_inbox_task` `apply_async task_engine_tasks.process_task_updates(uid, content_ids)` right after their `threads_updated` publish; `extend_inbox_history_task` enqueues `new_link_ids` (its freshly-linked subset) — this is the edge that lets a tracker's pipeline advance from ordinary background sync or "load more history," not just from a user manually attaching a thread. The publish set (`all_ids`) is deliberately WIDER than the extraction set: a flag-only touch (unread flip, archive/unarchive, soft-delete, reconcile-archive) must reach SSE consumers but has no new content for a tracker to extract from — routing it into extraction fired a Sonnet call per active tracker that deduped only after the LLM cost was spent (up to 200× on a 404-recovery full sync). **Phase 4**: `reclassify_user_inbox` (a former 3rd enqueue site, via its now-deleted `_reclassify_all`) and `draft_preview_bucket` (a former never-enqueues task) are both deleted; a bucket-kind `backfill_task` run — bucket creation's replacement trigger — never enqueues `process_task_updates` either, for the same reason `draft_preview_bucket` never did: it writes no `task_thread_links` and has no tracker/board state of its own to feed. `last_sync` marking keeps its own poll/full-vs-extend split for an unrelated reason (see WORKERS_INDEX gotchas; TASKS_INDEX for what `process_task_updates` does once enqueued).
- **The cross-route SSE-churn risk is gone structurally** (**rewritten** — supersedes the old "AppShell pins the SSE singleton open" entry). `lib/sse.ts`'s `EventSource` closes when its last `subscribeSse` handler unsubscribes and reopens fresh on the next subscribe. Previously each route mounted its own subscriber (`HudPage` via a direct `subscribeSse` call, `InboxPage` via `useInboxSse`), so a cross-route navigation could transiently drop the handler count to zero between the outgoing route's unmount-cleanup and the incoming route's mount-resubscribe — closing and reopening the connection and deregistering the user from `active_users` (this doc's steady-state loop above) on every navigation; `client/src/AppShell.tsx` papered over this with a permanent no-op handler. Now `client/src/state/InboxProvider.tsx` (mounted once by `AppShell` for the life of the authed session) owns the **one** `useInboxSse` subscription both routes' inbox state feeds off of — `HudPage` and `InboxPage` no longer mount their own SSE handlers at all, so the subscriber count never transiently (or otherwise) hits zero during `/`↔`/inbox` navigation. The old `AppShell` no-op pin is deleted; there's nothing left to churn (full detail in CLIENT_INDEX gotchas).
