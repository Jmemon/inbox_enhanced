# Phase 4.5 — Jobs Surface Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

> Spec: `specs/005_jobs_surface/design.md`.
> Plan stamped at commit `36aa27b` on branch `main`.

**Goal:** Creation and delete-retriage work becomes persisted, pollable `jobs`; a header chip + slide-over panel with a blue dot replaces every auto-opening waiting popup; bucket deletion re-triages its orphaned threads; SchemaEditor shows stage order (arrows + drag).

**Architecture:** A `jobs` Postgres table is the single source of truth for long-running work (SSE `job_updated` is a pure nudge; `GET /api/jobs` is the always-works poll path — the structural fix for the stranded popup). The wizard splits into two user-initiated moments (start → close; review-on-blue-dot → confirm → close). The draft cache and old draft routes retire; drafts live in `jobs.payload` and stop expiring.

**Tech Stack:** existing (uv / bun; no new dependencies).

## Global Constraints

- All prior conventions hold: `uv`/`bun` only; NEVER read `.env`; repo functions never commit; read helpers never flush; write helpers may flush; publish-after-commit; TDD server-side (suite baseline **491 passed**; old draft-route tests are deleted/rewritten — invariant is 0 failures); client verification = `bun run build` + `bun x tsc --noEmit`; worktrees lack `.env` → settings assertions via resolved `get_settings()`; SQLite migration tests → dialect-guard PG-only DDL; `sa.JSON().with_variant(JSONB, "postgresql")`.
- SSE events carry ids, never rows: `job_updated {job_id}` only.
- `jobs.user_id` is NOT NULL — jobs are always user-owned; `list_jobs` scoping is a security boundary (same posture as the feeds — probe it in review).
- Stage strings are exact: creation `proposing|draft_ready|backfilling|done|failed|dismissed`; delete_retriage `running|done|failed`. `needs_user` is true ONLY in `draft_ready`.
- The wizard's tracker path must remain behaviorally intact through the restructure (mode-split discipline as in Phase 4 T5).
- Commit per task, `type(scope): summary`, no attribution lines.

### Task 1: Jobs storage — migration 0010, model, repo

**Files:** Create `server/migrations/versions/0010_jobs.py`, `server/app/task_engine/jobs_repo.py`; Modify `server/app/db/models.py`; Test: create `server/tests/test_migration_0010.py` (per-revision convention), `server/tests/test_jobs_repo.py`.

- `Job` model per spec §1.1 exactly: `id String(36) PK`, `user_id String(36) FK users.id NOT NULL indexed`, `kind String(32) NOT NULL`, `task_kind String(16) nullable`, `stage String(32) NOT NULL`, `needs_user Boolean NOT NULL default False server_default "false"`, `payload JSON/JSONB(none_as_null=True) nullable`, `task_id String(36) FK tasks.id nullable`, `goal Text NOT NULL default ""`, `scanned/matched/total Integer NOT NULL default 0 server_default "0"`, `error Text nullable`, `created_at/updated_at DateTime(timezone=True) NOT NULL`, `dismissed_at DateTime(timezone=True) nullable`.
- Repo (never commits; writes flush): `create_job(db, *, user_id, kind, task_kind=None, goal="") -> Job` (stage `proposing` for creation / `running` for delete_retriage — derive from kind); `get_owned_job(db, *, user_id, job_id) -> Job | None`; `list_jobs(db, *, user_id, active_only=True) -> list[Job]` — active = `dismissed_at IS NULL AND (stage NOT IN ('done','failed') OR updated_at >= now-7d)`, newest first; `update_stage(db, *, job, stage)` (sets `needs_user = (stage == 'draft_ready')`, touches `updated_at`); `update_progress(db, *, job, scanned, matched, total)`; `set_payload(db, *, job, payload)`; `mark_failed(db, *, job, error)`; `dismiss(db, *, job)` (sets `dismissed_at`, idempotent).
- [ ] TDD: migration test seeds nothing (new table — assert exists + columns after upgrade, absent after downgrade); repo tests: create/list ordering, active-window semantics (terminal >7d excluded, non-terminal always included, dismissed excluded), **cross-user scoping probe**, needs_user toggling on stage transitions, dismiss idempotency. Full suite green.
- [ ] Commit: `feat(db): jobs table + repo for the jobs surface`

### Task 2: backfill_task job integration + failure hardening

**Files:** Modify `server/app/workers/task_engine_tasks.py`; Test extend `server/tests/test_task_engine_tasks.py`.

- `backfill_task(user_id, task_id, keyword_probes=None, job_id=None)` — new optional kwarg (direct `POST /api/tasks` keeps passing none; panel just won't show those).
- When `job_id` is set: per batch, after the existing commit, also `update_progress` on the job row (`total` = candidate-pool size, set once at start) and commit; terminal: `update_stage(job, 'done')`. Publish `job_updated {job_id}` after every job-row commit (publish-after-commit).
- Top-level `try/except Exception` around the whole body (both kinds): on exception, roll back, `mark_failed(job, str(exc))` + commit + `job_updated` publish when `job_id` present, re-raise (celery still records FAILURE). This closes the ledgered "raise before terminal publish silently hangs the client" latent.
- [ ] TDD: job progress rows updated per batch and terminal `done` (monkeypatched triage); injected exception → job `failed` with error text + still raises; `job_id=None` path byte-identical behavior (no job writes). Full suite green.
- [ ] Commit: `feat(engine): backfill writes job progress + fails loudly into the job row`

### Task 3: Jobs API + job-based propose; old draft path retired

**Files:** Create `server/app/api/jobs.py`; Modify `server/app/main.py` (mount router), `server/app/workers/task_engine_tasks.py` (propose rewrite), `server/app/api/tasks.py` (delete `POST /tasks/draft` + `GET /tasks/draft/{id}` routes and `_DraftBody`); Delete `server/app/task_engine/draft_cache.py`; Test: create `server/tests/test_jobs_api.py`, update `server/tests/test_propose_task.py` (job-based), delete draft-route tests.

- `POST /api/jobs` `{goal: str min_length=1, task_kind: Literal['tracker','bucket']}` → 202 `{job: {...}}`: `create_job` + commit + enqueue `propose_task_draft.apply_async(args=[user.id, job.id, goal])` (worker keeps its name; its `draft_id` param becomes `job_id`).
- Propose worker rewrite: writes the proposal into `jobs.payload` as `{name, description, criteria, state_schema, keyword_probes, positives, near_misses}` (same shape the draft cache held — read the current code for exact keys and keep them), sets `draft_ready`, commits, publishes `job_updated`. Existing 2-call retry cap preserved; on exhaustion/exception → `mark_failed` + publish (mirror Task 2's pattern). The old `task_draft_ready` SSE publish retires.
- `GET /api/jobs?active=` (default 1; `active=0` returns everything non-dismissed) → `{jobs: [...]}`; `GET /api/jobs/{job_id}` → `{job}`; 404 via `get_owned_job`. Serializer: all spec §1.1 columns except `user_id`.
- `POST /api/jobs/{job_id}/confirm` — body identical to `_CreateTaskBody` minus `kind` (the job's `task_kind` governs; schema 422 rules identical — share the validation helper with `create_task`, do not duplicate it). 409 `"job is not awaiting review"` unless stage == `draft_ready`. Creates the task through the same internals as `POST /api/tasks` (criteria formulation + `task_repo.create_task` + version publish), sets `job.task_id`, `stage='backfilling'`, commits, enqueues `backfill_task(..., job_id=job.id)`, publishes `job_updated`. Returns the task detail + job.
- `POST /api/jobs/{job_id}/dismiss` → 204, idempotent.
- [ ] TDD: create→202 + row `proposing`; propose success fills payload + `draft_ready` + `needs_user`; propose failure → `failed`; confirm stage-gating 409; confirm 422 parity (tracker-needs-schema / bucket-forbids-schema); confirm happy path creates task + enqueues backfill with `job_id` (capture apply_async); dismiss idempotent; ownership 404s; **cross-user probe on all routes**; old draft routes now 404/405. Full suite green.
- [ ] Commit: `feat(api): jobs routes + job-based propose; draft cache retired`

### Task 4: Bucket-delete re-triage job

**Files:** Modify `server/app/api/tasks.py` (delete route), `server/app/workers/task_engine_tasks.py` (new worker); Test extend `server/tests/test_tasks_api.py`, `test_task_engine_tasks.py`.

- In `delete_task` (api/tasks.py:396): after the existing soft-delete/bump/commit/publish, when `task.kind == 'bucket'`, count `inbox_threads WHERE user_id AND bucket_id == task.id`; if count > 0, `create_job(kind='delete_retriage', goal=f"reclassify after deleting {task.name}")` with `task_id=task.id`, `total=count`, stage `running`, commit, enqueue `retriage_deleted_bucket.apply_async(args=[user.id, job.id, task.id])`, publish `job_updated`. Zero orphans → no job.

Reuse `_run_bucket_backfill`'s write discipline verbatim: optimistic skip-if-moved guard, per-batch commit, `threads_updated` publish, job progress + terminal stage, top-level try/except → `failed`. `matched` counts threads that landed in a real bucket. No links/events/extraction ever.
- [ ] TDD: delete bucket with orphans → job created with total + worker enqueued (captured); worker rewrites only threads still pointing at the deleted id (guard test: mid-flight move is skipped); no-better-fit → NULL; deleting a TRACKER creates no job; zero-orphan bucket delete creates no job. Full suite green.
